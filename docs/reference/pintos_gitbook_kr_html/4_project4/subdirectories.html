하위 디렉터리와 소프트 링크

하위 디렉터리(Subdirectories)

계층적 이름 공간(hierarchical name space)을 구현하십시오. 기본 파일 시스템에서는 모든 파일이 하나의 디렉터리(directory)에 들어 있습니다. 디렉터리(directory) 엔트리가 파일이나 다른 디렉터리(directory)를 가리킬 수 있도록 이를 수정하십시오.

다른 파일과 마찬가지로 디렉터리(directory)도 원래 크기를 넘어 확장될 수 있어야 합니다.

기본 파일 시스템은 파일 이름을 14자로 제한합니다. 개별 파일 이름 구성 요소에 대해서는 이 제한을 유지해도 되고, 원한다면 확장해도 됩니다. 전체 경로 이름은 반드시 14자보다 훨씬 길어질 수 있어야 합니다.

각 프로세스(process)마다 별도의 현재 디렉터리(current directory)를 유지하십시오. 시작 시 초기 프로세스(process)의 현재 디렉터리(current directory)를 루트로 설정하십시오. 한 프로세스(process)가 fork 시스템 콜(system call)로 다른 프로세스를 시작하면, 자식 프로세스(process)는 부모의 현재 디렉터리(current directory)를 상속합니다. 그 후에는 두 프로세스(process)의 현재 디렉터리(current directory)가 서로 독립적이므로, 어느 한쪽이 자신의 현재 디렉터리(current directory)를 변경해도 다른 쪽에는 영향을 주지 않습니다. (이것이 Unix에서 cd 명령이 외부 프로그램이 아니라 셸 내장 명령인 이유입니다.)

호출자가 파일 이름을 제공하는 기존 시스템 콜(system call)들을 갱신하여, 어디에서든 절대 경로 이름이나 상대 경로 이름을 사용할 수 있게 하십시오. 디렉터리(directory) 구분 문자는 슬래시('/')입니다. 또한 Unix에서와 같은 의미를 갖는 특수 파일 이름 '.' 및 '..'도 반드시 지원해야 합니다.

open 시스템 콜(system call)을 갱신하여 디렉터리(directory)도 열 수 있게 하십시오. 기존 시스템 콜(system call) 중에서는 close만 디렉터리(directory)에 대한 파일 디스크립터(file descriptor)를 받을 수 있으면 됩니다.

remove 시스템 콜(system call)을 갱신하여 일반 파일뿐 아니라 빈 디렉터리(directory)(루트 제외)도 삭제할 수 있게 하십시오. 디렉터리(directory)는 파일이나 하위 디렉터리(subdirectory)('.' 및 '..' 제외)를 전혀 포함하지 않을 때만 삭제할 수 있습니다. 프로세스(process)가 열어 두었거나 프로세스의 현재 작업 디렉터리(current working directory)로 사용 중인 디렉터리(directory)의 삭제를 허용할지는 여러분이 결정할 수 있습니다. 이를 허용한다면, 삭제된 디렉터리(directory) 안에서 파일('.' 및 '..' 포함)을 열거나 새 파일을 만드는 시도는 반드시 허용되지 않아야 합니다. 다음 새 시스템 콜(system call)들을 구현하십시오:

bool chdir (const char *dir);

프로세스(process)의 현재 작업 디렉터리(current working directory)를 dir로 변경합니다. dir는 상대 경로일 수도 있고 절대 경로일 수도 있습니다. 성공하면 true를, 실패하면 false를 반환합니다.

bool mkdir (const char *dir);

dir라는 이름의 디렉터리(directory)를 만듭니다. dir는 상대 경로일 수도 있고 절대 경로일 수도 있습니다. 성공하면 true를, 실패하면 false를 반환합니다. dir가 이미 존재하거나, dir 안의 디렉터리(directory) 이름 중 마지막을 제외한 어떤 이름이라도 아직 존재하지 않으면 실패합니다. 즉, mkdir("/a/b/c")는 /a/b가 이미 존재하고 /a/b/c가 존재하지 않을 때만 성공합니다.

bool readdir (int fd, char *name);

디렉터리(directory)를 나타내야 하는 파일 디스크립터(file descriptor) fd에서 디렉터리(directory) 엔트리를 읽습니다. 성공하면 널 종료 파일 이름을 name에 저장하고 true를 반환합니다. name에는 READDIR_MAX_LEN + 1바이트를 담을 공간이 있어야 합니다. 디렉터리(directory)에 남은 엔트리가 없으면 false를 반환합니다.

. 및 ..은 readdir에서 반환되어서는 안 됩니다. 디렉터리(directory)가 열린 동안 변경되면, 일부 엔트리가 전혀 읽히지 않거나 여러 번 읽히는 것도 허용됩니다. 그렇지 않다면 각 디렉터리(directory) 엔트리는 어떤 순서로든 한 번씩 읽혀야 합니다.

READDIR_MAX_LEN는 lib/user/syscall.h에 정의되어 있습니다. 여러분의 파일 시스템이 기본 파일 시스템보다 더 긴 파일 이름을 지원한다면, 이 값을 기본값 14에서 늘려야 합니다.

bool isdir (int fd);

fd가 디렉터리(directory)를 나타내면 true를 반환하고, 일반 파일을 나타내면 false를 반환합니다.

int inumber (int fd);

fd와 연결된 아이노드(inode)의 아이노드(inode) 번호를 반환합니다. fd는 일반 파일이나 디렉터리(directory)를 나타낼 수 있습니다.

아이노드(inode) 번호는 파일이나 디렉터리(directory)를 지속적으로 식별합니다. 파일이 존재하는 동안 고유합니다. Pintos에서는 아이노드(inode)의 섹터 번호가 아이노드(inode) 번호로 사용하기에 적합합니다.

소프트 링크(Soft Link)

Pintos에서 소프트 링크(soft link) 메커니즘을 구현하십시오. 소프트 링크(soft link)(기호 링크(symbolic link)라고도 함)는 다른 파일이나 디렉터리(directory)를 참조하는 의사 파일 객체입니다. 이 파일에는 가리키는 파일의 경로 정보가 절대 경로나 상대 경로 형태로 들어 있습니다. 다음 상황을 가정해 봅시다:

/
├── a
│   ├── link1 -> /file
│   │
│   └── link2 -> ../file
└── file

이름이 link1인 /a 안의 소프트 링크(soft link)는 /file(절대 경로)을 가리키고, link2는 /a 안에서 ../file(상대 경로)을 가리킵니다. link1 또는 link2 파일을 읽는 것은 /file을 읽는 것과 같습니다.

int symlink (const char *target, const char *linkpath);

문자열 target을 포함하는 linkpath라는 이름의 기호 링크(symbolic link)를 만듭니다. 성공하면 0을 반환합니다. 그렇지 않으면 -1을 반환합니다.
