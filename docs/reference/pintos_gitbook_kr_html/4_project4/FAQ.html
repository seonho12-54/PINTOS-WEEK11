자주 묻는 질문

BLOCK_SECTOR_SIZE가 바뀔 수 있나요?

아니요. BLOCK_SECTOR_SIZE는 512로 고정되어 있습니다. IDE 디스크(disk)의 경우 이 값은 하드웨어의 고정 속성입니다. 다른 디스크(disk)가 반드시 512바이트 섹터(sector)를 가지는 것은 아니지만, 단순화를 위해 Pintos는 그런 디스크(disk)만 지원합니다.

인덱스 파일(Indexed Files) FAQ

지원해야 하는 가장 큰 파일(file) 크기는 얼마인가요?

우리가 만드는 파일 시스템(file system) 파티션(partition)은 8 MB 이하입니다. 하지만 개별 파일(file)은 메타데이터(metadata)를 수용하기 위해 파티션(partition)보다 작아야 합니다. 아이노드(inode) 구성을 결정할 때 이 점을 고려해야 합니다.

하위 디렉터리(Subdirectories) FAQ

a//b 같은 파일(file) 이름은 어떻게 해석해야 하나요?

연속된 여러 슬래시(slash)는 하나의 슬래시(slash)와 같으므로, 이 파일(file) 이름은 a/b와 같습니다.

/../x 같은 파일(file) 이름은 어떤가요?

루트 디렉터리(root directory)는 자기 자신의 부모이므로 /x와 같습니다.

/로 끝나는 파일(file) 이름은 어떻게 처리해야 하나요?

대부분의 Unix 시스템(system)은 디렉터리(directory) 이름 끝의 슬래시(slash)를 허용하고, 슬래시(slash)로 끝나는 다른 이름은 거부합니다. 이 동작을 허용해도 되고, 슬래시(slash)로 끝나는 이름을 단순히 거부해도 됩니다.

버퍼 캐시(buffer cache) FAQ

struct inode_disk를 struct inode 안에 보관할 수 있나요?

64블록(block) 제한의 목표는 캐시(cache)된 파일 시스템(file system) 데이터의 양을 제한하는 것입니다. 파일(file) 데이터든 메타데이터(metadata)든 디스크(disk) 데이터 블록(block)을 커널(kernel) 메모리 어디에든 보관한다면, 그것을 64블록(block) 제한에 포함해야 합니다. 디스크(disk) 데이터 블록(block)과 "비슷한" 모든 것에도 같은 규칙이 적용됩니다. 예를 들어 struct inode_disk에서 length 또는 sector_cnt 멤버를 뺀 것과 같은 경우입니다.

즉, 아이노드(inode) 구현이 현재 대응하는 온디스크 아이노드(on-disk inode)에 접근하는 방식을 바꾸어야 합니다. 현재는 struct inode_disk를 struct inode 안에 그대로 포함하고, 생성될 때 디스크(disk)에서 대응하는 섹터(sector)를 읽습니다. 아이노드(inode)의 추가 사본을 유지하면 캐시(cache)에 부과한 64블록(block) 제한을 무력화하게 됩니다.

struct inode에 아이노드(inode) 데이터에 대한 포인터(pointer)를 저장할 수는 있지만, 그렇게 한다면 이 때문에 OS가 동시에 열 수 있는 파일(file)을 64개로 제한하지 않는지 주의 깊게 확인해야 합니다. 필요할 때 아이노드(inode)를 찾는 데 도움이 되는 다른 정보를 저장할 수도 있습니다. 마찬가지로 64개의 캐시(cache) 항목 각각에 일부 메타데이터(metadata)를 함께 저장해도 됩니다.

원한다면 프리 맵(free map)의 캐시(cache)된 사본을 메모리에 영구적으로 보관할 수 있습니다. 이는 캐시(cache) 크기에 포함할 필요가 없습니다.

byte_to_sector()는 filesys/inode.c에서 struct inode_disk를 직접 사용하며, 저장소 계층(storage hierarchy)의 어디에 있든 그 섹터(sector)를 먼저 읽지 않습니다. 이 방식은 더 이상 동작하지 않습니다. inode_byte_to_sector()를 변경하여 캐시(cache)에서 struct inode_disk를 얻은 뒤 사용해야 합니다.
