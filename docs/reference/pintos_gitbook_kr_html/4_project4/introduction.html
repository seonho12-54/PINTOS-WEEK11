소개

프로젝트 4: 파일 시스템(File Systems)

앞선 두 과제에서는 파일 시스템(file system)이 내부에서 어떻게 구현되었는지 크게 신경 쓰지 않고도 파일 시스템을 폭넓게 사용했습니다. 이번 마지막 과제에서는 파일 시스템 구현을 개선합니다. 주로 filesys 디렉터리에서 작업하게 됩니다.

프로젝트 4는 프로젝트 2 또는 프로젝트 3 위에 구현할 수 있습니다. 어느 경우든 프로젝트 2에 필요한 모든 기능은 파일 시스템 제출물에서 동작해야 합니다. 프로젝트 3 위에 구현한다면 프로젝트 3의 모든 기능도 동작해야 하며, VM 기능을 활성화하려면 filesys/Make.vars를 수정해야 합니다. VM을 비활성화하면 10%의 점수를 감점합니다. 페이지 캐시(page cache, 추가 점수)는 VM 기능을 필요로 한다는 점에 유의하십시오.

배경

새 코드

다음은 여러분에게 새로울 가능성이 큰 파일들입니다. 별도로 표시된 경우를 제외하면 모두 filesys 디렉터리에 있습니다.

  • filesys/fsutil.c

    커널(kernel) 명령줄에서 접근할 수 있는 파일 시스템용 간단한 유틸리티입니다.

  • include/filesys/filesys.h, filesys/filesys.c

    파일 시스템에 대한 최상위 인터페이스입니다. 소개는 파일 시스템 사용하기를 참고하십시오.

  • include/filesys/directory.h, filesys/directory.c

    파일 이름을 아이노드(inode)로 변환합니다. 디렉터리(directory) 자료구조는 파일로 저장됩니다.

  • include/filesys/inode.h, filesys/inode.c

    디스크에 있는 파일 데이터의 배치를 나타내는 자료구조를 관리합니다.

  • include/filesys/fat.h, filesys/fat.c

    FAT 파일 시스템을 관리합니다.

  • include/filesys/file.h, filesys/file.c

    파일 읽기와 쓰기를 디스크 섹터 읽기와 쓰기로 변환합니다.

  • include/filesys/page_cache.h, filesys/page_cache.c

    VM 기능을 활용하는 페이지 캐시(page cache) 구현입니다. 이 템플릿을 사용하는 것을 강력히 권장합니다. 원한다면 직접 코드를 작성해도 됩니다. VM 플래그를 끄면 이 템플릿을 사용할 수 없다는 점에 유의하십시오.

  • include/lib/kernel/bitmap.h, lib/kernel/bitmap.c

    비트맵 자료구조와, 비트맵을 디스크 파일에 읽고 쓰기 위한 루틴입니다.

이 파일 시스템은 Unix와 비슷한 인터페이스를 제공하므로, creat, open, close, read, write, lseek와 unlink에 대한 Unix 매뉴얼 페이지를 읽어 보는 것도 좋습니다. 이 파일 시스템에는 이들과 유사하지만 완전히 같지는 않은 호출들이 있습니다. 파일 시스템은 이러한 호출들을 디스크 연산으로 변환합니다.

앞의 코드에는 기본 기능이 모두 들어 있으므로, 이전 두 프로젝트에서 보았듯이 파일 시스템은 처음부터 사용할 수 있습니다. 그러나 심각한 제약이 있으며, 여러분은 이를 제거해야 합니다.

대부분의 작업은 filesys에서 이루어지지만, 이전 모든 부분과 상호작용할 수 있음을 염두에 두어야 합니다.

파일 시스템 영속성 테스트

지금까지 각 테스트는 Pintos를 한 번만 실행했습니다. 그러나 파일 시스템의 중요한 목적은 한 번 부팅한 뒤에도 데이터가 계속 접근 가능하도록 보장하는 것입니다. 따라서 파일 시스템 프로젝트의 테스트는 Pintos를 두 번째로 실행합니다. 두 번째 실행에서는 파일 시스템 안의 모든 파일과 디렉터리를 하나의 파일로 묶은 다음, 그 파일을 Pintos 파일 시스템 밖의 호스트(Unix) 파일 시스템으로 복사합니다.

채점 스크립트는 두 번째 실행에서 복사되어 나온 파일의 내용을 바탕으로 파일 시스템의 정확성을 검사합니다. 즉, 복사되어 나오는 파일을 생성하는 Pintos 사용자 프로그램(user program)인 tar를 지원할 만큼 파일 시스템이 충분히 구현되기 전까지는 확장 파일 시스템 테스트를 통과할 수 없습니다. tar 프로그램은 상당히 까다롭고(확장 가능한 파일과 하위 디렉터리 지원이 모두 필요합니다), 따라서 어느 정도 작업이 필요합니다. 그 전까지는 추출된 파일 시스템과 관련된 make check 오류를 무시해도 됩니다.

참고로, 짐작했을 수도 있듯이 파일 시스템 내용을 복사해 내는 데 사용되는 파일 형식은 표준 Unix "tar" 형식입니다. Unix tar 프로그램을 사용해 이를 살펴볼 수 있습니다. 테스트 t에 대한 tar 파일의 이름은 t.tar입니다.
