인덱스 기반 확장 가능 파일

인덱스 기반 확장 가능 파일

기본 파일 시스템은 파일을 하나의 연속 영역(extent)으로 할당하므로 외부 단편화(external fragmentation)에 취약합니다. 즉, n개의 블록이 비어 있어도 n블록 파일을 할당하지 못할 수 있습니다. 온디스크 아이노드(on-disk inode) 구조를 수정해 이 문제를 제거하십시오.

실제로는 직접(direct), 간접(indirect), 이중 간접(doubly indirect) 블록을 사용하는 인덱스 구조가 필요하다는 뜻일 수 있습니다. 지난 학기들에는 대부분의 학생이 다단계 인덱싱을 사용하는 Berkeley UNIX FFS와 비슷한 방식을 채택했습니다. 그러나 여러분의 부담을 줄이기 위해 더 쉬운 방식인 FAT로 구현하게 합니다. 제공된 스켈레톤 코드로 FAT를 반드시 구현해야 합니다. 여러분의 코드는 다단계 인덱싱(강의의 FFS)을 위한 코드를 절대 포함해서는 안 됩니다. 그런 코드가 있으면 파일 증가 부분에서 0점을 받습니다.

참고: 파일 시스템 파티션은 8 MB보다 크지 않다고 가정해도 됩니다. 파티션 크기(메타데이터 제외)만큼 큰 파일을 지원해야 합니다. 각 아이노드(inode)는 하나의 디스크 섹터에 저장되므로, 담을 수 있는 블록 포인터 수가 제한됩니다.

FAT(File Allocation Table)로 큰 파일 인덱싱하기

경고: 이 문서는 여러분이 강의에서 일반적인 파일 시스템(general filesystem)과 FAT의 기본 원리를 이미 이해했다고 가정합니다. 그렇지 않다면 강의 노트로 돌아가 파일 시스템과 FAT가 무엇인지 이해하고 오십시오.

이전 프로젝트에서 사용한 기본 파일 시스템에서는 파일이 여러 디스크 섹터에 걸친 하나의 연속 청크(contiguous chunk)로 저장되었습니다. 클러스터(cluster, 청크)는 하나 이상의 연속 디스크 섹터를 포함할 수 있으므로, 이 연속 청크(contiguous chunk)를 클러스터(cluster)라고 부르겠습니다. 이러한 관점에서 기본 파일 시스템의 클러스터 크기는 그 클러스터에 저장된 파일의 크기와 같았습니다.

외부 단편화를 완화하려면 클러스터 크기를 줄일 수 있습니다(가상 메모리(virtual memory)의 페이지 크기를 떠올려 보십시오). 단순화를 위해 우리 스켈레톤 코드에서는 클러스터 하나에 들어가는 섹터 수를 1로 고정했습니다. 이렇게 더 작은 클러스터를 사용하면 클러스터 하나만으로는 전체 파일을 저장하기에 부족할 수 있습니다. 이 경우 파일 하나에 여러 클러스터가 필요하므로, 아이노드(inode) 안에서 파일의 클러스터들을 인덱싱할 자료구조가 필요합니다. 가장 쉬운 방법 중 하나는 연결 리스트(linked-list, 일명 체인(chain))를 사용하는 것입니다. 아이노드(inode)는 파일의 첫 번째 블록 섹터 번호를 담고, 첫 번째 블록은 두 번째 블록의 섹터 번호를 담을 수 있습니다. 그러나 이 단순한 방식은 실제로 필요한 것이 마지막 블록뿐이어도 파일의 모든 블록을 읽어야 하므로 너무 느립니다. 이를 해결하기 위해 FAT(File Allocation Table)는 블록 자체가 아니라 고정 크기의 File Allocation Table 안에 블록 간 연결 정보를 둡니다. FAT는 실제 데이터가 아니라 연결 값만 담기 때문에 DRAM에 캐시할 수 있을 만큼 작습니다. 결과적으로 테이블에서 해당 엔트리만 읽으면 됩니다.

여러분은 filesys/fat.c에 제공된 스켈레톤 코드로 아이노드(inode) 인덱싱을 구현하게 됩니다. 이 절에서는 fat.c에 이미 구현된 함수들이 무엇을 하는지, 그리고 여러분이 무엇을 구현해야 하는지를 간략히 설명합니다.

먼저 fat.c의 6개 함수(fat_init(), fat_open(), fat_close(), fat_create(), 그리고 fat_boot_create())는 부팅 시 디스크를 초기화하고 포맷하므로 수정할 필요가 없습니다. 하지만 fat_fs_init()는 직접 작성해야 하며, 이 함수들이 무엇을 하는지 간단히 이해해 두면 도움이 될 수 있습니다.

cluster_t fat_fs_init (void);

FAT 파일 시스템을 초기화합니다. fat_length와 data_start 필드를 fat_fs에서 초기화해야 합니다. fat_length는 파일 시스템에 몇 개의 클러스터가 있는지를 저장하고, data_start는 파일 저장을 시작할 수 있는 섹터를 저장합니다. fat_fs->bs에 저장된 일부 값을 활용하고 싶을 수 있습니다. 또한 이 함수에서 다른 유용한 데이터를 초기화할 수도 있습니다.

cluster_t fat_create_chain (cluster_t clst);

clst(클러스터 인덱싱 번호)로 지정된 클러스터 뒤에 클러스터를 덧붙여 체인을 확장합니다. clst가 0이면 새 체인을 만듭니다. 새로 할당된 클러스터의 클러스터 번호를 반환합니다.

void fat_remove_chain (cluster_t clst, cluster_t pclst);

clst부터 시작하여 체인에서 클러스터들을 제거합니다. pclst는 체인에서 바로 앞에 있는 클러스터여야 합니다. 즉, 이 함수가 실행된 뒤에는 pclst가 갱신된 체인의 마지막 원소여야 합니다. clst가 체인의 첫 번째 원소라면 pclst는 0이어야 합니다.

void fat_put (cluster_t clst, cluster_t val);

클러스터 번호 clst가 가리키는 FAT 엔트리를 val로 갱신합니다. FAT의 각 엔트리는 체인에서 다음 클러스터가 있으면 그 클러스터를 가리키고, 없으면 EOChain을 가리키므로, 이 함수는 연결 정보를 갱신하는 데 사용할 수 있습니다.

cluster_t fat_get (cluster_t clst);

주어진 클러스터 clst가 가리키는 클러스터 번호를 반환합니다.

disk_sector_t cluster_to_sector (cluster_t clst);

클러스터 번호 clst를 해당 섹터 번호로 변환하고 그 섹터 번호를 반환합니다.

기본 파일 시스템을 확장하기 위해 filesys.c와 inode.c에서 이 함수들을 활용할 수 있습니다.

파일 증가

확장 가능한 파일(extensible files)을 구현하십시오. 기본 파일 시스템에서는 파일을 만들 때 파일 크기가 지정됩니다. 하지만 대부분의 현대 파일 시스템에서는 파일이 처음에 크기 0으로 만들어지고, 파일 끝을 넘어 쓰기가 발생할 때마다 확장됩니다. 여러분의 파일 시스템은 이를 허용해야 합니다.

파일이 파일 시스템의 크기(메타데이터 제외)를 넘을 수 없다는 점 외에는 파일 크기에 미리 정해진 제한이 없어야 합니다. 이는 루트 디렉터리(directory) 파일에도 적용되며, 이제 루트 디렉터리는 초기 한계인 16개 파일을 넘어 확장될 수 있어야 합니다.

사용자 프로그램은 현재 파일 끝(end-of-file, EOF)을 넘어 seek할 수 있습니다. seek 자체는 파일을 확장하지 않습니다. EOF를 지난 위치에 쓰면 파일은 쓰는 위치까지 확장되며, 이전 EOF와 쓰기 시작 위치 사이의 빈 영역은 0으로 채워져야 합니다. EOF를 지난 위치에서 시작하는 읽기는 아무 바이트도 반환하지 않습니다.

EOF를 훨씬 넘어 쓰면 완전히 0인 블록이 많이 생길 수 있습니다. 어떤 파일 시스템은 이렇게 암묵적으로 0으로 채워지는 블록에 실제 데이터 블록을 할당하고 씁니다. 다른 파일 시스템은 명시적으로 쓰기 전까지 이러한 블록을 전혀 할당하지 않습니다. 후자의 파일 시스템은 "sparse file"을 지원한다고 합니다. 여러분의 파일 시스템에서는 두 할당 전략 중 어느 쪽이든 채택할 수 있습니다.
