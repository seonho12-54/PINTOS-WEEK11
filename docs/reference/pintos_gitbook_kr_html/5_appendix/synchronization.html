동기화(Synchronization)

동기화(Synchronization)

스레드(thread)들 사이의 자원 공유를 신중하고 제어된 방식으로 처리하지 않으면 대개 큰 혼란이 생깁니다. 잘못된 공유가 전체 머신을 크래시시킬 수 있는 운영체제 커널(kernel)에서는 특히 그렇습니다. Pintos는 이를 돕기 위해 여러 동기화(synchronization) 프리미티브를 제공합니다.

인터럽트 비활성화(Disabling Interrupts)

동기화(synchronization)를 수행하는 가장 투박한 방법은 인터럽트(interrupt)를 비활성화하는 것, 즉 CPU가 인터럽트(interrupt)에 응답하지 못하도록 일시적으로 막는 것입니다. 인터럽트(interrupt)가 꺼져 있으면 타이머(timer) 인터럽트(interrupt)가 스레드(thread) 선점을 구동하므로 다른 스레드(thread)가 실행 중인 스레드(thread)를 선점하지 못합니다. 인터럽트(interrupt)가 평소처럼 켜져 있으면 실행 중인 스레드(thread)는 두 C 문장 사이에서든 한 문장의 실행 중이든 언제든 다른 스레드(thread)에 의해 선점될 수 있습니다.

덧붙여, 이는 Pintos가 "선점 가능한 커널(preemptible kernel)"이라는 뜻입니다. 즉, 커널(kernel) 스레드(thread)는 언제든 선점될 수 있습니다. 전통적인 Unix 시스템은 "비선점형(nonpreemptible)"입니다. 즉, 커널(kernel) 스레드(thread)는 스케줄러(scheduler)를 명시적으로 호출하는 지점에서만 선점될 수 있습니다. 두 모델 모두에서 사용자 프로그램은 언제든 선점될 수 있습니다. 예상할 수 있듯 선점 가능한 커널(kernel)은 더 명시적인 동기화(synchronization)를 필요로 합니다.

인터럽트(interrupt) 상태를 직접 설정해야 할 필요는 거의 없습니다. 대부분의 경우 다음 절들에서 설명하는 다른 동기화(synchronization) 프리미티브를 사용해야 합니다. 인터럽트(interrupt)를 비활성화하는 주된 이유는 커널(kernel) 스레드(thread)를 외부 인터럽트(interrupt) 핸들러와 동기화하는 것입니다. 외부 인터럽트(interrupt) 핸들러는 sleep할 수 없으므로 대부분의 다른 동기화(synchronization) 형태를 사용할 수 없습니다.

일부 외부 인터럽트(interrupt)는 인터럽트(interrupt)를 비활성화해도 미룰 수 없습니다. 마스크 불가능 인터럽트(non-maskable interrupts)(NMIs)라고 불리는 이러한 인터럽트(interrupt)는 컴퓨터에 불이 나는 경우와 같은 비상 상황에서만 사용하도록 되어 있습니다. Pintos는 마스크 불가능 인터럽트(non-maskable interrupts)를 처리하지 않습니다.

인터럽트(interrupt)를 비활성화하고 활성화하기 위한 타입과 함수는 include/threads/interrupt.h에 있습니다.

enum intr_level;

INTR_OFF 또는 INTR_ON 중 하나이며, 각각 인터럽트(interrupt)가 비활성화되었거나 활성화되었음을 나타냅니다.

enum intr_level intr_get_level (void)

현재 인터럽트(interrupt) 상태를 반환합니다.

enum intr_level intr_set_level (enum intr_level level);

level에 따라 인터럽트(interrupt)를 켜거나 끕니다. 이전 인터럽트(interrupt) 상태를 반환합니다.

enum intr_level intr_enable (void);

인터럽트(interrupt)를 켭니다. 이전 인터럽트(interrupt) 상태를 반환합니다.

enum intr_level intr_disable (void);

인터럽트(interrupt)를 끕니다. 이전 인터럽트(interrupt) 상태를 반환합니다.

세마포어(Semaphores)

세마포어(semaphore)는 음이 아닌 정수와 그것을 원자적(atomic)으로 조작하는 두 연산자로 구성되며, 그 연산자는 다음과 같습니다.

  • "Down" 또는 "P": 값이 양수가 될 때까지 기다린 다음 값을 감소시킵니다.
  • "UP" 또는 "V": 값을 증가시키고, 기다리는 스레드(thread)가 있으면 하나를 깨웁니다.

0으로 초기화된 세마포어(semaphore)는 정확히 한 번 일어날 이벤트를 기다리는 데 사용할 수 있습니다. 예를 들어 스레드(thread) A가 다른 스레드(thread) B를 시작하고, B가 어떤 활동을 완료했다는 신호를 보내기를 기다리고 싶다고 합시다. A는 0으로 초기화된 세마포어(semaphore)를 만들어 B를 시작할 때 전달한 다음 그 세마포어(semaphore)에 "down"을 수행할 수 있습니다. B가 활동을 마치면 세마포어(semaphore)에 "up"을 수행합니다. A가 먼저 세마포어(semaphore)에 "down"을 수행하든 B가 먼저 "up"을 수행하든 이 방식은 동작합니다.

예:

struct semaphore sema;

/* Thread A */
void threadA (void) {
    sema_down (&sema);
}

/* Thread B */
void threadB (void) {
    sema_up (&sema);
}

/* main function */
void main (void) {
    sema_init (&sema, 0);
    thread_create ("threadA", PRI_MIN, threadA, NULL);
    thread_create ("threadB", PRI_MIN, threadB, NULL);
}

이 예에서 threadA는 sema_down ()에서 실행을 멈추며, threadB가 sema_up ()을 호출할 때까지 기다립니다.

1로 초기화된 세마포어(semaphore)는 일반적으로 자원에 대한 접근을 제어하는 데 사용됩니다. 코드 블록이 자원을 사용하기 시작하기 전에 세마포어(semaphore)에 "down"을 수행하고, 자원 사용이 끝난 뒤 그 자원에 "up"을 수행합니다. 이런 경우에는 아래에서 설명하는 락(lock)이 더 적절할 수 있습니다.

세마포어(semaphore)는 1보다 큰 값으로 초기화할 수도 있습니다. 이런 경우는 드물게 사용됩니다. 세마포어(semaphore)는 Edsger Dijkstra가 고안했고 THE 운영체제에서 처음 사용되었습니다. Pintos의 세마포어(semaphore) 타입과 연산은 include/threads/synch.h에 선언되어 있습니다.

struct semaphore;

세마포어(semaphore)를 나타냅니다.

void sema_init (struct semaphore *sema, unsigned value);

sema를 주어진 초기값을 가진 새 세마포어(semaphore)로 초기화합니다.

void sema_down (struct semaphore *sema);

sema에 대해 "down" 또는 "P" 연산을 수행하여, 값이 양수가 될 때까지 기다린 다음 값을 1 감소시킵니다.

bool sema_try_down (struct semaphore *sema);

기다리지 않고 sema에 대해 "down" 또는 "P" 연산을 수행하려고 시도합니다. sema가 성공적으로 감소되었으면 true를 반환하고, 이미 0이어서 기다리지 않고는 감소시킬 수 없으면 false를 반환합니다. 이 함수를 tight loop에서 호출하면 CPU 시간이 낭비되므로 대신 sema_down()을 사용하거나 다른 접근 방식을 찾으십시오.

void sema_up (struct semaphore *sema);

sema에 대해 "up" 또는 "V" 연산을 수행하여 값을 증가시킵니다. sema를 기다리는 스레드(thread)가 있으면 그중 하나를 깨웁니다. 대부분의 동기화(synchronization) 프리미티브와 달리 sema_up()은 외부 인터럽트(interrupt) 핸들러 안에서 호출될 수 있습니다.

세마포어(semaphore)는 내부적으로 인터럽트(interrupt) 비활성화 (인터럽트 비활성화(Disabling Interrupts) 참조)와 스레드(thread) 블록 및 언블록(thread_block()과 thread_unblock())으로 만들어집니다. 각 세마포어(semaphore)는 lib/kernel/list.c의 연결 리스트 구현을 사용해 기다리는 스레드(thread)들의 리스트를 유지합니다.

락(Locks)

락(lock)은 초기값이 1인 세마포어(semaphore)와 같습니다 (세마포어(Semaphores) 참조). 락(lock)에서 "up"에 해당하는 것은 "release"라고 부르고, "down" 연산은 "acquire"라고 부릅니다.

세마포어(semaphore)와 비교하면 락(lock)에는 제약이 하나 더 있습니다. 락(lock)을 획득한 스레드(thread), 즉 락(lock)의 "owner"만 그것을 해제할 수 있습니다. 이 제약이 문제가 된다면 락(lock) 대신 세마포어(semaphore)를 사용해야 한다는 좋은 신호입니다.

Pintos의 락(lock)은 "재귀적(recursive)"이지 않습니다. 즉, 현재 락(lock)을 보유한 스레드(thread)가 그 락(lock)을 다시 획득하려고 하면 오류입니다. 락(lock) 타입과 함수는 include/threads/synch.h에 선언되어 있습니다.

struct lock;

락(lock)을 나타냅니다.

void lock_init (struct lock *lock);

lock을 새 락(lock)으로 초기화합니다. 이 락(lock)은 처음에는 어떤 스레드(thread)도 소유하지 않습니다.

void lock_acquire (struct lock *lock);

현재 스레드(thread)가 lock을 획득합니다. 필요한 경우 현재 소유자가 그것을 해제할 때까지 먼저 기다립니다.

bool lock_try_acquire (struct lock *lock);

기다리지 않고 현재 스레드(thread)가 사용할 lock을 획득하려고 시도합니다. 성공하면 true를 반환하고, 락(lock)이 이미 소유되어 있으면 false를 반환합니다. 이 함수를 tight loop에서 호출하면 CPU 시간이 낭비되므로 좋지 않습니다. 대신 lock_acquire()를 사용하십시오.

void lock_release (struct lock *lock);

현재 스레드(thread)가 반드시 소유하고 있어야 하는 lock을 해제합니다.

bool lock_held_by_current_thread (const struct lock *lock):

실행 중인 스레드(thread)가 lock을 소유하면 true를, 그렇지 않으면 false를 반환합니다. 임의의 스레드(thread)가 락(lock)을 소유하는지 검사하는 함수는 없습니다. 호출자가 그 결과에 따라 행동하기 전에 답이 바뀔 수 있기 때문입니다.

모니터(Monitors)

모니터(monitor)는 세마포어(semaphore)나 락(lock)보다 더 높은 수준의 동기화(synchronization) 형태입니다. 모니터(monitor)는 동기화되는 데이터, 모니터 락(monitor lock)이라고 부르는 락(lock), 그리고 하나 이상의 조건 변수(condition variable)로 구성됩니다. 보호되는 데이터에 접근하기 전에 스레드(thread)는 먼저 모니터 락(monitor lock)을 획득합니다. 그러면 그 스레드(thread)는 "모니터 안에 있다(in the monitor)"고 말합니다. 모니터(monitor) 안에 있는 동안 스레드(thread)는 보호되는 모든 데이터를 제어하며, 자유롭게 검사하거나 수정할 수 있습니다. 보호된 데이터에 대한 접근이 끝나면 모니터 락(monitor lock)을 해제합니다.

조건 변수(condition variable)는 모니터(monitor) 안의 코드가 어떤 조건이 true가 될 때까지 기다릴 수 있게 합니다. 각 조건 변수(condition variable)는 예를 들어 "처리할 데이터가 도착했다" 또는 "사용자의 마지막 키 입력 이후 10초가 지났다" 같은 추상 조건과 연결됩니다. 모니터(monitor) 안의 코드가 조건이 true가 되기를 기다려야 할 때, 연결된 조건 변수(condition variable)에서 "wait"하며, 이는 락(lock)을 해제하고 조건이 signal될 때까지 기다립니다. 반대로 코드가 이러한 조건 중 하나를 true로 만들었다면, 대기자 하나를 깨우기 위해 조건에 "signal"하거나 모든 대기자를 깨우기 위해 조건에 "broadcast"합니다.

모니터(monitor)의 이론적 틀은 C.A.R.Hoare가 정리했습니다. 그 실제 사용법은 이후 Mesa 운영체제에 관한 논문에서 더 자세히 설명되었습니다. 조건 변수(condition variable) 타입과 함수는 include/threads/synch.h에 선언되어 있습니다.

struct condition;

조건 변수(condition variable)를 나타냅니다.

void cond_init (struct condition *cond);

cond를 새 조건 변수(condition variable)로 초기화합니다.

void cond_wait (struct condition *cond, struct lock *lock);

lock(모니터 락(monitor lock))을 원자적(atomic)으로 해제하고, 다른 코드 조각이 cond에 signal할 때까지 기다립니다. cond가 signal된 뒤에는 반환하기 전에 lock을 다시 획득합니다. 이 함수를 호출하기 전에 lock을 보유하고 있어야 합니다. signal을 보내는 것과 wait에서 깨어나는 것은 원자적(atomic) 연산이 아닙니다. 따라서 일반적으로 cond_wait()의 호출자는 wait가 끝난 뒤 조건을 다시 확인하고, 필요하다면 다시 기다려야 합니다. 예시는 다음 절을 참조하십시오.

void cond_signal (struct condition *cond, struct lock *lock);

cond(모니터 락(monitor lock) lock으로 보호됨)를 기다리는 스레드(thread)가 있으면, 이 함수는 그중 하나를 깨웁니다. 기다리는 스레드(thread)가 없으면 아무 동작도 수행하지 않고 반환합니다. 이 함수를 호출하기 전에 lock을 보유하고 있어야 합니다.

void cond_broadcast (struct condition *cond, struct lock *lock);

cond(모니터 락(monitor lock) lock으로 보호됨)를 기다리는 모든 스레드(thread)가 있으면 모두 깨웁니다. 이 함수를 호출하기 전에 lock을 보유하고 있어야 합니다.

모니터 예제(Monitor Example)

모니터(monitor)의 고전적인 예는 하나 이상의 "producer" 스레드(thread)가 문자를 쓰고 하나 이상의 "consumer" 스레드(thread)가 문자를 읽는 버퍼(buffer)를 다루는 것입니다. 이를 구현하려면 모니터 락(monitor lock) 외에도 not_full과 not_empty라고 부를 두 조건 변수(condition variable)가 필요합니다.

    char buf[BUF_SIZE];     /* Buffer. */
    size_t n = 0;         /* 0 <= n <= BUF SIZE: # of characters in buffer. */
    size_t head = 0;        /* buf index of next char to write (mod BUF SIZE). */
    size_t tail = 0;         /* buf index of next char to read (mod BUF SIZE). */
    struct lock lock;         /* Monitor lock. */
    struct condition not_empty; /* Signaled when the buffer is not empty. */
    struct condition not_full;     /* Signaled when the buffer is not full. */

    ...initialize the locks and condition variables...

    void put (char ch) {
      lock_acquire (&lock);
      while (n == BUF_SIZE)    /* Can't add to buf as long as it's full. */
        cond_wait (&not_full, &lock);
      buf[head++ % BUF_SIZE] = ch;    /* Add ch to buf. */
      n++;
      cond_signal (&not_empty, &lock);    /* buf can't be empty anymore. */
      lock_release (&lock);
    }

    char get (void) {
      char ch;
      lock_acquire (&lock);
      while (n == 0)        /* Can't read buf as long as it's empty. */
        cond_wait (&not_empty, &lock);
      ch = buf[tail++ % BUF_SIZE];    /* Get ch from buf. */
      n--;
      cond_signal (&not_full, &lock);    /* buf can't be full anymore. */
      lock_release (&lock);
    }

위 코드가 완전히 올바르려면 BUF_SIZE가 SIZE_MAX + 1을 나누어떨어지게 해야 한다는 점에 유의하십시오. 그렇지 않으면 head가 처음으로 0으로 wrap around될 때 실패합니다. 실제로는 BUF_SIZE를 보통 2의 거듭제곱으로 둡니다.

최적화 장벽(Optimization Barriers)

최적화 장벽(optimization barrier)은 장벽을 가로지르는 메모리 상태에 대해 컴파일러(compiler)가 가정을 하지 못하게 하는 특별한 문장입니다. 컴파일러(compiler)는 장벽을 가로질러 변수의 읽기나 쓰기를 재정렬하지 않으며, 주소가 한 번도 취해지지 않은 지역 변수를 제외하고는 장벽을 가로질러 변수 값이 변경되지 않았다고 가정하지 않습니다. Pintos에서 include/threads/synch.h는 barrier() 매크로를 최적화 장벽(optimization barrier)으로 정의합니다.

최적화 장벽(optimization barrier)을 사용하는 한 가지 이유는 컴파일러(compiler)가 모르는 사이에 데이터가 비동기적으로 변할 수 있을 때입니다. 예를 들어 다른 스레드(thread)나 인터럽트(interrupt) 핸들러가 데이터를 바꿀 수 있습니다. too_many_loops() 함수가 devices/timer.c에 있는 그 예입니다. 이 함수는 타이머(timer) tick이 발생할 때까지 루프에서 busy-wait하는 것으로 시작합니다.

    /* Wait for a timer tick. */
    int64_t start = ticks;
    while (ticks == start)
        barrier();

루프 안에 최적화 장벽(optimization barrier)이 없으면 컴파일러(compiler)는 루프가 절대 종료되지 않을 것이라고 결론 내릴 수 있습니다. start와 ticks가 같은 값으로 시작하고 루프 자체가 그것들을 결코 바꾸지 않기 때문입니다. 그러면 컴파일러(compiler)는 함수를 무한 루프로 "최적화"할 수 있는데, 이는 분명 바람직하지 않습니다.

최적화 장벽(optimization barrier)은 다른 컴파일러(compiler) 최적화를 피하는 데도 사용할 수 있습니다. busy_wait() 함수도 devices/timer.c에 있는 예입니다. 이 함수에는 다음 루프가 들어 있습니다.

    while (loops-- > 0)
      barrier ();

이 루프의 목적은 loops를 원래 값에서 0까지 세어 내려가며 busy-wait하는 것입니다. 장벽이 없으면 이 루프는 유용한 출력을 만들지 않고 부작용도 없기 때문에 컴파일러(compiler)가 루프 전체를 삭제할 수 있습니다. 장벽은 루프 본문에 중요한 효과가 있는 것처럼 컴파일러(compiler)가 가장하도록 강제합니다.

마지막으로, 최적화 장벽(optimization barrier)은 메모리 읽기나 쓰기의 순서를 강제하는 데 사용할 수 있습니다. 예를 들어 타이머(timer) 인터럽트(interrupt)가 발생할 때마다 전역 변수 timer_put_char의 문자를 콘솔에 출력하되, 전역 Boolean 변수 timer_do_put이 true일 때만 출력하는 "feature"를 추가한다고 합시다. 그러면 x가 출력되도록 설정하는 가장 좋은 방법은 다음처럼 최적화 장벽(optimization barrier)을 사용하는 것입니다.

    timer_put_char = 'x';
    barrier ();
    timer_do_put = true;

장벽이 없으면 컴파일러(compiler)는 같은 순서로 유지해야 할 이유를 보지 못할 때 연산을 자유롭게 재정렬할 수 있으므로 이 코드는 버그가 있습니다. 이 경우 컴파일러(compiler)는 대입 순서가 중요하다는 것을 모르므로 optimizer가 그 순서를 바꾸는 것이 허용됩니다. 실제로 그렇게 할지는 알 수 없으며, 컴파일러(compiler)에 다른 최적화 플래그를 전달하거나 다른 버전의 컴파일러(compiler)를 사용하면 다른 동작이 나올 수도 있습니다.

또 다른 해결책은 대입 주변에서 인터럽트(interrupt)를 비활성화하는 것입니다. 이것은 재정렬을 막지는 않지만, 인터럽트(interrupt) 핸들러가 대입 사이에 끼어드는 것을 막습니다. 또한 인터럽트(interrupt)를 비활성화하고 다시 활성화하는 추가 런타임 비용이 있습니다.

    enum intr_level old_level = intr_disable ();
    timer_put_char = 'x';
    timer_do_put = true;
    intr_set_level (old_level);

두 번째 해결책은 timer_put_char와 timer_do_put의 선언을 volatile로 표시하는 것입니다. 이 키워드는 해당 변수들이 외부에서 관찰 가능하다고 컴파일러(compiler)에 알리고 최적화의 자유도를 제한합니다. 그러나 volatile의 의미론은 잘 정의되어 있지 않으므로 좋은 일반 해법은 아닙니다. 기본 Pintos 코드는 volatile을 전혀 사용하지 않습니다.

다음은 해결책이 아닙니다. 락(lock)은 인터럽트(interrupt)를 막지도 않고, 락(lock)이 보유된 영역 안의 코드를 컴파일러(compiler)가 재정렬하는 것도 막지 않기 때문입니다.

    lock_acquire (&timer_lock);        /* INCORRECT CODE */
    timer_put_char = 'x';
    timer_do_put = true;
    lock_release (&timer_lock);

컴파일러(compiler)는 외부, 즉 다른 소스 파일에 정의된 함수의 호출을 제한된 형태의 최적화 장벽(optimization barrier)으로 취급합니다. 구체적으로 컴파일러(compiler)는 외부에 정의된 어떤 함수든 정적으로 또는 동적으로 할당된 데이터와 주소가 취해진 모든 지역 변수에 접근할 수 있다고 가정합니다. 이는 종종 명시적인 장벽을 생략할 수 있음을 뜻합니다. Pintos에 명시적인 장벽이 거의 없는 이유 중 하나입니다.

같은 소스 파일에 정의된 함수나 그 소스 파일에 포함된 헤더에 정의된 함수는 최적화 장벽(optimization barrier)으로 의존할 수 없습니다. 이는 정의보다 앞서 함수를 호출하는 경우에도 적용됩니다. 컴파일러(compiler)가 최적화를 수행하기 전에 전체 소스 파일을 읽고 파싱할 수 있기 때문입니다.
