디버깅 도구

디버깅 도구

Pintos를 디버깅(debugging)할 때 사용할 수 있는 도구는 많습니다. 이 부록에서는 그중 몇 가지를 소개합니다.

printf()

printf()의 가치를 과소평가하지 마십시오. Pintos에서 printf()가 구현된 방식상, 커널(kernel) 스레드(thread) 안이든 인터럽트(interrupt) 핸들러 안이든, 어떤 락(lock)을 잡고 있는지와 거의 무관하게 커널의 사실상 어디에서나 호출할 수 있습니다. printf()는 단순히 데이터를 들여다보는 용도 이상으로 유용합니다. 커널이 쓸모 있는 오류 메시지 없이 충돌하거나 패닉(panic)하더라도, 문제가 언제 어디서 발생하는지 파악하는 데 도움이 됩니다. 전략은 실패가 의심되는 코드 조각 전체에 서로 다른 문자열 (예: "<1>", "<2>", . . . )을 출력하는 printf() 호출을 흩뿌려 넣는 것입니다. <1>조차 출력되지 않는다면 그 지점 이전에 문제가 발생한 것이고, <1>은 보이지만 <2>가 보이지 않는다면 두 지점 사이에서 문제가 발생한 것입니다. 이런 식으로 알아낸 정보를 바탕으로, 새로 좁혀진 의심 코드 영역에 printf() 호출을 더 넣을 수 있습니다. 결국 문제를 단일 문장까지 좁힐 수 있습니다. 관련 기법은 트리플 폴트(Triple Faults)를 참조하십시오.

ASSERT

어설션(assertion)은 문제가 다른 방식으로 드러나기 전에 일찍 잡아낼 수 있어서 유용합니다. 이상적으로는 각 함수가 인자의 유효성을 검사하는 어설션 집합으로 시작해야 합니다. (함수의 지역 변수 초기화식은 어설션이 검사되기 전에 평가되므로, 초기화식에서 인자가 유효하다고 가정하지 않도록 주의하십시오.) 문제가 생길 가능성이 높다고 의심되는 위치마다 함수 본문 곳곳에 어설션을 넣어 둘 수도 있습니다. 어설션은 특히 루프 불변식(loop invariant)을 검사할 때 유용합니다.

Pintos는 어설션 검사용으로 ASSERT 매크로를 제공하며, 이 매크로는 <debug.h>에 정의되어 있습니다.

#define ASSERT(expression) { /* Omit details */ }

expression의 값을 검사합니다. 값이 0(false)으로 평가되면 커널(kernel)이 패닉(panic)합니다. 패닉 메시지에는 실패한 표현식, 해당 파일과 줄 번호, 문제를 찾는 데 도움이 되는 백트레이스(backtrace)가 포함됩니다. 자세한 내용은 백트레이스(Backtraces)를 참조하십시오.

함수 및 매개변수 속성

<debug.h>에 정의된 이 매크로들은 함수(function)나 함수 매개변수(function parameter)의 특수 속성을 컴파일러(compiler)에 알려 줍니다. 이들의 확장은 GCC 전용입니다.

#define UNUSED { /* Omit details */ }

함수 매개변수에 붙여, 해당 매개변수가 함수 안에서 사용되지 않을 수도 있음을 컴파일러에 알려 줍니다. 그렇지 않으면 나타날 경고를 억제합니다.

#define NO_RETURN { /* Omit details */ }

함수 프로토타입(function prototype)에 붙여, 해당 함수가 절대 반환하지 않는다는 사실을 컴파일러에 알려 줍니다. 이를 통해 컴파일러가 경고와 코드 생성을 더 정밀하게 조정할 수 있습니다.

#define NO_INLINE { /* Omit details */ }

함수 프로토타입에 붙여, 해당 함수를 절대 인라인(in-line)으로 내보내지 말라고 컴파일러에 알려 줍니다. 때때로 백트레이스(backtrace)의 품질을 높이는 데 유용합니다 (아래 참조).

#define PRINTF_FORMAT(format, first) { /* Omit details */ }

함수 프로토타입에 붙여, 해당 함수가 format번 인자(1부터 시작)를 printf()와 같은 형식 문자열로 받고, 그에 대응하는 값 인자들이 first번 인자부터 시작한다는 사실을 컴파일러에 알려 줍니다. 이렇게 하면 잘못된 인자 타입을 넘겼을 때 컴파일러가 알려 줄 수 있습니다.

백트레이스

커널(kernel)이 패닉(panic)하면 "백트레이스(backtrace)"를 출력합니다. 백트레이스는 패닉 시점에 실행 중이던 함수 안의 주소 목록으로, 프로그램이 현재 위치에 어떻게 도달했는지를 요약한 것입니다. 코드의 임의 지점에서 백트레이스를 출력하려면 debug_backtrace() 호출을 삽입할 수도 있으며, 이 함수의 프로토타입은 <debug.h>에 있습니다. debug_backtrace_all()도 <debug.h>에 선언되어 있으며, 모든 스레드(thread)의 백트레이스를 출력합니다.

백트레이스의 주소는 해석하기 어려운 원시 16진수로 나열됩니다. Pintos는 이 주소들을 함수 이름과 소스 파일 줄 번호로 변환하는 backtrace라는 도구를 제공합니다. 첫 번째 인자로 자신의 kernel.o 이름을 넘기고, 나머지 인자로는 백트레이스를 구성하는 16진수 숫자들(0x 접두사 포함)을 넘기십시오. 그러면 각 주소에 대응하는 함수 이름과 소스 파일 줄 번호를 출력합니다.

번역된 백트레이스가 깨져 있거나 말이 되지 않는다면(예: 함수 A가 함수 B보다 위에 나열되었지만 B가 A를 호출하지 않는 경우), 커널 스레드의 스택(stack)을 망가뜨리고 있을 가능성이 큽니다. 백트레이스는 스택에서 추출되기 때문입니다. 또는 backtrace에 넘긴 kernel.o가 그 백트레이스를 만든 커널과 다른 커널일 수도 있습니다.

손상이 전혀 없어도 백트레이스가 혼란스러울 때가 있습니다. 컴파일러 최적화(compiler optimization)는 예상 밖의 동작을 만들 수 있습니다. 어떤 함수가 마지막 동작으로 다른 함수를 호출한 경우(꼬리 호출, tail call), 호출한 함수가 백트레이스에 전혀 나타나지 않을 수 있습니다. 마찬가지로 함수 A가 반환하지 않는 다른 함수 B를 호출하면, 컴파일러는 A 대신 관련 없는 함수 C가 백트레이스에 나타나도록 최적화할 수 있습니다. 함수 C는 단지 A 바로 다음 메모리에 우연히 놓여 있던 함수일 뿐입니다. threads 프로젝트에서는 테스트 실패의 백트레이스에서 이런 현상이 흔히 보입니다.

예시

예를 들어 보겠습니다. Pintos가 다음 호출 스택(call stack)을 출력했다고 가정합시다. 이 예시는 파일 시스템 프로젝트의 실제 Pintos 제출물에서 가져온 것입니다.

Call stack: 0x800426eff 0x8004202fb 0x80042dc22 0x80042cf67 0x800422319
0x80042325a 0x40012c 0x400a96 0x400ac8.

그다음에는 아래와 같이 백트레이스 정보를 명령줄에 잘라 붙여 backtrace 유틸리티를 실행합니다. 여기서는 kernel.o가 현재 디렉터리(directory)에 있다고 가정합니다. 물론 아래 내용은 여기 지면 너비를 넘기더라도 실제로는 하나의 셸 명령줄에 모두 입력해야 합니다.

backtrace 0x800426eff 0x8004202fb 0x80042dc22 0x80042cf67 0x800422319
0x80042325a 0x400012c 0x400a96 0x4000ac8

그러면 백트레이스 출력은 대략 다음과 같은 모습이 됩니다.

0x0000000800426eff: debug_panic (lib/debug.c:86)
0x00000008004202fb: file_seek (filesys/file.c:405)
0x000000080042dc22: seek (userprog/syscall.c:744)
0x000000080042cf67: syscall_handler (userprog/syscall.c:444)
0x0000000800422319: intr_handler (threads/interrupt.c:334)
0x000000080042325a: intr_entry (threads/intr-stubs.S:38)
0x000000000040012c: (unknown)
0x0000000000400a96: (unknown)
0x0000000000400ac8: (unknown)

(자신의 커널(kernel) 바이너리에서 위 명령을 실행하면 아마 정확히 같은 주소가 보이지는 않을 것입니다. 컴파일한 소스 코드와 사용한 컴파일러가 다를 가능성이 높기 때문입니다.) 백트레이스의 첫 번째 줄은 커널 패닉을 구현하는 함수인 debug_panic()을 가리킵니다. 백트레이스는 보통 커널 패닉의 결과로 나오므로, debug_panic()은 백트레이스에서 첫 번째로 표시되는 함수인 경우가 많습니다. 두 번째 줄은 이 경우 어설션(assertion) 실패의 결과로 패닉한 함수가 file_seek()임을 보여 줍니다. 이 예시에서 사용한 소스 코드 트리에서는 filesys/file.c의 405번째 줄이 다음 어설션입니다.

ASSERT (file_ofs >= 0);

(이 줄은 어설션 실패 메시지에도 인용되었습니다.) 따라서 file_seek()은 음수 파일 오프셋(file offset) 인자를 받았기 때문에 패닉했습니다. 세 번째 줄은 seek()가 오프셋 인자를 검증하지 않은 채 file_seek()를 호출했을 가능성이 있음을 나타냅니다. 이 제출물에서 seek()는 seek 시스템 콜(system call)을 구현합니다. 네 번째 줄은 시스템 콜 핸들러인 syscall_handler()가 seek()를 호출했음을 보여 줍니다. 다섯 번째와 여섯 번째 줄은 인터럽트(interrupt) 핸들러 진입 경로입니다. 나머지 줄은 KERN_BASE 아래의 주소입니다. 이는 해당 주소들이 커널이 아니라 사용자 프로그램(user program)의 주소를 가리킨다는 뜻입니다.

GDB

GDB 디버거(debugger)의 제어 아래에서 Pintos를 실행할 수 있습니다. 먼저 --gdb 옵션으로 Pintos를 시작합니다. 예를 들어 pintos --gdb -- run mytest처럼 실행합니다. 그런 다음 같은 머신에서 두 번째 터미널을 열고 gdb를 사용해 'kernel.o'에 대해 GDB를 실행합니다.

gdb kernel.o

그리고 다음 GDB 명령을 입력합니다.

target remote localhost:1234

이제 GDB는 로컬 네트워크 연결을 통해 시뮬레이터에 연결되었습니다. 이제 일반적인 GDB 명령을 사용할 수 있습니다. 'c' 명령을 입력하면 시뮬레이션된 BIOS가 제어권을 잡고 Pintos를 로드한 뒤, Pintos가 평소와 같은 방식으로 실행됩니다. Ctrl+C로 언제든지 프로세스(process)를 일시 중지할 수 있습니다.

GDB 사용하기

터미널 명령 프롬프트에서 info gdb를 입력하면 GDB 매뉴얼을 읽을 수 있습니다. 다음은 자주 유용하게 쓰이는 몇 가지 GDB 명령입니다.

  • c [GDB 명령]

Ctrl+C 또는 다음 브레이크포인트(breakpoint)에 도달할 때까지 실행을 계속합니다.

  • break function [GDB 명령]

  • break file:line [GDB 명령]

  • break address [GDB 명령]

function, file 안의 line, 또는 address에 브레이크포인트를 설정합니다. (주소를 16진수로 지정하려면 '0x' 접두사를 사용하십시오.) Pintos가 실행을 시작할 때 GDB가 멈추게 하려면 break main을 사용하십시오.

  • p expression [GDB 명령]

주어진 expression을 평가하고 그 값을 출력합니다. expression에 함수 호출이 포함되어 있으면 그 함수가 실제로 실행됩니다.

  • l address [GDB 명령]

address 주변의 코드 몇 줄을 나열합니다. (주소를 16진수로 지정하려면 '0x' 접두사를 사용하십시오.)

  • bt [GDB 명령]

위에서 설명한 backtrace 프로그램의 출력과 비슷한 스택(stack) 백트레이스(backtrace)를 출력합니다.

  • p/a address [GDB 명령]

address를 차지하고 있는 함수나 변수의 이름을 출력합니다. (주소를 16진수로 지정하려면 '0x' 접두사를 사용하십시오.)

  • diassemble function [GDB 명령]

function을 디스어셈블(disassemble)합니다.

여기까지 읽은 독자를 위한 추가 팁입니다. pwndbg와 같은 gdb 스크립트를 설치하십시오. 작업이 훨씬 편해집니다.

자주 묻는 질문

  • Emacs 안에서 GDB를 사용할 수 있습니까?

예, 사용할 수 있습니다. Emacs는 GDB를 하위 프로세스(subprocess)로 실행하는 기능을 별도로 지원합니다. M-x gdb를 입력하고 프롬프트에 자신의 pintos-gdb 명령을 입력하십시오. Emacs 매뉴얼의 "Debuggers" 섹션에는 Emacs의 디버깅(debugging) 기능을 사용하는 방법이 설명되어 있습니다.

  • GDB가 이상하게 동작합니다.

GDB를 사용하는 동안 이상한 동작을 발견했다면 세 가지 가능성이 있습니다. 수정한 Pintos의 버그, Bochs의 GDB 인터페이스나 GDB 자체의 버그, 또는 원본 Pintos 코드의 버그입니다. 첫 번째와 두 번째 가능성은 꽤 높으므로 둘 다 진지하게 고려해야 합니다. 세 번째 가능성은 더 낮기를 바라지만, 역시 가능성은 있습니다.

트리플 폴트

페이지 폴트(page fault) 핸들러와 같은 CPU 예외(exception) 핸들러가 없거나 결함이 있어 호출될 수 없으면, CPU는 "더블 폴트(double fault)" 핸들러를 호출하려고 시도합니다. 더블 폴트 핸들러 자체도 없거나 결함이 있으면 이를 "트리플 폴트(triple fault)"라고 합니다. 트리플 폴트는 즉시 CPU 리셋을 일으킵니다.

따라서 머신이 루프를 돌며 계속 재부팅되는 상황에 빠졌다면, 이는 아마 "트리플 폴트"일 가능성이 높습니다. 트리플 폴트 상황에서는 printf()에 필요한 모든 것이 초기화되기도 전에 재부팅이 발생할 수 있으므로, printf()를 디버깅(debugging)에 사용하지 못할 수도 있습니다.

트리플 폴트를 디버깅하는 방법이 있습니다.

Pintos 코드에서 한 지점을 골라 그곳에 무한 루프 for (;;);를 삽입한 뒤, 다시 컴파일하고 실행하십시오. 가능성은 보통 두 가지입니다.

  • 머신이 재부팅하지 않고 멈춥니다. 이런 일이 발생하면 무한 루프가 실행 중임을 알 수 있습니다. 이는 재부팅을 일으킨 원인이 무한 루프를 삽입한 위치보다 뒤에 있어야 한다는 뜻입니다. 이제 무한 루프를 코드 순서상 더 뒤로 옮기십시오.

  • 머신이 루프를 돌며 재부팅됩니다. 이런 일이 발생하면 머신이 무한 루프까지 도달하지 못했음을 알 수 있습니다. 따라서 재부팅을 일으킨 원인은 무한 루프를 삽입한 위치보다 앞에 있어야 합니다. 이제 무한 루프를 코드 순서상 더 앞으로 옮기십시오.

    무한 루프를 "바이너리 서치(binary search)" 방식으로 옮기면, 이 기법으로 모든 것이 잘못되기 시작하는 정확한 지점을 찾아낼 수 있습니다. 길어도 몇 분밖에 걸리지 않을 것입니다.

팁

threads/palloc.c의 페이지 할당자(page allocator)와 threads/malloc.c의 블록 할당자(block allocator)는 해제 시점에 메모리의 모든 바이트를 '0xcc'로 지웁니다. 따라서 0xcccccccc 같은 포인터를 역참조하려는 시도나 0xcc에 대한 다른 참조가 보인다면, 이미 해제된 페이지를 다시 사용하려는 중일 가능성이 큽니다. 또한 바이트 0xcc는 "인터럽트(interrupt) 3 호출"을 의미하는 CPU opcode이므로, Interrupt 0x03 (#BP Breakpoint Exception) 같은 오류가 보인다면 Pintos가 해제된 페이지나 블록 안의 코드를 실행하려 했다는 뜻입니다. 표현식 sec_no < d->capacity에서 어설션 실패가 발생했다면, Pintos가 이미 닫히고 해제된 아이노드(inode)를 통해 파일에 접근하려 했다는 의미입니다. 아이노드를 해제하면 시작 섹터 번호가 0xcccccccc로 지워지는데, 이는 약 1.6 TB보다 작은 디스크에서는 유효한 섹터 번호가 아닙니다.
