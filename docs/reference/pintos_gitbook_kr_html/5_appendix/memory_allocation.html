메모리 할당(Memory Allocation)

Pintos에는 두 가지 메모리 할당자(memory allocator)가 있습니다. 하나는 페이지(page) 단위로 메모리를 할당하고, 다른 하나는 임의 크기의 블록(block)을 할당할 수 있습니다.

페이지 할당자(Page Allocator)

include/threads/palloc.h에 선언된 페이지 할당자(page allocator)는 페이지(page) 단위로 메모리를 할당합니다. 보통 한 번에 한 페이지씩 메모리를 할당하는 데 가장 자주 쓰이지만, 연속된 여러 페이지(contiguous pages)도 한 번에 할당할 수 있습니다.

페이지 할당자(page allocator)는 자신이 할당하는 메모리를 커널(kernel) 풀과 사용자(user) 풀이라는 두 풀(pool)로 나눕니다. 기본적으로 각 풀은 1MB를 초과하는 시스템 메모리의 절반씩을 받지만, 커널(kernel) 명령줄 옵션 ul로 이 분할을 바꿀 수 있습니다. 할당 요청은 두 풀 중 하나에서 메모리를 가져옵니다. 한 풀이 비어도 다른 풀에는 여전히 빈 페이지가 있을 수 있습니다. 사용자 풀은 사용자 프로세스(user process)를 위한 메모리 할당에 사용하고, 커널 풀은 그 밖의 모든 할당에 사용해야 합니다. 이는 프로젝트 3부터 중요해집니다. 그 전까지는 모든 할당을 커널 풀에서 수행해야 합니다.

각 풀의 사용 상태는 풀 안의 페이지마다 한 비트씩 사용하는 비트맵(bitmap)으로 추적됩니다. n개 페이지를 할당하라는 요청은 비트맵에서 false로 설정된 연속된 n개 비트를 찾습니다. 이는 해당 페이지들이 비어 있음을 뜻하며, 그런 다음 그 비트들을 true로 설정해 사용 중으로 표시합니다. 이것은 "최초 적합(first fit)" 할당 전략입니다.

페이지 할당자(page allocator)는 단편화(fragmentation)의 영향을 받습니다. 즉, 빈 페이지가 n개 이상 있더라도 그 빈 페이지들이 사용 중인 페이지 사이에 떨어져 있으면 연속된 n개 페이지를 할당하지 못할 수 있습니다. 실제로 병적인 경우에는 풀의 절반이 비어 있어도 연속된 2개 페이지를 할당하지 못할 수 있습니다. 단일 페이지 요청은 단편화 때문에 실패하지 않으므로, 연속된 여러 페이지 요청은 가능한 한 제한해야 합니다.

인터럽트 컨텍스트(interrupt context)에서는 페이지를 할당할 수 없지만, 해제할 수는 있습니다.

페이지가 해제되면 디버깅(debugging)을 돕기 위해 모든 바이트가 0xcc로 지워집니다 (디버깅 팁(Debugging Tips) 참조).

페이지 할당자(page allocator)의 타입과 함수는 아래에 설명되어 있습니다.

void *palloc_get_page (enum palloc_flags flags)
void *palloc_get_multiple (enum palloc_flags flags, size_t page_cnt)

각각 하나의 페이지 또는 page_cnt개의 연속 페이지를 얻어 반환합니다. 페이지를 할당할 수 없으면 null 포인터를 반환합니다.

flags 인자에는 다음 플래그들의 임의 조합을 지정할 수 있습니다.

  • PAL_ASSERT

    페이지를 할당할 수 없으면 커널(kernel)을 패닉(panic)시킵니다. 이는 커널 초기화 중에만 적절합니다. 사용자 프로세스(user process)가 커널을 패닉시킬 수 있어서는 안 됩니다.

  • PAL_ZERO

    반환하기 전에 할당된 페이지의 모든 바이트를 0으로 초기화합니다. 이 플래그가 설정되지 않으면 새로 할당된 페이지의 내용은 예측할 수 없습니다.

  • PAL_USER

    사용자(user) 풀에서 페이지를 얻습니다. 설정되지 않으면 커널(kernel) 풀에서 페이지를 할당합니다.

void palloc_free_page (void *page)
void palloc_free_multiple (void *pages, size_t page_cnt)

각각 pages에서 시작하는 하나의 페이지 또는 page_cnt개의 연속 페이지를 해제합니다. 모든 페이지는 palloc_get_page() 또는 palloc_get_multiple()를 사용해 얻은 것이어야 합니다.

블록 할당자(Block Allocator)

threads/malloc.h에 선언된 블록 할당자(block allocator)는 임의 크기의 블록(block)을 할당할 수 있습니다. 이는 앞 절에서 설명한 페이지 할당자(page allocator) 위에 계층화되어 있습니다. 블록 할당자가 반환하는 블록은 커널(kernel) 풀에서 얻습니다.

블록 할당자(block allocator)는 메모리 할당에 두 가지 다른 전략을 사용합니다. 첫 번째 전략은 1 kB 이하의 블록(페이지 크기의 1/4)에 적용됩니다. 이러한 할당은 가장 가까운 2의 거듭제곱 또는 16바이트 중 더 큰 값으로 올림됩니다. 그런 다음 그 크기의 할당에만 사용되는 페이지 하나에 함께 배치됩니다.

두 번째 전략은 1 kB보다 큰 블록에 적용됩니다. 이러한 할당은 약간의 오버헤드(overhead)를 더한 뒤 크기가 가장 가까운 페이지 단위로 올림되고, 블록 할당자는 페이지 할당자(page allocator)에 그 수만큼의 연속 페이지를 요청합니다.

어느 경우든 요청한 할당 크기와 실제 블록 크기의 차이는 낭비됩니다. 실제 운영체제라면 이 낭비를 최소화하도록 할당자(allocator)를 세심하게 조정하겠지만, Pintos 같은 교육용 시스템에서는 중요하지 않습니다.

페이지 할당자(page allocator)에서 페이지 하나를 얻을 수 있는 한, 작은 할당은 항상 성공합니다. 대부분의 작은 할당은 이미 할당된 페이지의 일부로 충족되므로 페이지 할당자에서 새 페이지를 전혀 요구하지 않습니다. 그러나 큰 할당은 항상 페이지 할당자를 호출해야 하며, 앞 절에서 이미 설명했듯 연속된 페이지를 둘 이상 필요로 하는 할당은 단편화(fragmentation) 때문에 실패할 수 있습니다. 따라서 여러분의 코드에서는 큰 할당, 특히 각각 약 4 kB를 넘는 할당의 수를 최소화해야 합니다.

블록이 해제되면 디버깅(debugging)을 돕기 위해 모든 바이트가 0xcc로 지워집니다 (디버깅 팁(Debugging Tips) 참조).

블록 할당자(block allocator)는 인터럽트 컨텍스트(interrupt context)에서 호출할 수 없습니다.

블록 할당자(block allocator) 함수는 아래에 설명되어 있습니다. 이 함수들의 인터페이스는 같은 이름의 표준 C 라이브러리 함수와 같습니다.

void *malloc (size_t size)

커널(kernel) 풀에서 길이가 적어도 size바이트인 새 블록을 얻어 반환합니다. size가 0이거나 사용할 수 있는 메모리가 없으면 null 포인터를 반환합니다.

void *calloc (size_t a, size_t b)

커널(kernel) 풀에서 길이가 적어도 a * b바이트인 새 블록을 얻어 반환합니다. 블록의 내용은 0으로 지워집니다. a 또는 b가 0이거나 메모리가 부족하면 null 포인터를 반환합니다.

void *realloc (void *block, size_t new_size)

block을 new_size바이트 크기로 조정하려고 시도하며, 그 과정에서 블록이 이동될 수 있습니다. 성공하면 새 블록을 반환하며, 이 경우 이전 블록에는 더 이상 접근해서는 안 됩니다. 실패하면 null 포인터를 반환하고 이전 블록은 여전히 유효합니다. block이 null인 호출은 malloc()과 같고, new_size가 0인 호출은 free()와 같습니다.

void free (void *block)

block을 해제합니다. 이 블록은 이전에 malloc(), calloc() 또는 realloc()가 반환했으며 아직 해제되지 않은 것이어야 합니다.
