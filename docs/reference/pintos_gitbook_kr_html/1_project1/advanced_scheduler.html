고급 스케줄러(Advanced Scheduler)

고급 스케줄러(Advanced Scheduler)

시스템에서 실행 중인 작업의 평균 응답 시간을 줄이기 위해 4.4BSD 스케줄러(4.4BSD scheduler)와 유사한 다단계 피드백 큐 스케줄러(multilevel feedback queue scheduler)를 구현하십시오.

우선순위 스케줄러(priority scheduler)와 마찬가지로, 고급 스케줄러는 우선순위를 기준으로 실행할 스레드(thread)를 선택합니다. 그러나 고급 스케줄러는 우선순위 기부(priority donation)를 수행하지 않습니다. 따라서 고급 스케줄러 작업을 시작하기 전에, 우선순위 기부를 제외하고라도 우선순위 스케줄러가 동작하도록 만들어 두는 것을 권장합니다.

스케줄링 알고리즘 정책을 Pintos 시작 시점에 선택할 수 있도록 코드를 작성해야 합니다. 기본적으로 우선순위 스케줄러가 활성화되어야 하지만, -mlfqs 커널(kernel) 옵션으로 4.4BSD 스케줄러를 선택할 수 있어야 합니다. 이 옵션을 전달하면 thread_mlfqs가 true로 설정됩니다. 이 변수는 threads/thread.h에 선언되어 있으며, 옵션이 parse_options()에 의해 파싱될 때 설정됩니다. 이 파싱은 main() 초기에 일어납니다.

4.4BSD 스케줄러가 활성화되면 스레드들은 더 이상 자신의 우선순위를 직접 제어하지 않습니다. thread_create()에 전달되는 우선순위 인자는 무시되어야 하며, thread_set_priority() 호출도 모두 무시되어야 합니다. thread_get_priority()는 스케줄러가 설정한 스레드의 현재 우선순위를 반환해야 합니다. 고급 스케줄러는 이후 어떤 프로젝트에서도 사용되지 않습니다.

4.4BSD 스케줄러(4.4BSD Scheduler)

범용 스케줄러의 목표는 스레드들의 서로 다른 스케줄링 요구를 균형 있게 만족시키는 것입니다. 많은 I/O를 수행하는 스레드는 입력 및 출력 장치를 계속 바쁘게 유지하기 위해 빠른 응답 시간이 필요하지만, CPU 시간은 거의 필요하지 않습니다. 반면 계산 중심(compute-bound) 스레드는 작업을 끝내기 위해 많은 CPU 시간을 받아야 하지만, 빠른 응답 시간은 필요하지 않습니다. 다른 스레드들은 그 중간 어딘가에 있으며, I/O 구간 사이에 계산 구간이 끼어드는 형태라서 시간에 따라 요구 사항이 달라집니다. 잘 설계된 스케줄러는 이런 모든 요구를 가진 스레드들을 동시에 수용할 수 있는 경우가 많습니다.

프로젝트 1에서는 이 부록에 설명된 스케줄러를 구현해야 합니다. 우리가 사용하는 스케줄러는 [McKusick]에 설명된 스케줄러와 유사하며, 이는 다단계 피드백 큐 스케줄러의 한 예입니다. 이 유형의 스케줄러는 실행 준비가 된 (ready-to-run) 스레드들의 큐를 여러 개 유지하며, 각 큐에는 서로 다른 우선순위를 가진 스레드들이 들어갑니다. 어떤 시점이든 스케줄러는 비어 있지 않은 큐 중 우선순위가 가장 높은 큐에서 스레드를 하나 선택합니다. 가장 높은 우선순위 큐에 여러 스레드가 들어 있다면, 이들은 "라운드 로빈(round robin)" 순서로 실행됩니다.

스케줄러의 여러 부분은 일정 수의 타이머 틱(timer tick)이 지난 뒤 데이터를 갱신해야 합니다. 모든 경우에 이러한 갱신은 일반 커널(kernel) 스레드가 실행될 기회를 얻기 전에 일어나야 합니다. 그래야 커널 스레드가 새로 증가한 timer_ticks() 값은 보지만 오래된 스케줄러 데이터 값은 보는 상황이 생기지 않습니다.

4.4BSD 스케줄러에는 우선순위 기부가 포함되지 않습니다.

나이스 값(Niceness)

스레드 우선순위는 아래에 주어진 공식을 사용하여 스케줄러가 동적으로 결정합니다. 그러나 각 스레드에는 다른 스레드에 얼마나 "친절(nice)"해야 하는지를 결정하는 정수 nice 값도 있습니다. nice 값이 0이면 스레드 우선순위에 영향을 주지 않습니다. 양수 nice 값은 최대 20까지 가능하며, 스레드의 우선순위를 낮추고 그렇지 않았다면 받을 CPU 시간 일부를 양보하게 합니다. 반면 음수 nice 값은 최소 -20까지 가능하며, 다른 스레드들로부터 CPU 시간을 빼앗아 오는 경향이 있습니다.

초기 스레드는 nice 값 0으로 시작합니다. 다른 스레드들은 부모 스레드에서 상속받은 nice 값으로 시작합니다. 아래 설명된 함수들을 구현해야 하며, 이 함수들은 테스트 프로그램에서 사용됩니다. 이 함수들의 골격(skeleton) 정의는 threads/thread.c에 제공되어 있습니다.

int thread_get_nice (void);

현재 스레드의 nice 값을 반환합니다.

int thread_set_nice (int nice);

현재 스레드의 nice 값을 새 nice 값으로 설정하고, 새 값을 기준으로 스레드의 우선순위를 다시 계산합니다 (우선순위 계산(Calculating Priority) 참조). 실행 중인 스레드가 더 이상 최고 우선순위를 갖지 않으면 양보합니다.

우선순위 계산(Calculating Priority)

우리 스케줄러에는 64개의 우선순위가 있고 따라서 64개의 준비 큐(ready queue)가 있으며, 번호는 0 (PRI_MIN)부터 63 (PRI_MAX)까지입니다. 숫자가 낮을수록 우선순위도 낮으므로, 우선순위 0은 최저 우선순위이고 우선순위 63은 최고 우선순위입니다. 스레드 우선순위는 처음에 스레드 초기화 시 계산됩니다. 또한 모든 스레드에 대해 매 네 번째 클록 틱마다 한 번씩 다시 계산됩니다. 어느 경우든 다음 공식으로 결정됩니다:

priority = PRI_MAX - (recent_cpu / 4) - (nice * 2),

여기서 recent cpu는 스레드가 최근 사용한 CPU 시간의 추정치이고(아래 참조), nice는 스레드의 nice 값입니다. 결과는 가장 가까운 정수로 내림해야 합니다(잘라냄). recent cpu와 nice에 각각 적용되는 계수 1/4와 2는 실제로 잘 동작하는 것으로 확인되었지만 더 깊은 의미는 없습니다. 계산된 우선순위는 항상 유효 범위 PRI_MIN에서 PRI_MAX 안에 있도록 조정됩니다.

이 공식은 최근 CPU 시간을 받은 스레드가 다음에 스케줄러가 실행될 때 CPU를 다시 배정받는 우선순위를 낮춥니다. 이것은 기아(starvation)를 방지하는 핵심입니다. 최근에 CPU 시간을 전혀 받지 못한 스레드는 recent cpu가 0이 되므로, 높은 nice 값이 없다면 곧 CPU 시간을 받을 수 있게 됩니다.

계산: recent_cpu

recent cpu는 각 프로세스(process)가 "최근(recently)" 얼마나 많은 CPU 시간을 받았는지를 측정해야 합니다. 또한 개선점으로, 더 최근의 CPU 시간에는 덜 최근의 CPU 시간보다 더 큰 가중치를 주어야 합니다. 한 가지 접근은 n개 원소 배열을 사용해 마지막 n초 각각에서 받은 CPU 시간을 추적하는 것입니다. 그러나 이 방식은 스레드마다 O(n) 공간이 필요하고 새 가중 평균을 계산할 때마다 O(n) 시간이 필요합니다.

대신, 우리는 다음과 같은 일반 형태의 지수 가중 이동 평균(exponentially weighted moving average)을 사용합니다:

x(0) = f(0),

x(t) = ax(t − 1) + (1 − a)f(t),

a = k/(k + 1),

여기서 x(t)는 정수 시간 t ≥ 0에서의 이동 평균이고, f(t)는 평균을 내는 함수이며, k > 0은 감쇠율을 제어합니다. 이 공식을 몇 단계 전개하면 다음과 같습니다.

x(1) = f(1),

x(2) = af(1) + f(2),

...

x(5) = a^4 * f(1) + a^3 * f(2) + a^2 * f(3) + a * f(4) + f(5).

f(t) 값은 시간 t에서 가중치 1, 가중치 a를 시간 t + 1에서, a^2를 시간 t + 2에서 갖는 식입니다. 또한 x(t)를 k와 연결해 볼 수도 있습니다. f(t)는 대략 1/e의 가중치를 시간 t + k에서, 대략 1/(e^2)의 가중치를 시간 t + 2k에서 갖는 식입니다. 반대 방향으로 보면, f(t)는 가중치 w로 시간 t + log_a(w)에서 감쇠합니다.

처음 생성된 스레드에서 recent_cpu의 초기값은 0이고, 다른 새 스레드에서는 부모의 값입니다. 타이머 인터럽트(timer interrupt)가 발생할 때마다, 유휴 스레드(idle thread)가 실행 중인 경우를 제외하고 실행 중인 스레드에 대해서만 recent_cpu가 1 증가합니다. 또한 1초마다 모든 스레드(실행 중, 준비 상태, 블록 상태 여부와 관계없이)에 대해 recent cpu 값이 다음 공식으로 다시 계산됩니다.

recent_cpu = (2 * load_avg)/(2 * load_avg + 1) * recent_cpu + nice

여기서 load_avg는 실행 준비가 된 스레드 수의 이동 평균입니다(아래 참조). load_avg가 1이면, 평균적으로 단일 스레드가 CPU를 두고 경쟁하고 있음을 의미하며, 이때 recent cpu의 현재 값은 가중치 .1로 log_(2/3) .1 ≈ 6초 안에 감쇠합니다. load avg가 2이면 가중치 .1로 감쇠하는 데 log_(3/4) .1 ≈ 8초가 걸립니다. 그 결과 recent cpu는 스레드가 "최근(recently)" 받은 CPU 시간의 양을 추정하되, 감쇠율은 CPU를 두고 경쟁하는 스레드 수에 반비례하게 되는 것입니다.

일부 테스트의 가정 때문에 recent cpu의 이러한 재계산은 시스템 틱 카운터가 정확히 1초의 배수에 도달했을 때, 즉 timer_ticks () % TIMER_FREQ == 0일 때만 수행되어야 하며, 다른 시점에는 수행되어서는 안 됩니다.

recent_cpu 값은 nice 값이 음수인 스레드에서 음수가 될 수 있습니다. 음수 recent_cpu를 0으로 고정(clamp)하지 마십시오.

이 공식에서 계산 순서를 생각해 볼 필요가 있을 수 있습니다. recent cpu의 계수를 먼저 계산한 다음 곱하는 것을 권장합니다. 일부 학생들은 load_avg에 recent_cpu를 직접 곱하면 오버플로(overflow)가 발생할 수 있다고 보고했습니다.

thread_get_recent_cpu()를 구현해야 하며, 이에 대한 골격은 threads/thread.c에 있습니다.

int thread_get_recent_cpu (void);

현재 스레드의 recent cpu 값에 100을 곱한 값을 가장 가까운 정수로 반올림하여 반환합니다

계산: load_avg

마지막으로, 흔히 시스템 부하 평균(system load average)이라고 알려진 부하 평균(load avg)은 지난 1분 동안 실행 준비가 된 스레드 수의 평균을 추정합니다. recent_cpu와 마찬가지로 지수 가중 이동 평균입니다. 우선순위(priority)와 recent_cpu와 달리, load_avg는 스레드별 값이 아니라 시스템 전체 값입니다. 시스템 부팅 시 0으로 초기화됩니다. 그 이후에는 1초마다 다음 공식에 따라 갱신됩니다:

load_avg = (59/60) * load_avg + (1/60) * ready_threads,

여기서 ready threads(준비 스레드 수)는 갱신 시점에 실행 중이거나 실행 준비가 된 스레드 수입니다(유휴 스레드는 제외).

일부 테스트의 가정 때문에 load_avg는 시스템 틱 카운터가 정확히 1초의 배수에 도달했을 때, 즉 timer_ticks () % TIMER_FREQ == 0일 때만 갱신되어야 하며, 다른 시점에는 갱신되어서는 안 됩니다. thread_get_load_avg()를 구현해야 하며, 이에 대한 골격은 threads/thread.c에 있습니다.

int thread_get_load_avg (void)

현재 시스템 부하 평균에 100을 곱한 값을 가장 가까운 정수로 반올림하여 반환합니다.

요약(Summary)

다음 공식들은 스케줄러를 구현하는 데 필요한 계산을 요약합니다. 스케줄러 요구 사항의 완전한 설명은 아닙니다.

모든 스레드는 직접 제어할 수 있는 -20에서 20 사이의 nice 값을 가집니다. 또한 각 스레드는 0(PRI_MIN)부터 63(PRI_MAX)까지의 우선순위를 가지며, 이 우선순위는 매 네 번째 틱마다 다음 공식으로 다시 계산됩니다:

priority = PRI_MAX - (recent_cpu / 4) - (nice * 2)

recent_cpu는 스레드가 "최근(recently)" 받은 CPU 시간의 양을 측정합니다. 각 타이머 틱마다 실행 중인 스레드의 recent cpu가 1 증가합니다. 1초마다 모든 스레드의 recent cpu는 다음과 같이 갱신됩니다:

recent_cpu = (2 * load_avg) / (2 * load_avg + 1) * recent_cpu + nice

load_avg는 지난 1분 동안 실행 준비가 된 스레드 수의 평균을 추정합니다. 부팅 시 0으로 초기화되고 1초마다 다음과 같이 다시 계산됩니다:

load_avg = (59/60) * load_avg + (1/60) * ready_threads

여기서 ready_threads는 갱신 시점에 실행 중이거나 실행 준비가 된 스레드 수입니다(유휴 스레드는 제외).

고정소수점 실수 연산(Fixed-Point Real Arithmetic)

위 공식에서 priority, nice, ready_threads는 정수이지만 recent_cpu와 load_avg는 실수입니다. 불행히도 Pintos는 커널에서 부동소수점 연산을 지원하지 않습니다. 커널을 복잡하게 만들고 느리게 만들기 때문입니다. 실제 커널도 같은 이유로 같은 제한을 두는 경우가 많습니다. 이는 실수량에 대한 계산을 정수를 사용해 시뮬레이션해야 함을 의미합니다. 어렵지는 않지만, 많은 학생들이 그 방법을 알지 못합니다. 이 절은 기본 사항을 설명합니다.

기본 아이디어는 정수의 오른쪽 끝 비트들을 소수부를 나타내는 데 사용하는 것입니다. 예를 들어, 부호 있는 32비트 정수의 가장 낮은 14비트를 소수부 비트로 지정하면, 정수 x는 실수 x/(2^14)를 나타냅니다. 이것을 17.14 고정소수점 수 표현이라고 부릅니다. 소수점 앞에 17비트, 뒤에 14비트, 그리고 부호 비트 1개가 있기 때문입니다. 17.14 형식의 수는 최대 (2^31 − 1)/(2^14) ≈ 131,071.999 값을 나타냅니다.

p.q 고정소수점 형식을 사용한다고 가정하고 f = 2^q라고 둡시다. 위 정의에 따라, 정수나 실수를 p.q 형식으로 변환하려면 f를 곱하면 됩니다. 예를 들어 17.14 형식에서 분수 59/60(위의 load_avg 계산에 사용됨)은 (59/60)2^14 = 16,110입니다. 고정소수점 값을 다시 정수로 변환하려면 f로 나눕니다. (C의 일반 / 연산자는 0 방향으로 잘라냅니다. 즉, 양수는 내리고 음수는 올립니다. 가장 가까운 정수로 반올림하려면 나누기 전에 양수에는 f / 2를 더하고 음수에서는 빼십시오.)

고정소수점 수에 대한 많은 연산은 간단합니다. x와 y를 고정소수점 수, n을 정수라고 합시다. 그러면 x와 y의 합은 x + y이고 차는 x - y입니다. x와 n의 합은 x + n * f, 차는 x - n * f, 곱은 x * n, 몫은 x / n입니다.

두 고정소수점 값을 곱할 때는 두 가지 문제가 있습니다. 첫째, 결과의 소수점이 왼쪽으로 q비트만큼 너무 멀리 가 있습니다. (59/60)(59/60)는 1보다 약간 작아야 하지만, 16,111 × 16,111 = 259,564,321은 2^14 = 16,384보다 훨씬 큽니다. q비트 오른쪽으로 시프트하면 259,564,321/2^14 = 15,842, 즉 약 0.97이 되어 올바른 답을 얻습니다. 둘째, 답을 표현할 수 있더라도 곱셈에서 오버플로가 발생할 수 있습니다. 예를 들어 17.14 형식의 64는 64 × 2^14 = 1,048,576이고 그 제곱 64^2 = 4,096은 17.14 범위 안에 충분히 들어오지만, 1,048,576^2 = 2^40은 최대 부호 있는 32비트 정수 값 2^31 − 1보다 큽니다. 간단한 해결책은 곱셈을 64비트 연산으로 수행하는 것입니다. 그러면 x와 y의 곱은 ((int64_t) x) * y / f입니다.

두 고정소수점 값을 나눌 때는 반대 문제가 있습니다. 소수점이 오른쪽으로 너무 멀리 가므로, 나누기 전에 피제수(dividend)를 왼쪽으로 q비트 시프트하여 이를 보정합니다. 왼쪽 시프트는 피제수의 상위 q비트를 버리는데, 이것도 64비트로 나눗셈을 수행하면 다시 해결할 수 있습니다. 따라서 x를 y로 나눈 몫은 ((int64_t) x) * f / y입니다.

이 절에서는 q비트 시프트 대신 일관되게 f를 곱하거나 나누는 방식을 사용했습니다. 이유는 두 가지입니다. 첫째, 곱셈과 나눗셈에는 C 시프트 연산자의 예상 밖 연산자 우선순위 문제가 없습니다. 둘째, 곱셈과 나눗셈은 음수 피연산자에 대해 잘 정의되어 있지만, C 시프트 연산자는 그렇지 않습니다. 구현할 때 이러한 문제에 주의하십시오.

다음 표는 고정소수점 산술 연산을 C에서 구현하는 방법을 요약합니다. 표에서 x와 y는 고정소수점 수이고, n은 정수이며, 고정소수점 수는 부호 있는 p.q 형식이고 p + q = 31이며, f는 1 << q입니다.

n을 고정소수점으로 변환 n * f
