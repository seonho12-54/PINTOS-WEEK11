소개

프로젝트 1: 스레드(Threads)

이번 과제에서는 최소한으로 동작하는 스레드(thread) 시스템을 제공합니다. 여러분의 일은 이 시스템의 기능을 확장하여 동기화(synchronization) 문제를 더 잘 이해하는 것입니다. 이번 과제에서는 주로 threads 디렉터리(directory)에서 작업하고, 일부 작업은 devices 디렉터리에서 하게 됩니다. 컴파일은 threads 디렉터리에서 해야 합니다. 이 프로젝트 설명을 읽기 전에 적어도 동기화(Synchronization) 자료를 훑어보아야 합니다.

배경

스레드 이해하기

첫 단계는 초기 스레드(thread) 시스템의 코드를 읽고 이해하는 것입니다. Pintos는 이미 스레드 생성과 스레드 종료, 스레드 사이를 전환하는 간단한 스케줄러(scheduler), 그리고 동기화 기본 요소(synchronization primitives) (세마포어(semaphore), 락(lock), 조건 변수(condition variable), 최적화 배리어(optimization barrier))를 구현해 두었습니다.

이 코드 중 일부는 약간 낯설게 보일 수 있습니다. 아직 기본 시스템을 컴파일하고 실행해 보지 않았다면, 소개에서 설명한 대로 지금 해 보아야 합니다. 소스 코드의 일부를 읽어 보며 무슨 일이 일어나는지 확인할 수 있습니다. 원한다면 거의 아무 곳에나 printf() 호출을 추가한 뒤 다시 컴파일하고 실행해서 어떤 일이 어떤 순서로 일어나는지 볼 수 있습니다. 또한 디버거(debugger) 안에서 커널(kernel)을 실행하고 흥미로운 지점에 브레이크포인트(breakpoint)를 걸어, 코드를 한 단계씩 실행하고 데이터를 검사하는 등의 작업도 할 수 있습니다.

스레드(thread)가 생성될 때 여러분은 스케줄될 새 컨텍스트(context)를 만드는 것입니다. 이 컨텍스트에서 실행할 함수를 thread_create()의 인자로 제공합니다. 해당 스레드가 처음 스케줄되어 실행될 때는 그 함수의 시작 지점부터 실행을 시작하고 그 컨텍스트 안에서 동작합니다. 함수가 반환되면 스레드는 종료됩니다. 따라서 각 스레드는 Pintos 안에서 실행되는 작은 프로그램처럼 동작하며, thread_create()에 전달된 함수는 main()처럼 동작합니다.

어느 시점이든 정확히 하나의 스레드(thread)만 실행되고, 나머지가 있다면 비활성 상태가 됩니다. 스케줄러(scheduler)는 다음에 실행할 스레드를 결정합니다. (어느 시점에도 실행 준비가 된 스레드가 없다면, idle()에 구현된 특수한 idle 스레드가 실행됩니다.) 동기화 기본 요소는 한 스레드가 다른 스레드가 어떤 일을 하기를 기다려야 할 때 컨텍스트 전환(context switch)을 강제할 수 있습니다.

컨텍스트 전환(context switch)의 세부 동작은 thread_launch()에 있으며, threads/thread.c 안에 있습니다(반드시 이해할 필요는 없습니다). 이 코드는 현재 실행 중인 스레드의 상태를 저장하고, 전환할 대상 스레드의 상태를 복원합니다.

GDB 디버거를 사용해 컨텍스트 전환을 천천히 추적하면서 어떤 일이 일어나는지 확인해 보십시오 (GDB 참조). 시작 지점으로 schedule()에 브레이크포인트를 설정한 다음, 거기서부터 한 단계씩 실행할 수 있습니다. 각 스레드의 주소와 상태, 그리고 각 스레드의 호출 스택(call stack)에 어떤 프로시저(procedure)가 있는지 반드시 추적하십시오. 한 스레드가 iret을 do_iret() 안에서 실행하면 다른 스레드가 실행을 시작한다는 점을 알 수 있습니다.

경고: Pintos에서는 각 스레드(thread)에 크기가 4 kB에 조금 못 미치는 작고 고정된 실행 스택(stack)이 할당됩니다. 커널(kernel)은 스택 오버플로(stack overflow)를 감지하려고 하지만 완벽하게 감지할 수는 없습니다. 예를 들어 int buf[1000];처럼 큰 자료 구조를 non-static 지역 변수로 선언하면, 원인을 알기 어려운 커널 패닉(kernel panic) 같은 이상한 문제가 생길 수 있습니다. 스택 할당의 대안으로는 페이지 할당자(page allocator)와 블록 할당자(block allocator)가 있습니다 (메모리 할당(Memory Allocation) 참조).

소스 파일

다음은 threads 및 include/threads 디렉터리에 있는 파일의 간략한 개요입니다. 대부분의 코드는 수정할 필요가 없지만, 이 개요가 어떤 코드를 살펴보아야 할지 출발점을 제공하기를 바랍니다.

threads 코드

  • loader.S, loader.h

    커널 로더입니다. PC BIOS가 메모리에 로드하는 512바이트의 코드와 데이터로 어셈블되며, 이어서 디스크에서 커널(kernel)을 찾아 메모리에 로드하고 bootstrap()으로 점프합니다. 이 진입점은 start.S에 있습니다. 이 코드를 보거나 수정할 필요는 없습니다. start.S는 메모리 보호에 필요한 기본 설정을 수행하고 64bit long mode로 점프합니다. 로더와 달리 이 코드는 실제로 커널의 일부입니다.

  • kernel.lds.S

    커널(kernel)을 링크하는 데 사용되는 링커 스크립트(linker script)입니다. 커널의 로드 주소를 설정하고 start.S가 커널 이미지의 시작 부분 근처에 오도록 배치합니다. 역시 이 코드를 보거나 수정할 필요는 없지만, 궁금할 경우를 위해 여기에 있습니다.

  • init.c, init.h

    커널(kernel) 초기화 코드입니다. 여기에는 main(), 즉 커널의 main program.이 포함됩니다. 무엇이 초기화되는지 보기 위해 적어도 main()은 살펴보아야 합니다. 여기에 여러분의 초기화 코드를 추가하고 싶을 수도 있습니다.

  • thread.c, thread.h

    기본 스레드(thread) 지원입니다. 여러분 작업의 상당 부분은 이 파일들에서 이루어집니다. thread.h는 struct thread를 정의하며, 네 프로젝트 모두에서 이 구조체를 수정하게 될 가능성이 큽니다. 자세한 내용은 스레드(Threads)를 참조하십시오.

  • palloc.c, palloc.h

    페이지 할당자(page allocator)입니다. 시스템 메모리를 4 kB 페이지의 배수 단위로 나누어 줍니다. 자세한 내용은 페이지 할당자(Page Allocator)를 참조하십시오.

  • malloc.c, malloc.h

    커널(kernel)용 malloc() 및 free()의 간단한 구현입니다. 자세한 내용은 블록 할당자(Block Allocator)를 참조하십시오.

  • interrupt.c, interrupt.h

    기본 인터럽트(interrupt) 처리와 인터럽트를 켜고 끄는 함수들입니다.

  • intr-stubs.S, intr-stubs.h

    저수준 인터럽트(interrupt) 처리를 위한 어셈블리 코드입니다.

  • synch.c, synch.h

    기본 동기화 기본 요소(synchronization primitives)입니다. 세마포어(semaphore), 락(lock), 조건 변수(condition variable), 최적화 배리어(optimization barrier)를 포함합니다. 네 프로젝트 모두에서 동기화를 위해 이들을 사용해야 합니다. 자세한 내용은 동기화(Synchronization)를 참조하십시오.

  • mmu.c, mmu.h

    x86-64 페이지 테이블(page table) 작업을 위한 함수들입니다. lab1 이후 이 파일을 더 자세히 살펴보게 됩니다.

  • io.h

    I/O 포트 접근을 위한 함수들입니다. 주로 여러분이 건드릴 필요가 없는 devices 디렉터리(directory)의 소스 코드에서 사용됩니다.

  • vaddr.h, pte.h

    가상 주소(virtual address)와 페이지 테이블(page table) 엔트리(entry)를 다루기 위한 함수와 매크로입니다. 이들은 프로젝트 3에서 더 중요해집니다. 지금은 무시해도 됩니다.

  • flags.h

    x86-64 flags 레지스터(register)의 몇몇 비트를 정의하는 매크로입니다. 지금은 관심을 가질 필요가 거의 없습니다.

devices 코드

기본 스레드 커널(kernel)에는 devices 디렉터리(directory)의 다음 파일들도 포함됩니다.

  • timer.c, timer.h

    기본적으로 초당 100번 틱(tick)을 발생시키는 시스템 타이머(timer)입니다. 이 프로젝트에서 이 코드를 수정하게 됩니다.

  • vga.c, vga.h

    VGA 디스플레이 드라이버입니다. 화면에 텍스트를 쓰는 일을 담당합니다. 이 코드를 살펴볼 필요는 없습니다. printf()가 여러분 대신 VGA 디스플레이 드라이버를 호출하므로, 이 코드를 직접 호출할 이유는 거의 없습니다.

  • serial.c, serial.h

    시리얼 포트(serial port) 드라이버입니다. 역시 printf()가 여러분 대신 이 코드를 호출하므로 직접 호출할 필요는 없습니다. 시리얼 입력을 입력 계층(input layer)에 전달하여 처리합니다(아래 참조).

  • block.c, block.h

    블록 장치(block device), 즉 고정 크기 블록의 배열로 구성된 무작위 접근(random-access) 디스크형 장치에 대한 추상화 계층입니다. Pintos는 기본적으로 IDE 디스크와 파티션(partition)이라는 두 종류의 블록 장치를 지원합니다. 어떤 종류이든 블록 장치는 프로젝트 2가 되기 전까지 실제로 사용되지 않습니다.

  • ide.c, ide.h

    최대 4개의 IDE 디스크에서 섹터(sector)를 읽고 쓰는 기능을 지원합니다.

  • partition.c, partition.h

    디스크의 파티션(partition) 구조를 이해하여, 하나의 디스크를 독립적으로 사용할 수 있는 여러 영역 (파티션)으로 나눌 수 있게 합니다.

  • kbd.c, kbd.h

    키보드 드라이버입니다. 키 입력을 입력 계층(input layer)에 전달하여 처리합니다(아래 참조).

  • input.c, input.h

    입력 계층(input layer)입니다. 키보드 또는 시리얼 드라이버가 전달한 입력 문자를 큐(queue)에 넣습니다.

  • intq.c, intq.h

    인터럽트 큐(interrupt queue)입니다. 커널(kernel) 스레드와 인터럽트(interrupt) 핸들러(handler)가 모두 접근하려는 원형 큐(circular queue)를 관리합니다. 키보드와 시리얼 드라이버가 사용합니다.

  • rtc.c, rtc.h

    실시간 시계(real-time clock) 드라이버입니다. 커널(kernel)이 현재 날짜와 시간을 알아낼 수 있게 합니다. 기본적으로 이는 난수 생성기의 초기 시드(seed)를 고르기 위해 thread/init.c에서만 사용됩니다.

  • speaker.c, speaker.h

    PC 스피커에서 톤(tone)을 낼 수 있는 드라이버입니다.

  • pit.c, pit.h

    8254 Programmable Interrupt Timer를 설정하는 코드입니다. 각 장치가 PIT의 출력 채널(output channel) 중 하나를 사용하므로, 이 코드는 devices/timer.c와 devices/speaker.c 양쪽에서 사용됩니다.

lib 코드

마지막으로 lib와 lib/kernel에는 유용한 라이브러리 루틴(library routine)이 들어 있습니다. (lib/user는 프로젝트 2부터 사용자 프로그램에서 사용되지만 커널(kernel)의 일부는 아닙니다.) 조금 더 자세히는 다음과 같습니다.

  • ctype.h, inttypes.h, limits.h, stdarg.h, stdbool.h, stddef.h, stdint.h, stdio.c, stdio.h, stdlib.c, stdlib.h, string.c, string.h

    표준 C 라이브러리의 부분 집합입니다.

  • debug.c, debug.h

    디버깅(debugging)을 돕는 함수와 매크로입니다. 자세한 내용은 디버깅 도구(Debugging Tools)를 참조하십시오.

  • random.c, random.h

    의사 난수(pseudo-random number) 생성기입니다. 실제 난수 값의 순서는 Pintos 실행마다 달라지지 않습니다.

  • round.h

    반올림 및 정렬(rounding)을 위한 매크로입니다.

  • syscall-nr.h

    시스템 콜(system call) 번호입니다. 프로젝트 2가 되기 전까지는 사용되지 않습니다.

  • kernel/list.c, kernel/list.h

    이중 연결 리스트(doubly linked list) 구현입니다. Pintos 코드 전반에서 사용되며, 프로젝트 1에서도 여러분이 몇 군데에서 직접 사용하고 싶을 가능성이 큽니다. 시작하기 전에 이 코드를 훑어볼 것을 권장합니다(특히 헤더 파일의 주석).

  • kernel/bitmap.c, kernel/bitmap.h

    비트맵(bitmap) 구현입니다. 원한다면 여러분의 코드에서 사용할 수 있지만, 프로젝트 1에서는 아마 필요하지 않을 것입니다.

  • kernel/hash.c, kernel/hash.h

    해시 테이블(hash table) 구현입니다. 프로젝트 3에서 유용할 가능성이 큽니다.

  • kernel/console.c, kernel/console.h, kernel/stdio.h

    printf()와 몇 가지 다른 함수를 구현합니다.

동기화(Synchronization)

적절한 동기화는 이 문제들을 해결하는 데 중요한 부분입니다. 모든 동기화 문제는 인터럽트를 꺼서 쉽게 해결할 수 있습니다. 인터럽트가 꺼져 있는 동안에는 동시성(concurrency)이 없으므로 경쟁 상태(race condition)가 발생할 가능성도 없습니다. 그래서 모든 동기화 문제를 이런 방식으로 해결하고 싶어질 수 있지만, 그렇게 하지 마십시오. 대신 세마포어(semaphore), 락(lock), 조건 변수(condition variable)를 사용해 동기화 문제의 대부분을 해결하십시오. 어떤 상황에서 어떤 동기화 기본 요소(synchronization primitive)를 사용할 수 있는지 확신이 없다면, 동기화 둘러보기 섹션(동기화(Synchronization) 참조)이나 threads/synch.c의 주석을 읽어 보십시오.

Pintos 프로젝트에서 인터럽트 비활성화로 해결하는 것이 가장 적합한 유일한 종류의 문제는 커널(kernel) 스레드(thread)와 인터럽트(interrupt) 핸들러(handler)가 공유하는 데이터를 조정하는 것입니다. 인터럽트 핸들러는 sleep할 수 없으므로 락을 획득할 수 없습니다. 이는 커널 스레드와 인터럽트 핸들러 사이에 공유되는 데이터는 커널 스레드 안에서 인터럽트를 꺼서 보호해야 한다는 뜻입니다.

이 프로젝트에서는 인터럽트 핸들러에서 약간의 스레드(thread) 상태에 접근하기만 하면 됩니다. 알람 시계(alarm clock)에서는 타이머(timer) 인터럽트가 잠자는 스레드를 깨워야 합니다. 고급 스케줄러(advanced scheduler)에서는 타이머 인터럽트가 몇몇 전역 변수와 스레드별(per-thread) 변수에 접근해야 합니다. 커널 스레드에서 이러한 변수에 접근할 때는 타이머 인터럽트가 끼어들지 못하도록 인터럽트를 비활성화해야 합니다.

인터럽트를 꺼야 할 때는 가능한 한 가장 적은 코드 범위에서만 그렇게 하도록 주의하십시오. 그렇지 않으면 타이머 틱(timer tick)이나 입력 이벤트(input event) 같은 중요한 것을 잃을 수 있습니다. 인터럽트를 끄면 인터럽트 처리 지연 시간(interrupt handling latency)도 증가하므로, 지나치게 사용하면 머신이 느리게 느껴질 수 있습니다.

synch.c의 동기화 기본 요소 자체도 인터럽트를 비활성화하여 구현되어 있습니다. 여기서 인터럽트가 비활성화된 상태로 실행되는 코드의 양을 늘려야 할 수도 있지만, 그래도 최소한으로 유지하려고 해야 합니다.

코드의 한 구간이 인터럽트되지 않음을 확인하고 싶을 때는 인터럽트 비활성화가 디버깅(debugging)에 유용할 수 있습니다. 프로젝트를 제출하기 전에는 디버깅 코드를 제거해야 합니다. (단순히 주석 처리만 하지 마십시오. 코드가 읽기 어려워질 수 있습니다.)

제출물에는 바쁜 대기(busy waiting)가 없어야 합니다. thread_yield()를 호출하는 빡빡한 루프는 바쁜 대기의 한 형태입니다.

개발 제안

과거에는 많은 그룹이 과제를 여러 조각으로 나눈 다음, 각 구성원이 마감 직전까지 자기 부분을 작업하고 그때 다시 모여 코드를 합친 뒤 제출했습니다. 이는 좋지 않은 생각입니다. 우리는 이 방식을 권장하지 않습니다. 이렇게 한 그룹들은 두 변경 사항이 서로 충돌하여 막판에 많은 디버깅(debugging)이 필요해지는 일을 자주 겪었습니다. 이 방식을 택한 일부 그룹은 테스트를 통과하기는커녕 컴파일이나 부팅조차 되지 않는 코드를 제출하기도 했습니다.

대신 git 같은 소스 코드 관리 시스템(source code control system)을 사용하여 팀의 변경 사항을 일찍, 자주 통합할 것을 권장합니다. 이렇게 하면 모두가 완성된 뒤에만 보는 것이 아니라 작성되는 즉시 서로의 코드를 볼 수 있으므로 예상치 못한 일이 생길 가능성이 줄어듭니다. 이러한 시스템은 변경 사항을 검토할 수 있게 해 주며, 어떤 변경이 버그를 도입했을 때는 동작하던 버전의 코드로 되돌아갈 수 있게 해 줍니다.

이 프로젝트와 이후 프로젝트를 진행하면서 여러분이 전혀 이해하지 못하는 버그를 만날 것을 예상해야 합니다. 그럴 때는 유용한 디버깅 팁으로 가득한 디버깅 도구 부록을 다시 읽어 보십시오. 다시 속도를 낼 수 있도록 도움이 될 것입니다 (디버깅 도구(Debugging Tools) 참조). 백트레이스 섹션도 반드시 읽으십시오 (백트레이스(Backtraces) 참조). 이 섹션은 모든 커널 패닉(kernel panic)이나 어설션 실패(assertion failure)에서 최대한 많은 정보를 얻는 데 도움이 됩니다.
