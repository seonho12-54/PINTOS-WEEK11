시스템 콜(System Calls)

시스템 콜(system call) 기반 구조를 구현하십시오.

userprog/syscall.c에 시스템 콜 핸들러(system call handler)를 구현하십시오. 제공된 골격 구현은 프로세스(process)를 종료하는 방식으로 시스템 콜을 "처리"합니다. 이 핸들러는 시스템 콜 번호(system call number)를 가져오고, 그 뒤에 시스템 콜 인자가 있다면 가져온 다음, 적절한 동작을 수행해야 합니다.

시스템 콜 세부 사항(System Call Details)

첫 번째 프로젝트에서는 운영체제가 사용자 프로그램(user program)으로부터 제어권을 되찾는 한 가지 방법, 즉 타이머와 I/O 장치에서 발생하는 인터럽트 (interrupt)를 이미 다루었습니다. 이러한 인터럽트는 CPU 외부의 대상이 일으키므로 "외부(external)" 인터럽트입니다.

운영체제는 프로그램 코드에서 발생하는 이벤트인 소프트웨어 예외(software exception)도 처리합니다. 페이지 폴트(page fault)나 0으로 나누기 같은 오류가 여기에 해당합니다. 예외는 사용자 프로그램이 운영체제에 서비스, 즉 "시스템 콜(system calls)"을 요청하는 수단이기도 합니다.

전통적인 x86 아키텍처에서는 시스템 콜을 다른 소프트웨어 예외와 같은 방식으로 처리했습니다. 하지만 x86-64에서는 제조사들이 시스템 콜을 위한 특수 명령어 syscall을 도입했습니다. 이 명령어는 시스템 콜 핸들러를 빠르게 호출하는 방법을 제공합니다.

오늘날 x86-64에서 시스템 콜을 호출하는 데 가장 널리 쓰이는 수단은 syscall 명령어입니다. Pintos에서 사용자 프로그램은 시스템 콜을 만들기 위해 syscall을 호출합니다. syscall 명령어를 호출하기 전에 시스템 콜 번호와 추가 인자들은 일반적인 방식으로 레지스터에 설정되어 있어야 하지만, 다음 두 가지 차이가 있습니다.

  • %rax는 시스템 콜 번호입니다.
  • 네 번째 인자는 %r10이며 %rcx가 아닙니다.

따라서 시스템 콜 핸들러 syscall_handler()가 제어권을 얻었을 때, 시스템 콜 번호는 rax에 있으며 인자는 %rdi, %rsi, %rdx, %r10, %r8, %r9 순서로 전달됩니다.

호출자(caller)의 레지스터는 핸들러에 전달되는 struct intr_frame을 통해 접근할 수 있습니다. (struct intr_frame은 커널 스택(kernel stack)에 있습니다.)

x86-64 함수 반환값 규약은 반환값을 RAX 레지스터에 둡니다. 값을 반환하는 시스템 콜은 rax 멤버를 수정하여 값을 반환할 수 있습니다. 이 멤버는 struct intr_frame에 있습니다.

다음 시스템 콜을 구현하십시오.

아래에 나열된 프로토타입(prototype)은 include/lib/user/syscall.h를 포함한 사용자 프로그램(user program)에서 보이는 형태입니다. 이 헤더와 include/lib/user 안의 다른 모든 헤더는 사용자 프로그램 전용입니다. 각 시스템 콜의 시스템 콜 번호는 include/lib/syscall-nr.h에 정의되어 있습니다.

void halt (void);

power_off()를 호출하여 Pintos를 종료합니다. 이 함수는 src/include/threads/init.h에 선언되어 있습니다. 가능한 데드락(deadlock) 상황 등에 관한 일부 정보를 잃게 되므로 이 호출은 드물게 사용해야 합니다.

void exit (int status);

현재 사용자 프로그램(user program)을 종료하고 status를 커널(kernel)에 반환합니다. 프로세스(process)의 부모가 이 프로세스에 대해 wait를 호출하면 (아래 참고), 이 값이 반환될 상태입니다. 관례적으로 status가 0이면 성공을 의미하고, 0이 아닌 값은 오류를 의미합니다.

pid_t fork (const char *thread_name);

현재 프로세스(process)의 복제본인 새 프로세스를 THREAD_NAME이라는 이름으로 생성합니다. 피호출자 저장(callee-saved) 레지스터인 %RBX, %RSP, %RBP, %R12 - %R15를 제외하면 레지스터 값을 복제할 필요는 없습니다. 자식 프로세스의 pid를 반환해야 하며, 실패하면 유효한 pid가 아닌 값을 반환해야 합니다. 자식 프로세스에서 반환값은 0이어야 합니다. 자식은 파일 디스크립터(file descriptor)와 가상 메모리(virtual memory) 공간을 포함한 리소스를 복제된(DUPLICATED) 상태로 가져야 합니다. 부모 프로세스는 자식 프로세스가 성공적으로 복제되었는지 알기 전에는 fork에서 반환해서는 안 됩니다. 즉, 자식 프로세스가 리소스 복제에 실패하면 부모의 fork () 호출은 TID_ERROR를 반환해야 합니다.

템플릿은 pml4_for_each()를 사용해 해당 페이지 테이블 (page table) 구조를 포함한 전체 사용자 메모리 공간을 복사합니다. 하지만 이 함수는 threads/mmu.c에 있습니다. 전달된 pte_for_each_func에서 빠진 부분은 여러분이 채워야 합니다 (참조: 가상 주소(virtual address)).

int exec (const char *cmd_line);

현재 프로세스(process)를 cmd_line에 주어진 이름의 실행 파일로 바꾸고, 주어진 인자들을 전달합니다. 성공하면 이 호출은 절대 반환하지 않습니다. 반대로 어떤 이유로든 프로그램을 로드하거나 실행할 수 없으면 프로세스는 종료 상태 -1로 종료됩니다. 이 함수는 exec를 호출한 스레드(thread)의 이름을 바꾸지 않습니다. 파일 디스크립터는 exec 호출을 거쳐서도 열린 상태로 유지된다는 점에 유의하십시오.

int wait (pid_t pid);

자식 프로세스(process) pid를 기다리고 자식의 종료 상태(exit status)를 가져옵니다. pid가 아직 살아 있다면 종료될 때까지 기다립니다. 그런 다음 pid가 exit에 전달한 상태를 반환합니다. pid가 exit()를 호출하지 않고 커널(kernel)에 의해 종료된 경우, 예를 들어 예외 때문에 죽은 경우에는 wait(pid)가 -1을 반환해야 합니다. 부모 프로세스가 wait를 호출하는 시점에 이미 종료된 자식 프로세스를 기다리는 것은 완전히 합법입니다. 하지만 커널은 여전히 부모가 자식의 종료 상태를 가져오거나, 자식이 커널에 의해 종료되었음을 알 수 있게 해야 합니다.

다음 조건 중 하나라도 참이면 wait는 즉시 실패하고 -1을 반환해야 합니다.

  • pid가 호출 프로세스(process)의 직접적인 자식을 가리키지 않습니다. pid는 호출 프로세스가 성공적인 호출의 반환값으로 pid를 받은 경우에만 호출 프로세스의 직접적인 자식입니다. 여기서 그 호출은 fork입니다. 자식은 상속되지 않는다는 점에 유의하십시오. A가 자식 B를 만들고 B가 자식 프로세스 C를 만들었다면, B가 죽었더라도 A는 C를 기다릴 수 없습니다. 프로세스 A가 wait(C)를 호출하면 반드시 실패해야 합니다. 마찬가지로 부모 프로세스가 먼저 종료되더라도 고아가 된 프로세스(orphaned process)는 새 부모에게 배정되지 않습니다.
  • wait를 호출한 프로세스가 이미 wait를 pid에 대해 호출했습니다. 즉, 프로세스는 어떤 주어진 자식에 대해서도 최대 한 번만 wait할 수 있습니다.

프로세스는 원하는 수의 자식을 만들 수 있고, 어떤 순서로든 그 자식들에 대해 wait할 수 있으며, 일부 또는 모든 자식을 기다리지 않고 종료할 수도 있습니다. 여러분의 설계는 wait가 발생할 수 있는 모든 방식을 고려해야 합니다. 부모가 기다렸는지 여부와 자식이 부모보다 먼저 또는 나중에 종료되었는지와 무관하게, 프로세스의 모든 리소스는 struct thread를 포함하여 반드시 해제되어야 합니다.

초기 프로세스(process)가 종료될 때까지 Pintos가 종료되지 않도록 반드시 보장해야 합니다. 제공된 Pintos 코드는 process_wait()를 호출하여 이를 시도합니다. 이 함수는 userprog/process.c에 있고, main()에서 호출됩니다(threads/init.c). process_wait()를 함수 맨 위의 주석에 맞게 구현한 다음, wait 시스템 콜(system call)을 process_wait()를 이용해 구현하는 것을 권장합니다.

이 시스템 콜을 구현하는 작업은 다른 시스템 콜보다 훨씬 더 많은 노력이 필요합니다.

bool create (const char *file, unsigned initial_size);

새 파일 file을 생성하며, 초기 크기는 initial_size바이트입니다. 성공하면 true, 실패하면 false를 반환합니다. 새 파일을 생성한다고 해서 파일이 열리지는 않습니다. 새 파일을 여는 것은 open 시스템 콜(system call)이 필요한 별도의 작업입니다.

bool remove (const char *file);

file이라는 파일을 삭제합니다. 성공하면 true, 실패하면 false를 반환합니다. 파일은 열려 있든 닫혀 있든 삭제될 수 있으며, 열린 파일을 삭제해도 그 파일이 닫히지는 않습니다. 자세한 내용은 자주 묻는 질문(FAQ)의 열린 파일 삭제 (Removing an Open File)를 참고하십시오.

int open (const char *file);

file이라는 파일을 엽니다. 파일을 열 수 있으면 "파일 디스크립터(file descriptor)"(fd)라고 하는 음수가 아닌 정수 핸들(handle)을 반환하고, 열 수 없으면 -1을 반환합니다. 번호 0과 1의 파일 디스크립터는 콘솔(console)을 위해 예약되어 있습니다. fd 0(STDIN_FILENO)은 표준 입력, fd 1 (STDOUT_FILENO)은 표준 출력입니다. open 시스템 콜은 이 두 파일 디스크립터 중 어느 것도 반환하지 않으며, 이 값들은 아래에서 명시적으로 설명하는 경우에만 시스템 콜 인자로 유효합니다. 각 프로세스(process)는 독립적인 파일 디스크립터 집합을 가집니다. 파일 디스크립터는 자식 프로세스에 상속됩니다. 하나의 파일을 한 프로세스가 여러 번 열거나 여러 프로세스가 열면, 각 open은 새 파일 디스크립터를 반환합니다. 같은 파일에 대한 서로 다른 파일 디스크립터는 별도의 close 호출에서 독립적으로 닫히며 파일 위치(file position)를 공유하지 않습니다. Extra를 수행하려면 0부터 시작하는 정수를 반환하는 Linux 방식을 따라야 합니다.

int filesize (int fd);

fd로 열린 파일의 크기를 바이트 단위로 반환합니다.

int read (int fd, void *buffer, unsigned size);

size바이트를 fd로 열린 파일에서 읽어 buffer에 저장합니다. 실제로 읽은 바이트 수를 반환하며, 파일 끝(end of file)에서는 0을 반환합니다. 파일 끝이 아닌 다른 조건 때문에 읽을 수 없으면 -1을 반환합니다. fd 0은 input_getc()를 사용해 키보드에서 읽습니다.

int write (int fd, const void *buffer, unsigned size);

size바이트를 buffer에서 열린 파일 fd로 씁니다. 실제로 쓴 바이트 수를 반환하며, 일부 바이트를 쓸 수 없으면 이 값은 size보다 작을 수 있습니다. 일반적으로 파일 끝(end-of-file)을 넘어 쓰면 파일이 확장되지만, 기본 파일 시스템에는 파일 증가가 구현되어 있지 않습니다. 기대되는 동작은 파일 끝까지 가능한 한 많은 바이트를 쓰고 실제로 쓴 수를 반환하거나, 전혀 쓸 수 없으면 0을 반환하는 것입니다. fd 1은 콘솔에 씁니다. 콘솔에 쓰는 여러분의 코드는 적어도 size가 몇백 바이트보다 크지 않은 한 전체 버퍼를 putbuf() 한 번으로 써야 합니다. 더 큰 버퍼를 나누는 것은 합리적입니다. 그렇지 않으면 서로 다른 프로세스가 출력한 텍스트 줄이 콘솔에서 섞여, 사람이 읽기에도 채점 스크립트에도 혼란을 줄 수 있습니다.

void seek (int fd, unsigned position);

열린 파일 fd에서 다음에 읽거나 쓸 바이트 위치를 position으로 변경합니다. position은 파일 시작부터의 바이트 단위 위치입니다. 따라서 position이 0이면 파일의 시작입니다. 현재 파일 끝을 넘어 seek하는 것은 오류가 아닙니다. 이후 read는 0바이트를 얻어 파일 끝임을 나타냅니다. 이후 write는 파일을 확장하고, 쓰이지 않은 간격은 0으로 채웁니다. 하지만 Pintos에서는 프로젝트 4가 완료될 때까지 파일 길이가 고정되어 있으므로, 파일 끝을 넘는 write는 오류를 반환합니다. 이러한 의미론(semantics)은 파일 시스템에 구현되어 있으며 시스템 콜(system call) 구현에서 별도의 특별한 노력이 필요하지 않습니다.

unsigned tell (int fd);

열린 파일 fd에서 다음에 읽거나 쓸 바이트의 위치를 파일 시작부터의 바이트 단위로 반환합니다.

void close (int fd);

파일 디스크립터(file descriptor) fd를 닫습니다. 프로세스(process)가 exit하거나 종료되면, 열린 각 파일 디스크립터에 대해 이 함수를 호출한 것처럼 모든 열린 파일 디스크립터가 암묵적으로 닫힙니다.

이 파일에는 다른 syscall도 정의되어 있습니다. 지금은 무시하십시오. 그중 일부는 프로젝트 3에서, 나머지는 프로젝트 4에서 구현하게 되므로 확장성을 염두에 두고 시스템을 설계해야 합니다.

여러 사용자 프로세스가 동시에 시스템 콜을 호출할 수 있도록 시스템 콜을 동기화(synchronize)해야 합니다. 특히 여러 스레드(thread)가 동시에 filesys 디렉터리(directory)에 제공된 파일 시스템 코드로 진입하는 것은 안전하지 않습니다. 여러분의 시스템 콜 구현은 파일 시스템 코드를 임계 영역 (critical section)으로 취급해야 합니다. process_exec()도 파일에 접근한다는 점을 잊지 마십시오. 현재로서는 filesys 디렉터리의 코드를 수정하지 않는 것을 권장합니다.

lib/user/syscall.c에는 각 시스템 콜(system call)에 대응하는 사용자 수준 함수가 제공되어 있습니다. 이 함수들은 사용자 프로세스가 C 프로그램에서 각 시스템 콜을 호출하는 방법을 제공합니다. 각 함수는 약간의 인라인 어셈블리 (inline assembly) 코드를 사용해 시스템 콜을 호출하고, 필요한 경우 시스템 콜의 반환값을 반환합니다.

이 부분을 완료한 뒤부터 Pintos는 항상 견고해야 합니다. 사용자 프로그램 (user program)이 어떤 일을 하더라도 OS가 크래시(crash)하거나 패닉(panic)에 빠지거나, assertion에 실패하거나, 그 밖의 오동작을 일으켜서는 안 됩니다. 이 점은 특히 중요합니다. 테스트는 매우 다양한 방식으로 여러분의 시스템 콜을 깨뜨리려 할 것입니다. 모든 코너 케이스(corner case)를 생각하고 처리해야 합니다. 사용자 프로그램이 OS를 정지시킬 수 있는 유일한 방법은 halt 시스템 콜을 호출하는 것뿐이어야 합니다.

시스템 콜(system call)에 잘못된 인자가 전달되면, 허용 가능한 선택지는 오류 값을 반환하거나(값을 반환하는 호출의 경우), 정의되지 않은 값을 반환하거나, 프로세스(process)를 종료하는 것입니다.
