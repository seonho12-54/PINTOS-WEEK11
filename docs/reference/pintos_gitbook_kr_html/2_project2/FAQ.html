자주 묻는 질문

얼마나 많은 코드를 작성해야 하나요?

다음은 git diff --stat으로 생성한 참조 해법(reference solution)의 요약입니다. 마지막 행은 삽입된 줄과 삭제된 줄의 총합을 보여 줍니다. 변경된 한 줄은 삽입 1줄과 삭제 1줄로 모두 계산됩니다. 참조 해법은 가능한 해법 중 하나일 뿐입니다. 다른 해법도 많이 가능하며, 그중 상당수는 참조 해법과 크게 다를 수 있습니다. 훌륭한 해법 중에는 참조 해법이 수정한 모든 파일을 수정하지 않는 것도 있고, 참조 해법이 수정하지 않은 파일을 수정하는 것도 있을 수 있습니다. 또한 이 통계에는 추가 요구사항(extra requirement) 구현도 포함되어 있습니다. 참고로 약 150줄은 추가 과제와 관련되어 있습니다.

pintos -p file -- -q를 실행하면 커널이 항상 패닉을 일으킵니다.

파일 시스템을 포맷했나요(pintos -f)?

파일 이름이 너무 길지는 않나요? 파일 시스템은 파일 이름을 14자로 제한합니다.

pintos -p ../../examples/echo -- -q 같은 명령은 이 제한을 초과합니다.

대신 pintos -p ../../examples/echo:echo -- -q를 사용해 파일을 echo라는 이름으로 넣으세요.

파일 시스템이 가득 찼나요?

파일 시스템에 이미 16개의 파일이 들어 있나요? 기본 Pintos 파일 시스템에는 16개 파일 제한이 있습니다.

파일 시스템이 너무 단편화되어 파일에 필요한 연속 공간이 부족할 수도 있습니다.

pintos -p ../file --를 실행해도 'file'이 복사되지 않습니다.

기본적으로 파일은 지정한 이름 그대로 기록됩니다. 따라서 이 경우 복사된 파일의 이름은 '../file'이 됩니다. 아마도 대신 pintos -p ../file:file --를 실행하려는 의도였을 것입니다.

모든 사용자 프로그램이 페이지 폴트(page fault)로 종료됩니다.

인자 전달(argument passing)을 구현하지 않았거나 올바르게 구현하지 않았을 때 이런 일이 발생합니다. 사용자 프로그램용 기본 C 라이브러리는 스택(stack)에서 argv를 읽으려고 합니다. 스택이나 레지스터가 올바르게 설정되어 있지 않으면 페이지 폴트가 발생합니다.

모든 사용자 프로그램이 system call!과 함께 종료됩니다.

다른 동작을 보기 전에 먼저 시스템 콜(system call)을 구현해야 합니다. 합리적인 프로그램은 적어도 하나의 시스템 콜(exit())을 시도하며, 대부분의 프로그램은 그보다 더 많이 호출합니다. 특히 printf()는 write 시스템 콜을 호출합니다. 기본 시스템 콜 핸들러는 단순히 system call!을 출력하고 프로그램을 종료합니다. 그 전까지는 hex_dump()를 사용해 인자 전달이 올바르게 구현되었는지 확인할 수 있습니다.

사용자 프로그램을 어떻게 디스어셈블할 수 있나요?

objdump 유틸리티는 사용자 프로그램 전체나 오브젝트 파일을 디스어셈블할 수 있습니다. objdump -d file 형태로 실행하세요. 개별 함수를 디스어셈블하려면 GDB의 disassemble 명령을 사용할 수도 있습니다.

많은 C include 파일이 Pintos 프로그램에서 동작하지 않는 이유는 무엇인가요?

Pintos 프로그램에서 libfoo를 사용할 수 있나요?

제공되는 C 라이브러리는 매우 제한적입니다. 실제 운영체제의 C 라이브러리에서 기대하는 많은 기능이 포함되어 있지 않습니다. C 라이브러리는 I/O와 메모리 할당(memory allocation)을 위해 시스템 콜을 수행해야 하므로, 운영체제와 아키텍처에 맞게 별도로 빌드되어야 합니다. 물론 모든 함수가 시스템 콜을 하지는 않지만, 보통 라이브러리는 하나의 단위로 컴파일됩니다.

사용하려는 라이브러리가 Pintos가 구현하지 않은 C 라이브러리 기능을 사용할 가능성이 큽니다. Pintos에서 동작하게 하려면 적어도 어느 정도의 포팅 작업이 필요할 것입니다. 특히 Pintos 사용자 프로그램(user program) C 라이브러리에는 malloc() 구현이 없습니다.

새 사용자 프로그램은 어떻게 컴파일하나요?

src/examples/Makefile을 수정한 뒤 make를 실행하세요.

디버거에서 사용자 프로그램을 실행할 수 있나요?

가능하지만 몇 가지 제한이 있습니다. GDB를 참고하세요.

tid_t와 pid_t의 차이는 무엇인가요?

tid_t는 커널 스레드(kernel thread)를 식별합니다. 이 스레드는 process_fork()로 생성되었다면 사용자 프로세스를 실행하고 있을 수 있고, thread_create()로 생성되었다면 그렇지 않을 수 있습니다. tid_t는 커널 안에서만 사용되는 데이터 타입입니다. pid_t는 사용자 프로세스(user process)를 식별합니다. 사용자 프로세스와 커널 모두 exec 및 wait 시스템 콜에서 pid_t를 사용합니다. tid_t와 pid_t에는 적절한 타입을 자유롭게 선택할 수 있습니다. 기본적으로 둘 다 int입니다. 같은 값이 같은 프로세스를 식별하도록 일대일 매핑을 사용할 수도 있고, 더 복잡한 매핑을 사용할 수도 있습니다. 선택은 구현에 달려 있습니다.

struct file *을 캐스팅해서 파일 디스크립터(file descriptor)를 얻어도 되나요?

struct thread *를 pid_t로 캐스팅해도 되나요?

아니요. pid_t와 file descriptor는 포인터 타입보다 작으므로 단순 캐스팅으로 처리할 수 없습니다.

프로세스마다 열 수 있는 파일 수의 최댓값을 정해도 되나요?

임의의 제한을 두지 않는 편이 좋습니다. 필요하다면 프로세스당 열린 파일을 128개로 제한할 수는 있습니다. 하지만 추가 요구사항을 구현하려면 제한이 없어야 합니다.

열린 파일이 제거되면 어떻게 되나요?

파일에 대해 표준 Unix 의미론을 구현해야 합니다. 즉, 어떤 파일이 제거되더라도 그 파일에 대한 파일 디스크립터(file descriptor)를 가진 프로세스는 해당 디스크립터를 계속 사용할 수 있어야 합니다. 따라서 그 프로세스는 파일을 계속 읽고 쓸 수 있습니다. 파일에는 더 이상 이름이 없고 다른 프로세스는 열 수 없지만, 그 파일을 참조하는 모든 파일 디스크립터가 닫히거나 머신이 종료될 때까지 파일은 계속 존재합니다.

4 kB보다 큰 스택(stack) 공간이 필요한 사용자 프로그램은 어떻게 실행할 수 있나요?

스택 설정 코드를 수정해 각 프로세스에 둘 이상의 스택 페이지를 할당할 수 있습니다. 다음 프로젝트에서는 더 나은 해법을 구현하게 됩니다.
