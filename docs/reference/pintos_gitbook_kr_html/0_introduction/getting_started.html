시작하기

시작하기

시작하려면 Pintos를 빌드할 수 있는 머신에 로그인해야 합니다. 수강생은 담당자가 제공한 Linux 머신이나 본인의 Solaris 또는 Linux 머신을 사용할 수 있습니다. 하지만 여러분의 코드는 제공된 머신에서 테스트하며, 여기서 제시하는 지침도 이 환경을 전제로 합니다. 본인 머신에 Pintos를 설치하고 작업하는 일에 대해서는 지원을 제공할 수 없습니다. 아래의 서버 환경이 각 학생에게 제공됩니다.

이제 다음을 실행하여 Pintos 소스 코드를 'pintos'라는 디렉터리에 가져올 수 있습니다.

git clone https://github.com/casys-kaist/pintos-kaist

다만 소스 코드는 프로젝트 기간 전에 변경될 가능성이 있으므로, 저장소를 복제해 두기를 권장합니다. 템플릿 코드를 변경하면, 여러분은 자신의 코드를 템플릿과 쉽게 동기화할 수 있습니다. 공개 포크를 만들지 마십시오. 템플릿 코드에 치명적인 버그가 있는 경우를 제외하고, 프로젝트 기간 중에는 소스 코드를 절대 변경하지 않습니다.

참고로, 우리는 제공된 Linux 머신과 동일한 환경인 Ubuntu 16.04.6 LTS에서 gcc(gcc (Ubuntu 7.4.0-1ubuntu1~16.04~ppa1) 7.4.0)와 qemu-system-x86_64 (QEMU emulator version 2.5.0 (Debian 1:2.5+dfsg-5ubuntu10.43))를 사용해 pintos 코드와 해답을 테스트했습니다.

소스 코드를 가져온 뒤에는 pintos 루트 디렉터리에서 아래 명령으로 환경을 설정할 수 있습니다.

$ source ./activate

이 줄을 홈 디렉터리의 '.bashrc'에 추가해 두는 것이 좋습니다. 그렇지 않으면 로그인할 때마다 이 명령을 입력해야 합니다.

소스 트리 개요

안에 무엇이 있는지 살펴봅시다. pintos/에서 볼 수 있는 디렉터리 구조는 다음과 같습니다.

  • threads/: 프로젝트 1부터 수정하게 될 기본 커널의 소스 코드입니다.
  • userprog/: 프로젝트 2부터 수정하게 될 사용자 프로그램 로더의 소스 코드입니다.
  • vm/: 거의 비어 있는 디렉터리입니다. 프로젝트 3에서 이곳에 가상 메모리를 구현합니다.
  • filesys/: 기본 파일 시스템의 소스 코드입니다. 프로젝트 2부터 이 파일 시스템을 사용하지만, 프로젝트 4가 되기 전까지는 수정하지 않습니다.
  • devices/: 키보드, 타이머, 디스크 등 I/O 장치 인터페이스의 소스 코드입니다. 프로젝트 1에서 타이머 구현을 수정합니다. 그 외에는 이 코드를 변경할 필요가 없을 것입니다.
  • lib/: 표준 C 라이브러리 일부의 구현입니다. 이 디렉터리의 코드는 Pintos 커널과, 프로젝트 2부터는 그 아래에서 실행되는 사용자 프로그램 모두에 컴파일되어 들어갑니다. 이 코드를 수정할 필요는 거의 없습니다.
  • include/lib/kernel/: Pintos 커널에만 포함되는 C 라이브러리 부분입니다. 또한 커널 코드에서 자유롭게 사용할 수 있는 몇 가지 자료형의 구현도 포함합니다. 비트맵, 이중 연결 리스트, 해시 테이블이 여기에 해당합니다. 커널에서는 이 디렉터리의 헤더를 #include <...> 표기법으로 포함할 수 있습니다.
  • include/lib/user/: Pintos 사용자 프로그램에만 포함되는 C 라이브러리 부분입니다. 사용자 프로그램에서는 이 디렉터리의 헤더를 #include <...> 표기법으로 포함할 수 있습니다.
  • tests/: 각 프로젝트의 테스트입니다. 제출물을 테스트하는 데 도움이 된다면 이 코드를 수정할 수 있지만, 우리가 테스트를 실행하기 전에는 원본으로 교체합니다.
  • examples/: 프로젝트 2부터 사용하는 예제 사용자 프로그램입니다.
  • include/: 헤더 파일(*.h)의 소스 코드입니다.

Pintos 빌드하기

다음 단계로, 첫 번째 프로젝트에 제공된 소스 코드를 빌드합니다. 먼저 cd로 threads 디렉터리로 이동합니다. 그런 다음 make 명령을 실행합니다. 그러면 build 디렉터리가 threads 아래에 생성되고, 그 안에 Makefile과 몇 개의 하위 디렉터리가 채워진 뒤, 그 안에서 커널이 빌드됩니다.

빌드 후 build 디렉터리에서 살펴볼 만한 파일은 다음과 같습니다.

  • Makefile: pintos/src/Makefile.build의 복사본입니다. 커널을 빌드하는 방법을 설명합니다.
  • kernel.o: 전체 커널의 목적 파일입니다. 각 커널 소스 파일에서 컴파일된 목적 파일들을 하나의 목적 파일로 링크한 결과입니다. 디버그 정보를 포함하므로, 이 파일에 대해 GDB (GDB 참조)나 백트레이스 (Backtraces)를 실행할 수 있습니다.
  • kernel.bin: 커널의 메모리 이미지, 즉 Pintos 커널을 실행하기 위해 메모리에 로드되는 정확한 바이트들입니다. 이는 kernel.o에서 디버그 정보를 제거한 것일 뿐입니다. 이렇게 하면 많은 공간을 절약할 수 있고, 그 결과 커널 로더 설계상 부과되는 512 kB 크기 제한에 커널이 걸리지 않게 됩니다.
  • loader.bin: 커널 로더의 메모리 이미지입니다. 디스크에서 커널을 메모리로 읽어 들여 시작하는, 어셈블리 언어로 작성된 작은 코드 조각입니다. PC BIOS가 정한 크기인 정확히 512바이트입니다. build의 하위 디렉터리에는 목적 파일(.o)과 의존성 파일(.d)이 들어 있으며, 둘 다 컴파일러가 생성합니다. 의존성 파일은 다른 소스 파일이나 헤더 파일이 변경되었을 때 어떤 소스 파일을 다시 컴파일해야 하는지 make에 알려 줍니다.

Pintos 실행하기

Pintos를 시뮬레이터에서 편리하게 실행할 수 있도록 pintos라는 프로그램을 제공합니다. 가장 단순한 경우에는 pintos argument... 형태로 pintos를 호출할 수 있습니다. 각 인자는 Pintos 커널에 전달되어 커널이 처리합니다. 한번 실행해 보십시오. 먼저 새로 생성된 build 디렉터리로 cd합니다. 그런 다음 pintos run alarm-multiple 명령을 실행합니다. 이 명령은 run alarm-multiple 인자를 Pintos 커널에 전달합니다. 이 인자들에서 run은 커널에 테스트를 실행하라고 지시하며, alarm-multiple은 실행할 테스트입니다. Pintos가 부팅되어 alarm-multiple 테스트 프로그램을 실행하고, 몇 화면 분량의 텍스트를 출력합니다.

명령줄에서 리다이렉션을 사용해 직렬 출력을 파일에 기록할 수 있습니다. 예를 들어 pintos -- run alarm-multiple > logfile처럼 실행합니다. pintos 프로그램은 qemu 또는 가상 하드웨어를 설정하기 위한 여러 옵션을 제공합니다. 옵션을 지정하는 경우, 그 옵션들은 Pintos 커널에 전달되는 명령보다 앞에 와야 하며 --로 구분되어야 합니다. 따라서 전체 명령은 pintos option... -- argument....와 같은 형태가 됩니다. 사용 가능한 옵션 목록을 보려면 아무 인자 없이 pintos를 호출하십시오. 옵션에는 VM 출력을 어떻게 표시할지 정하는 것들이 포함됩니다. -v는 VGA 표시를 끄고, -s는 stdin의 직렬 입력과 stdout의 직렬 출력을 억제합니다. Pintos 커널에는 run 외에도 명령과 옵션이 있습니다. 지금은 별로 중요하지 않지만, -h를 사용하면 목록을 볼 수 있습니다. 예를 들면 pintos -h입니다.
