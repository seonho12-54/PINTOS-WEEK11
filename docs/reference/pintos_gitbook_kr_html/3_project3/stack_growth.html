스택 증가(Stack Growth)

프로젝트 2에서 스택(stack)은 USER_STACK에서 시작하는 단일 페이지였고, 프로그램 실행은 이 크기로 제한되었습니다. 이제 스택이 현재 크기를 넘어 커지면 필요한 만큼 추가 페이지를 할당합니다.

추가 페이지는 스택 접근처럼 "보이는" 경우에만 할당하십시오. 스택 접근과 다른 접근을 구분하려고 시도하는 휴리스틱을 설계하십시오.

사용자 프로그램(user program)이 스택 포인터 아래의 스택에 쓰면 버그입니다. 실제 일반적인 OS는 프로세스(process)에 "시그널(signal)"을 전달하기 위해 언제든지 인터럽트를 걸 수 있고, 시그널은 스택의 데이터를 수정하기 때문입니다. 하지만 x86-64 PUSH 명령어는 스택 포인터를 조정하기 전에 접근 권한을 확인하므로, 스택 포인터보다 8 bytes 아래에서 페이지 폴트(page fault)를 일으킬 수 있습니다.

사용자 프로그램의 현재 스택 포인터 값을 얻을 수 있어야 합니다. 사용자 프로그램이 생성한 시스템 콜(system call) 또는 페이지 폴트 안에서는 rsp 멤버에서 이를 가져올 수 있으며, 이 멤버는 struct intr_frame 안에 있고 그 구조체는 각각 syscall_handler() 또는 page_fault()에 전달됩니다. 유효하지 않은 메모리 접근을 감지하기 위해 페이지 폴트에 의존한다면, 커널(kernel)에서 페이지 폴트가 발생하는 또 다른 경우를 처리해야 합니다. 프로세서는 예외가 사용자 모드에서 커널 모드로의 전환을 일으킬 때에만 스택 포인터를 저장하므로, rsp를 struct intr_frame에서 읽는 방식은, 그 구조체가 page_fault()에 전달되더라도 사용자 스택 포인터가 아니라 정의되지 않은 값을 얻게 됩니다. 사용자 모드에서 커널 모드로 처음 전환될 때 rsp를 struct thread에 저장하는 등 다른 방법을 마련해야 합니다.

스택 증가(stack growth) 기능을 구현하십시오. 이를 구현하려면 먼저 vm_try_handle_fault를 vm/vm.c에서 수정하여 스택 증가를 식별하십시오. 스택 증가를 식별한 뒤에는 스택을 증가시키기 위해 vm_stack_growth를 vm/vm.c에서 호출해야 합니다. vm_stack_growth를 구현하십시오.

bool vm_try_handle_fault (struct intr_frame *f, void *addr,
    bool user, bool write, bool not_present);

이 함수는 페이지 폴트(page fault) 예외를 처리하는 동안 page_fault(userprog/exception.c)에서 호출됩니다. 이 함수에서는 페이지 폴트가 스택 증가(stack growth)에 해당하는 유효한 경우인지 확인해야 합니다. 폴트를 스택 증가로 처리할 수 있다고 확인했다면, 폴트가 난 주소로 vm_stack_growth를 호출하십시오.

void vm_stack_growth (void *addr);

addr가 더 이상 폴트 주소가 아니도록 하나 이상의 익명 페이지(anonymous page)를 할당하여 스택 크기를 늘립니다. 할당을 처리할 때 addr를 PGSIZE로 내림 정렬해야 합니다.

대부분의 OS는 스택 크기에 어떤 절대 제한을 둡니다. 일부 OS는 많은 Unix 시스템의 ulimit 명령처럼 사용자가 이 제한을 조정할 수 있게 합니다. 많은 GNU/Linux 시스템에서 기본 제한은 8 MB입니다. 이 프로젝트에서는 스택 크기를 최대 1MB로 제한해야 합니다.

이제 모든 stack-growth 테스트 케이스가 통과해야 합니다.
