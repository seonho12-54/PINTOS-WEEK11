메모리 매핑 파일(memory-mapped file)

메모리 매핑 파일(memory-mapped file)

이 절에서는 메모리 매핑 페이지를 구현합니다. 익명 페이지와 달리 메모리 매핑 페이지는 파일 기반 매핑입니다. 페이지의 내용은 기존 파일의 데이터를 반영합니다. 페이지 폴트(page fault)가 발생하면 물리 프레임(frame)이 즉시 할당되고, 파일의 내용이 메모리로 복사됩니다. 메모리 매핑 페이지가 매핑 해제되거나 스왑 아웃(swap out)될 때 내용의 변경 사항은 파일에 반영됩니다.

mmap 및 munmap 시스템 콜(system call)

mmap과 munmap을 구현합니다. 이 둘은 메모리 매핑 파일(memory-mapped file)을 위한 두 시스템 콜(system call)입니다. VM 시스템은 mmap 영역의 페이지를 지연 방식으로 로드하고, 매핑의 백업 저장소(backing store)로 mmap된 파일 자체를 사용해야 합니다. 이 두 시스템 콜을 구현하려면 do_mmap과 do_munmap을 구현하고, vm/file.c에 정의된 이 함수들을 사용해야 합니다.

void *mmap (void *addr, size_t length, int writable, int fd, off_t offset);

length bytes를 fd로 열린 파일에서 offset byte부터 프로세스(process)의 가상 주소 공간 addr에 매핑합니다. 전체 파일은 addr에서 시작하는 연속적인 가상 페이지에 매핑됩니다. 파일 길이가 PGSIZE의 배수가 아니면 마지막으로 매핑된 페이지의 일부 bytes가 파일 끝을 넘어 "삐져나옵니다". 이 페이지에서 페이지 폴트(page fault)가 발생해 메모리로 읽어 들일 때 해당 bytes를 0으로 설정하고, 페이지를 디스크에 다시 쓸 때는 버립니다. 성공하면 이 함수는 파일이 매핑된 가상 주소를 반환합니다. 실패하면 파일 매핑에 유효한 주소가 아닌 NULL을 반환해야 합니다.

mmap 호출은 fd로 열린 파일의 길이가 0 bytes이면 실패할 수 있습니다. addr이 페이지 정렬(page-aligned)되어 있지 않거나, 매핑되는 페이지 범위가 스택(stack) 또는 실행 파일 로드 시 매핑된 페이지를 포함하여 기존 매핑 페이지 집합과 겹치면 반드시 실패해야 합니다. Linux에서 addr이 NULL이면 커널(kernel)이 매핑을 생성할 적절한 주소를 찾습니다. 단순화를 위해 주어진 addr에서 mmap을 시도하기만 해도 됩니다. 따라서 addr이 0이면 실패해야 합니다. 일부 Pintos 코드는 가상 페이지 0이 매핑되지 않는다고 가정하기 때문입니다. 또한 length가 0이면 mmap도 실패해야 합니다. 마지막으로 콘솔 입력과 출력을 나타내는 파일 디스크립터(file descriptor)는 매핑할 수 없습니다.

메모리 매핑 페이지도 익명 페이지와 마찬가지로 지연 방식으로 할당되어야 합니다. 페이지 객체를 만들려면 vm_alloc_page_with_initializer 또는 vm_alloc_page를 사용할 수 있습니다.

void munmap (void *addr);

지정된 주소 범위 addr에 대한 매핑을 해제합니다. 이 주소는 같은 프로세스(process)가 이전에 mmap을 호출해 반환받았고 아직 매핑 해제되지 않은 가상 주소여야 합니다.

모든 매핑은 프로세스(process)가 exit을 통해서든 다른 방식으로든 종료될 때 암묵적으로 매핑 해제됩니다. 매핑이 암묵적이든 명시적이든 해제될 때, 프로세스(process)가 쓴 모든 페이지는 파일에 다시 기록되어야 하며 쓰지 않은 페이지는 기록되면 안 됩니다. 그런 다음 해당 페이지들은 프로세스(process)의 가상 페이지 목록에서 제거됩니다.

파일을 닫거나 제거해도 해당 파일의 어떤 매핑도 해제되지 않습니다. Unix 관례에 따라 한 번 생성된 매핑은 munmap이 호출되거나 프로세스(process)가 종료될 때까지 유효합니다. 자세한 내용은 열린 파일 제거(Removing an Open File)를 참조하십시오. 각 매핑마다 파일에 대한 별도의 독립적인 참조를 얻으려면 file_reopen 함수를 사용해야 합니다.

둘 이상의 프로세스가 같은 파일을 매핑하는 경우, 이들이 일관된 데이터를 보아야 한다는 요구사항은 없습니다. Unix는 두 매핑이 같은 물리 페이지를 공유하게 하여 이를 처리하며, mmap 시스템 콜(system call)에는 클라이언트가 페이지를 공유(shared)로 할지 비공개(private), 즉 쓰기 시 복사(copy-on-write)로 할지를 지정할 수 있는 인자도 있습니다.

필요에 따라 vm_file_init과 vm_file_initializer를 vm/vm.c에서 수정할 수 있습니다.

void vm_file_init (void);

파일 기반 페이지(file-backed page) 서브시스템을 초기화합니다. 이 함수에서 파일 기반 페이지와 관련된 모든 것을 설정할 수 있습니다.

bool file_backed_initializer (struct page *page, enum vm_type type, void *kva);

파일 기반 페이지(file-backed page)를 초기화합니다. 이 함수는 먼저 page->operations에서 파일 기반 페이지의 핸들러(handler)를 설정합니다. 메모리를 뒷받침하는 파일 같은 일부 정보를 page 구조체에 업데이트할 수 있습니다.

static void file_backed_destroy (struct page *page);

연관된 파일을 닫아 파일 기반 페이지(file-backed page)를 파괴합니다. 내용이 더티(dirty)이면 변경 사항을 파일에 다시 기록해야 합니다. 이 함수에서 page 구조체를 해제할 필요는 없습니다. file_backed_destroy의 호출자가 이를 처리해야 합니다.
