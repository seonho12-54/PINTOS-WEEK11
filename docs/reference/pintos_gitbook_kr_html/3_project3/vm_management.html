메모리 관리

메모리 관리

가상 메모리(virtual memory) 시스템을 지원하려면 가상 페이지(virtual page)와 물리 프레임(physical frame)을 효과적으로 관리해야 합니다. 이는 어떤 가상 또는 물리 메모리 영역이 사용 중인지, 어떤 목적으로 쓰이는지, 누가 사용하는지 등을 추적해야 한다는 뜻입니다. 먼저 보조 페이지 테이블(supplemental page table)을 다루고, 그다음 물리 프레임을 다룹니다. 이해를 돕기 위해 이 문서에서는 가상 페이지를 "페이지(page)"라고 부르고, 물리 페이지를 "프레임(frame)"이라고 부릅니다.

페이지 구조와 연산

struct page

page는 include/vm/vm.h에 정의되어 있으며, 가상 메모리의 페이지(page)를 나타내는 구조체입니다. 페이지에 대해 알아야 하는 모든 필요한 데이터를 저장합니다. 현재 템플릿에서 이 구조체는 다음과 같습니다.

struct page {
  const struct page_operations *operations;
  void *va;              /* Address in terms of user space */
  struct frame *frame;   /* Back reference for frame */

  union {
    struct uninit_page uninit;
    struct anon_page anon;
    struct file_page file;
#ifdef EFILESYS
    struct page_cache page_cache;
#endif
  };
};

이 구조체는 페이지 연산(page operations, 아래 설명), 가상 주소, 물리 프레임(frame)을 가집니다. 추가로 union 필드도 있습니다. union은 하나의 메모리 영역에 서로 다른 타입의 데이터를 저장할 수 있게 해 주는 특수한 데이터 타입입니다. union 안에는 여러 멤버가 있지만, 한 번에 하나의 멤버만 값을 가질 수 있습니다. 이는 우리 시스템의 페이지가 uninit_page, anon_page, file_page 또는 page_cache 중 하나일 수 있음을 의미합니다. 예를 들어 어떤 페이지가 익명 페이지(anonymous page)라면(익명 페이지(Anonymous Page) 참조), page 구조체는 멤버 중 하나로 struct anon_page anon 필드를 갖습니다. anon_page는 익명 페이지를 위해 보관해야 하는 모든 필요한 정보를 담습니다.

페이지 연산

위에서 설명했듯이, 그리고 include/vm/vm.h에 정의되어 있듯이 페이지는 VM_UNINIT, VM_ANON 또는 VM_FILE일 수 있습니다. 페이지에는 스왑 인(swap in), 스왑 아웃(swap out), 페이지 파괴(destroy) 같은 여러 동작이 있습니다. 페이지 타입마다 이러한 동작에 필요한 단계와 작업이 다릅니다. 다시 말해 서로 다른 destroy 함수가 VM_ANON 페이지와 VM_FILE 페이지에 대해 호출되어야 합니다. 한 가지 방법은 각 함수에서 switch-case 문법을 사용해 각 경우를 처리하는 것입니다. 여기서는 이를 다루기 위해 객체 지향 프로그래밍의 "class inheritance" 개념을 도입합니다. 실제로 C 프로그래밍 언어에는 "class"도 "inheritance"도 없지만, Linux 같은 실제 운영체제 코드와 비슷한 방식으로 함수 포인터(function pointer)를 사용해 그 개념을 구현합니다.

함수 포인터(function pointer)는 지금까지 배운 다른 포인터와 마찬가지로 포인터이며, 함수 또는 메모리 안의 실행 가능한 코드를 가리킵니다. 함수 포인터는 런타임 값에 따라 특정 함수를 별도 검사 없이 호출해 실행하는 간단한 방법을 제공하므로 유용합니다. 우리의 경우 코드 수준에서는 단순히 destroy (page)를 호출하는 것으로 충분하며, 컴파일러는 올바른 함수 포인터를 호출하여 페이지 타입에 맞는 적절한 destroy 루틴을 선택합니다.

페이지 연산을 위한 구조체 struct page_operations는 include/vm/vm.h에 정의되어 있습니다. 이 구조체를 3개의 함수 포인터를 담은 함수 테이블로 생각하면 됩니다.

struct page_operations {
  bool (*swap_in) (struct page *, void *);
  bool (*swap_out) (struct page *);
  void (*destroy) (struct page *);
  enum vm_type type;
};

이제 page_operation 구조체를 어디에서 찾을 수 있는지 살펴보겠습니다. 페이지 구조체 struct page를 include/vm/vm.h에서 보면 operations라는 필드가 있습니다. 이제 vm/file.c로 가면 함수 프로토타입 앞에 선언된 page_operations 구조체 file_ops를 볼 수 있습니다. 이것이 파일 기반 페이지(file-backed page)를 위한 함수 포인터 테이블입니다. .destroy 필드의 값은 file_backed_destroy이며, 이는 같은 파일에 정의되어 페이지를 파괴하는 함수입니다.

함수 포인터 인터페이스를 통해 file_backed_destroy가 어떻게 호출되는지 이해해 보겠습니다. vm_dealloc_page (page)가 호출되었고(vm/vm.c 안에서), 이 페이지가 파일 기반 페이지(VM_FILE)라고 가정합니다. 이 함수 안에서는 destroy (page)를 호출합니다. destroy (page)는 include/vm/vm.h에 다음과 같은 매크로로 정의되어 있습니다.

#define destroy(page) if ((page)->operations->destroy) (page)->operations->destroy (page)

이는 destroy 함수를 호출하면 실제로는 (page)->operations->destroy (page), 즉 페이지 구조체에서 가져온 destroy 함수가 호출된다는 뜻입니다. 해당 페이지는 VM_FILE 페이지이므로 그 .destroy 필드는 file_backed_destory를 가리킵니다. 결과적으로 파일 기반 페이지의 destroy 루틴이 수행됩니다.

보조 페이지 테이블 구현

이 시점의 Pintos에는 메모리의 가상-물리 매핑을 관리하는 페이지 테이블 (pml4)이 있습니다. 그러나 이것만으로는 충분하지 않습니다. 이전 절의 논의처럼, 페이지 폴트(page fault)와 자원 관리를 처리하려면 각 페이지에 대한 추가 정보를 담는 보조 페이지 테이블(supplemental page table)도 필요합니다. 따라서 프로젝트 3의 첫 번째 작업으로 보조 페이지 테이블의 몇 가지 기본 기능을 구현할 것을 제안합니다.

vm/vm.c에 보조 페이지 테이블 관리 함수를 구현하십시오.

먼저 여러분의 Pintos에서 보조 페이지 테이블을 어떻게 설계할지 결정해야 합니다. 자신만의 보조 페이지 테이블을 설계한 뒤, 그 설계에 맞춰 아래 세 함수를 구현하십시오.

void supplemental_page_table_init (struct supplemental_page_table *spt);

보조 페이지 테이블을 초기화합니다. 보조 페이지 테이블에 사용할 자료구조는 여러분이 선택할 수 있습니다. 이 함수는 새 프로세스(process)가 시작될 때 (initd, userprog/process.c 안에서)와 프로세스가 fork될 때 (__do_fork, userprog/process.c 안에서) 호출됩니다.

struct page *spt_find_page (struct supplemental_page_table *spt, void *va);

주어진 보조 페이지 테이블에서 va에 해당하는 struct page를 찾습니다. 실패하면 NULL을 반환합니다.

bool spt_insert_page (struct supplemental_page_table *spt, struct page *page);

주어진 보조 페이지 테이블에 struct page를 삽입합니다. 이 함수는 해당 가상 주소가 주어진 보조 페이지 테이블에 존재하지 않는지 확인해야 합니다.

프레임 관리

이제부터 모든 페이지는 생성될 때의 메모리 메타데이터만 보관하지 않습니다. 따라서 물리 메모리를 관리하기 위한 다른 방식이 필요합니다. include/vm/vm.h에는 물리 메모리를 나타내는 struct frame이 있습니다. 현재 이 구조체는 다음과 같습니다.

/* The representation of "frame" */
struct frame {
  void *kva;
  struct page *page;
};

여기에는 두 필드만 있습니다. kva는 커널 가상 주소이고, page는 페이지 구조체입니다. 프레임 관리 인터페이스를 구현하면서 필요한 멤버를 더 추가할 수 있습니다.

vm_get_frame, vm_claim_page, vm_do_claim_page를 vm/vm.c에 구현하십시오.

static struct frame *vm_get_frame (void);

palloc_get_page를 호출하여 user pool에서 새 물리 페이지를 얻습니다. user pool에서 페이지를 성공적으로 얻으면 프레임(frame)도 할당하고, 그 멤버를 초기화한 뒤 반환합니다. vm_get_frame을 구현한 뒤에는 모든 사용자 공간 페이지(PALLOC_USER)를 이 함수를 통해 할당해야 합니다. 지금은 페이지 할당 실패 시 스왑 아웃(swap out)을 처리할 필요가 없습니다. 그런 경우는 당분간 PANIC ("todo")로 표시해 두면 됩니다.

bool vm_do_claim_page (struct page *page);

페이지를 claim한다는 것은 물리 프레임(physical frame)을 할당한다는 뜻입니다. 먼저 vm_get_frame을 호출하여 프레임을 얻습니다. 이 부분은 템플릿에서 이미 처리되어 있습니다. 그런 다음 MMU를 설정해야 합니다. 다시 말해 페이지 테이블(page table)에 가상 주소에서 물리 주소로의 매핑을 추가해야 합니다. 반환값은 이 작업이 성공했는지 여부를 나타내야 합니다.

bool vm_claim_page (void *va);

va를 할당하기 위해 페이지를 claim합니다. 먼저 페이지를 얻은 뒤, 그 페이지로 vm_do_claim_page를 호출해야 합니다.
