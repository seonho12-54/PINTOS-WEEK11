Project 3을 구현하려면 정상 동작하는 Project 2가 필요한가요?

예.

페이지 폴트(page fault)를 처리한 뒤 프로세스(process)를 어떻게 재개하나요?

page_fault()에서 반환하면 현재 사용자 프로세스(user process)가 재개됩니다. 그러면 명령 포인터(instruction pointer)가 가리키는 명령을 다시 시도합니다. 사용자 프로세스(user process)가 때때로 스택 포인터(stack pointer)보다 위 주소에서 폴트(fault)를 일으키는 이유는 무엇인가요? 스택 증가(stack growth) 테스트에서 사용자 프로그램(user program)이 현재 스택 포인터(stack pointer)보다 위에 있는 주소에서 폴트(fault)를 일으키는 것을 볼 수 있습니다.

가상 메모리(virtual memory) 시스템이 데이터 세그먼트(data segment) 증가를 지원해야 하나요?

아니요. 데이터 세그먼트(data segment)의 크기는 링커(linker)가 결정합니다. Pintos에는 아직 동적 할당(dynamic allocation)이 없습니다(메모리 매핑 파일(memory-mapped file)을 사용하면 사용자 수준(user level)에서 이를 “흉내 낼(fake)” 수는 있습니다). 데이터 세그먼트 증가 지원은 잘 설계된 시스템에 큰 추가 복잡도를 더하지 않아야 합니다.

페이지 프레임(page frame)을 할당할 때 PAL_USER를 사용해야 하는 이유는 무엇인가요?

PAL_USER를 palloc_get_page()에 전달하면 기본 커널 풀(main kernel pool) 대신 사용자 풀(user pool)에서 메모리를 할당합니다. 사용자 풀에서 페이지가 부족해지면 사용자 프로그램(user program)이 페이징(page)을 하게 될 뿐이지만, 커널 풀(kernel pool)에서 페이지가 부족하면 많은 커널(kernel) 함수가 메모리를 확보해야 하므로 여러 실패가 발생합니다. 원한다면 palloc_get_page() 위에 다른 할당자(allocator)를 계층화할 수 있지만, 그 기본 메커니즘은 이 함수여야 합니다. 또한 -ul 커널(kernel) 명령줄 옵션을 사용하여 사용자 풀(user pool)의 크기를 제한할 수 있으므로, 다양한 사용자 메모리(user memory) 크기에서 VM 구현을 쉽게 테스트할 수 있습니다.
