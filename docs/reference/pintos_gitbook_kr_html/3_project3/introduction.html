소개

프로젝트 3: 가상 메모리(Virtual Memory)

이제 여러분은 Pintos의 내부 동작에 어느 정도 익숙해졌을 것입니다. 여러분의 OS는 적절한 동기화를 통해 여러 실행 스레드를 올바르게 처리할 수 있고, 여러 사용자 프로그램을 동시에 로드할 수 있습니다. 그러나 실행할 수 있는 프로그램의 수와 크기는 머신의 주 메모리 크기에 의해 제한됩니다. 이 과제에서는 무한한 메모리가 있는 것처럼 보이게 만들어 그 제한을 제거합니다.

이 과제는 이전 과제 위에 구현합니다. 프로젝트 2의 테스트 프로그램도 프로젝트 3에서 동작해야 합니다. 프로젝트 3 작업을 시작하기 전에 프로젝트 2 제출물의 버그를 모두 수정하도록 주의해야 합니다. 그런 버그는 프로젝트 3에서도 같은 문제를 일으킬 가능성이 높기 때문입니다.

프로젝트 3에서는 여러분이 쉽게 따라갈 수 있도록 단계별 지침을 제공합니다.

배경

소스 파일

이 프로젝트에서는 vm 디렉터리에서 작업합니다. Makefile은 -DVM 설정을 켜도록 업데이트되어 있습니다. 우리는 많은 양의 템플릿 코드를 제공합니다. 여러분은 반드시 주어진 템플릿을 따라야 합니다. 즉, 주어진 템플릿을 기반으로 하지 않은 코드를 제출하면 0pts를 받습니다. 또한 "DO NOT CHANGE"라고 표시된 템플릿 부분은 절대 변경해서는 안 됩니다. 여기서는 여러분이 수정하게 될 각 템플릿 파일에 대해 몇 가지 세부 사항을 제공합니다.

  • include/vm/vm.h, vm/vm.c

    가상 메모리(virtual memory)를 위한 일반 인터페이스를 제공합니다. 헤더 파일에서 여러분의 가상 메모리 시스템이 지원해야 하는 여러 vm_type -- VM_UNINIT, VM_ANON, VM_FILE, VM_PAGE_CACHE -- 의 정의와 설명을 볼 수 있습니다. 지금은 VM_PAGE_CACHE를 무시해도 됩니다. 이것은 프로젝트 4를 위한 것입니다. 여기에서 보조 페이지 테이블(supplemental page table)도 구현합니다. 자세한 내용은 아래를 읽으십시오.

  • include/vm/uninit.h, vm/uninit.c

    초기화되지 않은 페이지(uninitialized page)를 위한 연산을 제공합니다 (vm_type = VM_UNINIT). 현재 설계에서는 모든 페이지(page)가 처음에는 초기화되지 않은 페이지로 설정되고, 이후 익명 페이지(anonymous page) 또는 파일 기반 페이지(file-backed page)로 변환됩니다.

  • include/vm/anon.h, vm/anon.c

    익명 페이지(anonymous page)를 위한 연산을 제공합니다 (vm_type = VM_ANON).

  • include/vm/file.h, vm/file.c

    파일 기반 페이지(file-backed page)를 위한 연산을 제공합니다 (vm_type = VM_FILE).

  • include/vm/inspect.h, vm/inspect.c

    채점을 위한 메모리 검사 연산을 포함합니다. 이 파일들은 변경하지 마십시오.

이 프로젝트에서 여러분이 작성할 코드의 대부분은 vm 디렉터리의 파일과 이전 프로젝트에서 소개된 파일에 들어갑니다. 처음 접하게 될 파일은 아마 다음과 같은 몇 개뿐일 것입니다.

  • include/devices/block.h, devices/block.c

    블록 장치에 대한 섹터 단위 읽기 및 쓰기 접근을 제공합니다. 이 인터페이스를 사용하여 스왑(swap) 파티션에 블록 장치로 접근하게 됩니다.

메모리 용어

먼저 메모리와 저장장치에 관한 몇 가지 용어를 소개합니다. 이 용어들 중 일부는 프로젝트 2에서 익숙해졌을 것입니다 (가상 메모리 배치(Virtual Memory Layout) 참조). 하지만 대부분은 새로운 내용입니다.

페이지(Pages)

페이지(page)는 가상 페이지(virtual page)라고도 하며, 길이가 4,096 bytes인 가상 메모리의 연속 영역입니다. 이 크기를 페이지 크기(page size)라고 합니다. 페이지는 반드시 페이지 정렬(page-aligned)되어야 합니다. 즉, 페이지 크기로 나누어떨어지는 가상 주소에서 시작해야 합니다. 따라서 64-bit 가상 주소의 마지막 12 bits는 페이지 오프셋(page offset) 또는 간단히 오프셋(offset)입니다. 상위 비트들은 곧 소개할 페이지 테이블(page table)의 인덱스를 나타내는 데 사용됩니다. 64-bit 시스템에서는 4단계 페이지 테이블을 사용하므로 가상 주소는 다음과 같은 형태가 됩니다.

63          48 47            39 38            30 29            21 20         12 11         0
+-------------+----------------+----------------+----------------+-------------+------------+
| Sign Extend |    Page-Map    | Page-Directory | Page-directory |  Page-Table |    Page    |
|             | Level-4 Offset |    Pointer     |     Offset     |   Offset    |   Offset   |
+-------------+----------------+----------------+----------------+-------------+------------+
              |                |                |                |             |            |
              +------- 9 ------+------- 9 ------+------- 9 ------+----- 9 -----+---- 12 ----+
                                          Virtual Address

각 프로세스(process)는 독립적인 사용자 가상 페이지 집합을 가집니다. 이는 가상 주소 KERN_BASE (0x8004000000) 아래에 있는 페이지들입니다. 반면 커널 가상 페이지(kernel virtual pages)의 집합은 전역적이므로, 어떤 스레드(thread)나 프로세스가 실행 중인지와 관계없이 항상 같은 위치에 남아 있습니다. 커널(kernel)은 사용자 페이지와 커널 페이지 모두에 접근할 수 있지만, 사용자 프로세스는 자신의 사용자 페이지에만 접근할 수 있습니다. 자세한 내용은 가상 메모리 배치(Virtual Memory Layout)를 참조하십시오.

Pintos는 가상 주소를 다루기 위한 여러 유용한 함수를 제공합니다. 자세한 내용은 가상 주소(Virtual Addresses) 절을 참조하십시오.

프레임(Frames)

프레임(frame)은 물리 프레임(physical frame) 또는 페이지 프레임(page frame)이라고도 하며, 물리 메모리의 연속 영역입니다. 페이지와 마찬가지로 프레임도 페이지 크기여야 하며 페이지 정렬되어야 합니다. 따라서 64-bit 물리 주소는 다음과 같이 프레임 번호(frame number)와 프레임 오프셋(offset) 또는 간단히 오프셋(offset)으로 나눌 수 있습니다.

                          12 11         0
    +-----------------------+-----------+
    |      Frame Number     |   Offset  |
    +-----------------------+-----------+
              Physical Address

x86-64는 물리 주소의 메모리에 직접 접근할 방법을 제공하지 않습니다. Pintos는 커널 가상 메모리(kernel virtual memory)를 물리 메모리에 직접 매핑하여 이 문제를 우회합니다. 커널 가상 메모리의 첫 번째 페이지는 물리 메모리의 첫 번째 프레임에, 두 번째 페이지는 두 번째 프레임에 매핑되는 식입니다. 따라서 프레임은 커널 가상 메모리를 통해 접근할 수 있습니다.

Pintos는 물리 주소와 커널 가상 주소 사이를 변환하는 함수들을 제공합니다. 자세한 내용은 가상 주소(Virtual Addresses)를 참조하십시오.

페이지 테이블(Page Tables)

페이지 테이블(page table)은 CPU가 가상 주소를 물리 주소로, 즉 페이지(page)를 프레임(frame)으로 변환하는 데 사용하는 자료구조입니다. 페이지 테이블 형식은 x86-64 아키텍처에 의해 결정됩니다. Pintos는 threads/mmu.c에서 페이지 테이블 관리 코드를 제공합니다.

아래 그림은 페이지와 프레임의 관계를 보여 줍니다. 왼쪽의 가상 주소는 페이지 번호와 오프셋으로 구성됩니다. 페이지 테이블은 페이지 번호를 프레임 번호로 변환하고, 이 프레임 번호와 변경되지 않은 오프셋을 결합하여 오른쪽의 물리 주소를 얻습니다.

                          +----------+
         .--------------->|Page Table|-----------.
        /                 +----------+            |
        |   12 11 0                               V  12 11 0
    +---------+----+                         +---------+----+
    | Page Nr | Ofs|                         |Frame Nr | Ofs|
    +---------+----+                         +---------+----+
     Virt Addr   |                            Phys Addr    ^
                  \_______________________________________/

스왑 슬롯(Swap Slots)

스왑 슬롯(swap slot)은 스왑(swap) 파티션 안에 있는 페이지 크기의 디스크 공간 영역입니다. 슬롯 배치를 지배하는 하드웨어 제약은 프레임보다 더 유연하지만, 스왑 슬롯은 페이지 정렬되어야 합니다. 그렇게 해서 손해가 없기 때문입니다.

반드시 세 가지 자료구조를 완전히 별개로 구현할 필요는 없습니다. 관련 자원을 하나의 통합 자료구조로 전체 또는 일부 병합하는 편이 더 편리할 수도 있습니다.

각 자료구조에 대해 각 원소가 어떤 정보를 담아야 하는지 결정해야 합니다. 또한 자료구조의 범위가 로컬(프로세스별)인지 전역(전체 시스템에 적용)인지, 그리고 그 범위 안에서 몇 개의 인스턴스가 필요한지도 결정해야 합니다.

설계를 단순화하기 위해 이러한 자료구조를 페이지 아웃되지 않는(non-pageable) 메모리(예: calloc 또는 malloc으로 할당한 메모리)에 저장할 수 있습니다. 이는 자료구조들 사이의 포인터가 계속 유효하게 유지됨을 보장할 수 있다는 뜻입니다.

구현 선택지(성능 관점)

가능한 구현 선택지에는 배열, 리스트, 비트맵, 해시 테이블(hash table)이 있습니다. 배열은 대개 가장 단순한 접근법이지만, 드문드문 채워진 배열은 메모리를 낭비합니다. 리스트도 단순하지만 특정 위치를 찾기 위해 긴 리스트를 순회하면 시간을 낭비합니다. 배열과 리스트는 모두 크기를 변경할 수 있지만, 리스트는 중간 삽입과 삭제를 더 효율적으로 지원합니다.

Pintos는 lib/kernel/bitmap.c와 include/lib/kernel/bitmap.h에 비트맵 자료구조를 포함합니다. 비트맵은 비트의 배열이며, 각 비트는 true 또는 false일 수 있습니다. 비트맵은 보통 동일한 자원 집합의 사용 상태를 추적하는 데 사용됩니다. 자원 n이 사용 중이면 비트맵의 비트 n이 true입니다. Pintos 비트맵의 크기는 고정되어 있지만, 구현을 확장하여 크기 변경을 지원할 수도 있습니다.

Pintos는 해시 테이블(hash table) 자료구조도 포함합니다 (해시 테이블(Hash Table) 참조). Pintos 해시 테이블은 넓은 범위의 테이블 크기에서 삽입과 삭제를 효율적으로 지원합니다.

더 복잡한 자료구조가 더 나은 성능이나 다른 이점을 제공할 수도 있지만, 여러분의 구현을 불필요하게 복잡하게 만들 수도 있습니다. 따라서 설계의 일부로 고급 자료구조(예: 균형 이진 트리)를 구현하는 것은 권장하지 않습니다.

보조 페이지 테이블 관리

보조(supplemental) 페이지 테이블은 각 페이지에 대한 추가 데이터로 페이지 테이블(page table)을 보완합니다. 페이지 테이블 형식이 갖는 제한 때문에 필요합니다. 이런 자료구조도 흔히 "페이지 테이블"이라고 부르지만, 혼동을 줄이기 위해 "보조(supplemental)"라는 단어를 붙입니다.

보조 페이지 테이블(supplemental page table)은 적어도 두 가지 용도로 사용됩니다. 가장 중요한 용도는 페이지 폴트(page fault)가 발생했을 때, 커널(kernel)이 폴트를 일으킨 가상 페이지를 보조 페이지 테이블에서 찾아 그 페이지에 어떤 데이터가 있어야 하는지 알아내는 것입니다. 두 번째로, 커널은 프로세스(process)가 종료될 때 어떤 자원을 해제할지 결정하기 위해 보조 페이지 테이블을 참조합니다.

보조 페이지 테이블의 구성

보조 페이지 테이블은 원하는 방식으로 구성할 수 있습니다. 기본적인 접근법은 적어도 두 가지가 있습니다. 세그먼트(segment) 단위로 구성하거나 페이지 단위로 구성하는 방식입니다. 여기서 세그먼트는 연속된 페이지들의 묶음, 즉 실행 파일이나 메모리 매핑 파일(memory-mapped file)을 포함하는 메모리 영역을 의미합니다.

선택 사항으로, 페이지 테이블 자체를 사용해 보조 페이지 테이블의 멤버를 추적할 수도 있습니다. 그렇게 하려면 threads/mmu.c의 Pintos 페이지 테이블 구현을 수정해야 합니다. 이 접근법은 고급 학생에게만 권장합니다.

페이지 폴트 처리

보조 페이지 테이블의 가장 중요한 사용자는 페이지 폴트(page fault) 핸들러입니다. 프로젝트 2에서 페이지 폴트는 항상 커널이나 사용자 프로그램(user program)의 버그를 의미했습니다. 프로젝트 3에서는 더 이상 그렇지 않습니다. 이제 페이지 폴트는 페이지를 파일이나 스왑 슬롯(swap slot)에서 가져와야 함을 나타낼 뿐일 수 있습니다. 이러한 경우를 처리하려면 더 정교한 페이지 폴트 핸들러를 구현해야 합니다. 페이지 폴트 핸들러인 page_fault()는 userprog/exception.c에 있으며, 여러분의 페이지 폴트 핸들러인 vm_try_handle_fault()를 호출합니다. 이 함수는 vm/vm.c에 있습니다. 여러분의 페이지 폴트 핸들러는 대략 다음을 수행해야 합니다.

  1. 보조 페이지 테이블에서 폴트를 일으킨 페이지를 찾습니다. 메모리 참조가 유효하다면, 보조 페이지 테이블 엔트리를 사용해 해당 페이지에 들어갈 데이터를 찾습니다. 그 데이터는 파일 시스템이나 스왑 슬롯에 있을 수도 있고, 단순히 모두 0인 페이지일 수도 있습니다. 공유(예: 쓰기 시 복사(Copy-on-Write))를 구현한다면 페이지의 데이터가 이미 페이지 프레임(page frame)에 있지만 페이지 테이블에는 없을 수도 있습니다. 보조 페이지 테이블이 사용자 프로세스가 접근하려던 주소에서 어떤 데이터도 기대해서는 안 된다고 나타내거나, 페이지가 커널 가상 메모리 안에 있거나, 접근이 읽기 전용 페이지에 쓰려는 시도라면 그 접근은 유효하지 않습니다. 유효하지 않은 접근은 모두 프로세스를 종료하고 그 결과 모든 자원을 해제합니다.

  2. 페이지를 저장할 프레임(frame)을 얻습니다. 공유를 구현한다면 필요한 데이터가 이미 프레임에 있을 수 있으므로, 그 경우 해당 프레임을 찾을 수 있어야 합니다.

  3. 파일 시스템이나 스왑(swap)에서 읽거나 0으로 채우는 등의 방식으로 데이터를 프레임 안으로 가져옵니다. 공유를 구현한다면 필요한 페이지가 이미 프레임 안에 있을 수 있으며, 그 경우 이 단계에서는 아무 작업도 필요하지 않습니다.

  4. 폴트를 일으킨 가상 주소의 페이지 테이블 엔트리가 물리 페이지를 가리키도록 합니다. threads/mmu.c의 함수들을 사용할 수 있습니다.

프레임 테이블 관리

프레임 테이블(frame table)은 각 프레임마다 하나의 엔트리를 포함합니다. 프레임 테이블의 각 엔트리는 현재 그 프레임을 점유하고 있는 페이지가 있다면 그 페이지에 대한 포인터와, 여러분이 선택한 다른 데이터를 포함합니다. 프레임 테이블은 빈 프레임이 없을 때 축출할 페이지를 선택함으로써 Pintos가 축출 정책을 효율적으로 구현할 수 있게 합니다.

사용자 페이지에 쓰이는 프레임은 palloc_get_page(PAL_USER)를 호출하여 "user pool"에서 얻어야 합니다. 예기치 않게 일부 테스트 케이스가 실패할 수 있는 "kernel pool"에서 할당하지 않도록 반드시 PAL_USER를 사용해야 합니다. 프레임 테이블 구현의 일부로 palloc.c를 수정한다면 두 풀의 구분을 반드시 유지하십시오.

프레임 테이블에서 가장 중요한 연산은 사용되지 않는 프레임을 얻는 것입니다. 프레임이 비어 있으면 이는 쉽습니다. 비어 있는 프레임이 하나도 없으면 어떤 페이지를 그 프레임에서 축출하여 프레임을 비워야 합니다.

스왑 슬롯을 할당하지 않고는 어떤 프레임도 축출할 수 없지만 스왑(swap)이 가득 차 있다면 커널을 panic하십시오. 실제 OS는 이런 상황에서 복구하거나 이를 방지하기 위해 다양한 정책을 적용하지만, 이러한 정책은 이 프로젝트의 범위를 벗어납니다.

축출(eviction) 과정은 대략 다음 단계로 구성됩니다.

  1. 여러분의 페이지 교체(page replacement) 알고리즘을 사용해 축출할 프레임을 선택합니다. 아래에서 설명하는 페이지 테이블의 "accessed" 비트와 "dirty" 비트가 유용하게 쓰일 것입니다.

  2. 해당 프레임을 참조하는 모든 페이지 테이블에서 그 프레임에 대한 참조를 제거합니다. 공유를 구현하지 않았다면, 어떤 시점이든 하나의 프레임은 오직 하나의 페이지에서만 참조되어야 합니다.

  3. 필요하다면 페이지를 파일 시스템이나 스왑(swap)에 씁니다. 그러면 축출된 프레임은 다른 페이지를 저장하는 데 사용할 수 있습니다.

접근 비트와 더티 비트(Accessed and Dirty Bits)

x86-64 하드웨어는 각 페이지의 페이지 테이블 엔트리(PTE)에 있는 한 쌍의 비트를 통해 페이지 교체 알고리즘 구현을 일부 지원합니다. 페이지를 읽거나 쓸 때마다 CPU는 그 페이지의 PTE에 있는 접근 비트(accessed bit)를 1로 설정하고, 쓸 때마다 CPU는 더티 비트(dirty bit)를 1로 설정합니다. CPU는 이 비트들을 0으로 재설정하지 않지만, OS는 그렇게 할 수 있습니다.

aliases, 즉 같은 프레임을 참조하는 두 개 이상의 페이지에 주의해야 합니다. 별칭이 있는 프레임에 접근하면 접근 비트와 더티 비트는 하나의 페이지 테이블 엔트리, 즉 접근에 사용된 페이지의 엔트리에서만 갱신됩니다. 다른 별칭의 접근 비트와 더티 비트는 갱신되지 않습니다.

Pintos에서는 모든 사용자 가상 페이지가 해당 커널 가상 페이지의 별칭입니다. 여러분은 이러한 별칭을 어떻게든 관리해야 합니다. 예를 들어, 여러분의 코드는 두 주소의 접근 비트와 더티 비트를 모두 확인하고 갱신할 수 있습니다. 또는 커널이 사용자 가상 주소를 통해서만 사용자 데이터에 접근하여 이 문제를 피할 수도 있습니다.

다른 별칭은 공유를 구현했거나 여러분의 코드에 버그가 있을 때에만 생겨야 합니다.

접근 비트와 더티 비트를 다루는 함수의 자세한 내용은 페이지 테이블 접근 비트와 더티 비트(Page Table Accessed and Dirty Bits) 절을 참조하십시오.

스왑 테이블 관리

스왑 테이블(swap table)은 사용 중인 스왑 슬롯과 비어 있는 스왑 슬롯을 추적합니다. 페이지를 프레임에서 스왑 파티션으로 축출할 때 사용되지 않는 스왑 슬롯을 선택할 수 있어야 합니다. 또한 해당 페이지를 다시 읽어 오거나, 스왑된 페이지를 가진 프로세스가 종료될 때 스왑 슬롯을 해제할 수 있어야 합니다.

vm/build 디렉터리에서 pintos-mkdisk swap.dsk --swap-size=n 명령을 사용하여 n-MB 스왑 파티션을 포함하는 swap.dsk라는 디스크를 만드십시오. 이후 pintos를 실행하면 swap.dsk가 자동으로 추가 디스크로 연결됩니다. 또는 --swap-size=n을 사용해 단일 실행에 임시 n-MB 스왑 디스크를 사용하도록 pintos에 지시할 수도 있습니다.

스왑 슬롯(swap slot)은 지연 할당(lazy allocation)되어야 합니다. 즉, 축출에 실제로 필요할 때에만 할당되어야 합니다. 프로세스 시작 시 실행 파일에서 데이터 페이지를 읽어 곧바로 스왑에 쓰는 것은 지연 방식이 아닙니다. 특정 페이지를 저장하기 위해 스왑 슬롯을 미리 예약해서는 안 됩니다.

스왑 슬롯의 내용이 프레임으로 다시 읽혀 들어오면 그 스왑 슬롯을 해제하십시오.

메모리 매핑 파일 관리

파일 시스템은 가장 흔히 read와 write 시스템 콜(system call)로 접근합니다. 보조 인터페이스는 mmap 시스템 콜을 사용해 파일을 가상 페이지에 "매핑"하는 것입니다. 그러면 프로그램은 파일 데이터에 대해 메모리 명령을 직접 사용할 수 있습니다. 파일 foo의 길이가 0x1000 bytes(4 kB, 즉 페이지 하나)라고 가정해 보겠습니다. foo가 주소 0x5000부터 메모리에 매핑되었다면, 위치 0x5000. . .0x5fff에 대한 모든 메모리 접근은 foo의 해당 바이트에 접근합니다.

다음은 mmap을 사용해 파일을 콘솔에 출력하는 프로그램입니다. 명령줄에 지정된 파일을 열고, 가상 주소 0x10000000에 매핑한 뒤, 매핑된 데이터를 콘솔(fd 1)에 쓰고 파일 매핑을 해제합니다.

#include <stdio.h>
#include <syscall.h>
int main (int argc UNUSED, char *argv[])
{
  void *data = (void *) 0x10000000;                 /* Address at which to map. */
  int fd = open (argv[1]);                          /* Open file. */
  void *map = mmap (data, filesize (fd), 0, fd, 0); /* Map file. */
  write (1, data, filesize (fd));                   /* Write file to console. */
  munmap (map);                                     /* Unmap file (optional). */
  return 0;
}

여러분의 제출물은 메모리 매핑 파일(memory mapped file)이 어떤 메모리를 사용하는지 추적할 수 있어야 합니다. 이는 매핑된 영역의 페이지 폴트(page fault)를 올바르게 처리하고, 매핑된 파일이 프로세스(process) 안의 다른 세그먼트와 겹치지 않도록 보장하는 데 필요합니다.
