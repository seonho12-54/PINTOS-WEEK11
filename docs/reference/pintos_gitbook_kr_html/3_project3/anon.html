익명 페이지(Anonymous Page)

익명 페이지(Anonymous Page)

이 프로젝트의 이 부분에서는 익명 페이지(anonymous page)라고 하는 디스크 기반이 아닌 이미지를 구현합니다.

익명 매핑(anonymous mapping)에는 백업 파일이나 장치가 없습니다. 파일 기반 페이지(file-backed page)와 달리 이름이 붙은 파일 소스가 없기 때문에 익명이라고 부릅니다. 익명 페이지는 스택(stack)과 힙(heap)처럼 실행 파일에서 사용됩니다.

익명 페이지를 설명하는 구조체인 anon_page는 include/vm/anon.h에 있습니다. 현재는 비어 있지만, 구현하면서 익명 페이지에 필요한 정보나 상태를 저장할 멤버를 추가할 수 있습니다. 또한 페이지의 일반 정보를 담고 있는 struct page는 include/vm/page.h에서 확인할 수 있습니다. 익명 페이지의 경우 페이지 구조체 안에 struct anon_page anon이 포함됩니다.

지연 로딩(lazy loading)을 통한 페이지 초기화

지연 로딩(lazy loading)은 메모리 로드를 실제로 필요한 시점까지 미루는 설계입니다. 페이지가 할당되었다는 것은 해당 페이지에 대응하는 페이지 구조체가 존재한다는 뜻이지만, 전용 물리 프레임(frame)은 없고 페이지의 실제 내용도 아직 로드되지 않았습니다. 내용은 페이지 폴트(page fault)로 필요성이 신호될 때, 즉 실제로 필요한 시점에만 로드됩니다.

페이지 타입이 세 가지이므로 초기화 루틴은 페이지마다 다릅니다. 아래 섹션에서 다시 설명하겠지만, 여기서는 페이지 초기화 흐름을 높은 수준에서 설명합니다. 먼저 커널(kernel)이 새 페이지 요청을 받으면 vm_alloc_page_with_initializer가 호출됩니다. 초기화 함수(initializer)는 페이지 구조체를 할당하고 페이지 타입에 맞는 초기화 함수를 설정하여 새 페이지를 초기화한 뒤 제어를 사용자 프로그램(user program)에 돌려줍니다. 사용자 프로그램이 실행되다 보면 어느 시점에 프로그램이 자신이 가지고 있다고 믿는 페이지에 접근하려고 하지만 그 페이지에 아직 내용이 없어서 페이지 폴트가 발생합니다. 폴트 처리 과정에서 uninit_initialize가 호출되고, 앞서 설정한 초기화 함수를 호출합니다. 익명 페이지의 초기화 함수는 anon_initializer, 파일 기반 페이지의 초기화 함수는 file_backed_initializer가 됩니다.

페이지는 initialize->(page_fault->lazy-load->swap-in>swap-out->...)->destroy. 형태의 생명 주기를 가질 수 있습니다. 생명 주기의 각 전환마다 필요한 절차는 페이지 타입(또는 VM_TYPE)에 따라 달라지며, 이전 단락은 초기화의 예시였습니다. 이 프로젝트에서는 각 페이지 타입에 대해 이러한 전환 과정을 구현합니다.

실행 파일의 지연 로딩(lazy loading)

지연 로딩에서는 프로세스(process)가 실행을 시작할 때 즉시 필요한 메모리 부분만 메인 메모리에 로드됩니다. 이렇게 하면 전체 바이너리 이미지를 한 번에 메모리에 로드하는 즉시 로딩(eager loading)과 비교해 오버헤드를 줄일 수 있습니다.

지연 로딩을 지원하기 위해 VM_UNINIT이라는 페이지 타입을 include/vm/vm.h에 도입합니다. 모든 페이지는 처음에 VM_UNINIT 페이지로 생성됩니다. 또한 미초기화 페이지(uninitialized page)를 위한 페이지 구조체인 struct uninit_page를 include/vm/uninit.h에 제공합니다. 미초기화 페이지를 생성, 초기화, 소멸하는 함수는 include/vm/uninit.c에서 찾을 수 있습니다. 이 함수들은 나중에 완성해야 합니다.

페이지 폴트(page fault)가 발생하면 페이지 폴트 핸들러(page_fault, userprog/exception.c)가 제어를 vm_try_handle_fault(vm/vm.c)로 넘기며, 이 함수는 먼저 유효한 페이지 폴트인지 확인합니다. 여기서 valid는 유효하지 않은 영역에 접근한 폴트를 의미합니다. bogus 폴트라면 페이지에 일부 내용을 로드하고 사용자 프로그램(user program)에 제어를 반환합니다.

bogus 페이지 폴트에는 지연 로드된 페이지, 스왑 아웃된 페이지(swapped-out page), 쓰기 보호된 페이지 (write-protected page)의 세 가지 경우가 있습니다(쓰기 시 복사(Copy-on-Write, Extra) 참조). 지금은 첫 번째 경우인 지연 로드된 페이지만 고려하십시오. 지연 로딩을 위한 페이지 폴트라면, 커널(kernel)은 세그먼트를 지연 로드하기 위해 이전에 vm_alloc_page_with_initializer에서 설정한 초기화 함수 중 하나를 호출합니다. lazy_load_segment를 userprog/process.c에 구현해야 합니다.

vm_alloc_page_with_initializer()를 구현하십시오. 전달된 vm_type에 따라 적절한 초기화 함수를 가져와 그것으로 uninit_new를 호출해야 합니다.

bool vm_alloc_page_with_initializer (enum vm_type type, void *va,
        bool writable, vm_initializer *init, void *aux);

주어진 타입으로 미초기화 페이지를 생성합니다. uninit 페이지의 swap_in 핸들러는 타입에 따라 페이지를 자동으로 초기화하고, 주어진 AUX로 INIT을 호출합니다. 페이지 구조체를 얻으면 해당 페이지를 프로세스(process)의 보조 페이지 테이블(supplemental page table)에 삽입하십시오. VM_TYPE 매크로는 vm.h에 정의되어 있으며 유용할 수 있습니다.

페이지 폴트(page fault) 핸들러는 호출 체인을 따라가고, swap_in을 호출할 때 마침내 uninit_intialize에 도달합니다. 이에 대한 전체 구현은 제공되어 있습니다. 다만 설계에 따라 uninit_initialize를 수정해야 할 수도 있습니다.

static bool uninit_initialize (struct page *page, void *kva);

첫 번째 폴트에서 페이지를 초기화합니다. 템플릿 코드는 먼저 vm_initializer와 aux를 가져온 뒤, 함수 포인터를 통해 해당 page_initializer를 호출합니다. 설계에 따라 이 함수를 수정해야 할 수도 있습니다.

vm_anon_init과 anon_initializer는 필요에 따라 vm/anon.c에서 수정할 수 있습니다.

void vm_anon_init (void);

익명 페이지(anonymous page) 서브시스템을 초기화합니다. 이 함수에서는 익명 페이지와 관련된 모든 것을 설정할 수 있습니다.

bool anon_initializer (struct page *page,enum vm_type type, void *kva);

이 함수는 먼저 page->operations에 익명 페이지(anonymous page)의 핸들러를 설정합니다. 현재 빈 구조체인 anon_page에 일부 정보를 업데이트해야 할 수도 있습니다. 이 함수는 익명 페이지(즉 VM_ANON)의 초기화 함수로 사용됩니다.

load_segment와 lazy_load_segment를 userprog/process.c에 구현하십시오. 실행 파일에서 세그먼트 로딩을 구현합니다. 이 모든 페이지는 지연 로드되어야 합니다. 즉, 커널(kernel)이 이 페이지들에 대한 페이지 폴트(page fault)를 가로챌 때에만 로드됩니다.

프로그램 로더의 핵심인 load_segment(userprog/process.c) 안의 루프를 수정해야 합니다. 루프를 한 번 돌 때마다 대기 중인 페이지 객체를 만들기 위해 vm_alloc_page_with_initializer를 호출합니다. 페이지 폴트가 발생하는 바로 이 시점에 세그먼트가 실제로 파일에서 로드됩니다.

static bool load_segment (struct file *file, off_t ofs, uint8_t *upage,
        uint32_t read_bytes, uint32_t zero_bytes, bool writable);

현재 코드는 메인 루프 안에서 파일에서 읽을 바이트 수와 0으로 채울 바이트 수를 계산합니다. 그런 다음 대기 중인 객체를 만들기 위해 vm_alloc_page_with_initializer를 호출합니다. 보조 값은 aux 인자로 설정해야 하며, 이 인자는 vm_alloc_page_with_initializer에 전달됩니다. 바이너리를 로드하는 데 필요한 정보를 담은 구조체를 만들고 싶을 수 있습니다.

static bool lazy_load_segment (struct page *page, void *aux);

lazy_load_segment가 vm_alloc_page_with_initializer의 네 번째 인자로 load_segment에서 제공된다는 점을 눈치챘을 수 있습니다. 이 함수는 실행 파일 페이지의 초기화 함수이며 페이지 폴트가 발생할 때 호출됩니다. 페이지 구조체와 aux를 인자로 받습니다. aux는 load_segment에서 설정한 정보입니다. 이 정보를 사용해 세그먼트를 읽을 파일을 찾아 결국 세그먼트를 메모리로 읽어 와야 합니다.

setup_stack을 userprog/process.c에서 조정하여 스택(stack) 할당이 새 메모리 관리 시스템에 맞게 하십시오. 첫 번째 스택 페이지는 지연 할당할 필요가 없습니다. 폴트가 발생하기를 기다릴 필요 없이 로드 시점에 명령줄 인자로 할당하고 초기화할 수 있습니다. 스택을 식별하는 방법을 제공해야 할 수도 있습니다. 페이지를 표시하기 위해 vm_type(vm/vm.h)의 보조 마커(예: VM_MARKER_0)를 사용할 수 있습니다.

마지막으로 vm_try_handle_fault 함수를 수정하여, spt_find_page를 통해 보조 페이지 테이블을 조회함으로써 폴트가 난 주소에 대응하는 페이지 구조체를 찾도록 하십시오.

요구사항을 모두 구현하면 fork를 제외한 프로젝트 2의 모든 테스트가 통과해야 합니다.

보조 페이지 테이블(supplemental page table) 다시 살펴보기

이제 복사와 정리 작업을 지원하기 위해 보조 페이지 테이블(supplemental page table) 인터페이스를 다시 살펴봅니다. 이 작업들은 프로세스를 생성할 때(더 구체적으로는 자식 프로세스를 만들 때) 또는 프로세스를 소멸할 때 필요합니다. 자세한 내용은 아래에서 설명합니다. 이 시점에 보조 페이지 테이블을 다시 살펴보는 이유는 위에서 구현한 초기화 함수 일부를 사용하고 싶을 수 있기 때문입니다.

supplemental_page_table_copy와 supplemental_page_table_kill을 vm/vm.c에 구현하십시오.

bool supplemental_page_table_copy (struct supplemental_page_table *dst,
    struct supplemental_page_table *src);

src에서 dst로 보조 페이지 테이블(supplemental page table)을 복사합니다. 이는 자식이 부모의 실행 컨텍스트 (즉 fork())를 상속해야 할 때 사용됩니다. src의 보조 페이지 테이블에 있는 각 페이지를 순회하여 dst의 보조 페이지 테이블에 엔트리를 정확히 복사하십시오. uninit 페이지를 할당하고 즉시 클레임(claim)해야 합니다.

void supplemental_page_table_kill (struct supplemental_page_table *spt);

보조 페이지 테이블(supplemental page table)이 보유하던 모든 리소스를 해제합니다. 이 함수는 프로세스(process)가 종료될 때(process_exit(), userprog/process.c) 호출됩니다. 페이지 엔트리를 순회하며 테이블 안의 페이지들에 대해 destroy(page)를 호출해야 합니다. 이 함수에서는 실제 페이지 테이블(page table)(pml4)과 물리 메모리(palloc-ed memory)를 걱정할 필요가 없습니다. 호출자가 보조 페이지 테이블을 정리한 뒤 그것들을 정리합니다.

페이지 정리(Page Cleanup)

uninit_destroy를 vm/uninit.c에, anon_destroy를 vm/anon.c에 구현하십시오. 이 함수는 미초기화 페이지의 destroy 연산을 처리하는 핸들러입니다. 미초기화 페이지가 다른 페이지 객체로 변환되더라도, 프로세스(process)가 종료될 때 uninit 페이지가 여전히 남아 있을 수 있습니다.

static void uninit_destroy (struct page *page);

페이지 구조체가 보유하던 리소스를 해제합니다. 페이지의 vm type을 확인하고 그에 맞게 처리하고 싶을 수 있습니다.

지금은 익명 페이지(anonymous page)만 처리하면 됩니다. 나중에 이 함수를 다시 살펴보고 파일 기반 페이지를 정리하게 됩니다.

static void anon_destroy (struct page *page);

익명 페이지(anonymous page)가 보유하던 리소스를 해제합니다. 페이지 구조체를 명시적으로 해제할 필요는 없습니다. 호출자가 수행해야 합니다.
