이 map은 “어디에 놓여 있는가”보다 “누가 어떤 주소로 접근하는가”를 보여 주는 자료입니다. kernel direct mapping과 user page table mapping을 분리해서 보세요.

Memory map

가운데는 현재 process의 virtual address space입니다. 위쪽 kernel 영역은 모든 process에서 공통이고, 아래쪽 user 영역은 process마다 다릅니다. 오른쪽은 실제 RAM입니다.

solid arrow = kernel direct map, dashed arrow = user page table map

kernel은 physical address를 직접 포인터처럼 쓰지 않고, `KERN_BASE + physical address`라는 kernel virtual address로 RAM을 봅니다.

physical frame이 user page에 붙어 있으면, 같은 frame을 user virtual address와 kernel direct-map address 양쪽 virtual mapping으로 볼 수 있습니다.

user pool은 오른쪽 physical memory 안의 전역 frame 출처입니다. kernel direct map으로 접근할 수 있지만, process마다 따로 생기는 kernel 영역이 아니라 allocator가 나눈 물리 frame 집합입니다.

stack, BSS, data, code 같은 mapped page는 user-pool frame을 붙일 수 있지만, gap이나 unmapped area는 아직 frame을 갖지 않습니다.

process마다 user virtual address space와 page table 묶음이 따로 있습니다. 가운데 user 공간은 현재 선택된 process의 관점이고, 같은 upage라도 다른 process에서는 다른 physical frame 또는 아직 frame이 없는 상태를 가리킬 수 있습니다.

user pool 자체는 switch되지 않습니다. 바뀌는 것은 CPU가 사용하는 active PML4이고, 그 결과 같은 user VA가 전역 user pool 안의 어느 frame을 보느냐가 달라집니다. 같은 process 안의 thread들은 같은 user address space를 공유하고, 다른 process의 thread는 다른 page-table 묶음을 봅니다.

CPU의 virtual address 표현 범위는 가능한 주소 이름의 최대치이고, Pintos가 실제로 resident page를 담는 한계는 physical RAM, user/kernel pool 분할, 현재 만든 page-table mapping에 의해 더 작게 정해집니다.

이 탭은 주소 영역을 실제 데이터 종류와 연결합니다. 아래 예시는 각 영역에 어떤 byte나 metadata가 들어가는지 설명하기 위한 축약 예시입니다.

user code / data / BSS / stack

사용자 프로그램 ELF와 실행 중 stack이 가운데 아래 user virtual range에 놓입니다.

int initialized = 7; // user initialized data static char zero_buf[4096]; // user BSS int main(int argc, char **argv) { int local = argc; // user stack return initialized + local; // instruction bytes = user code }

user pool frame

mapped user page가 실제 RAM에 올라오면 오른쪽 user pool의 frame 하나가 그 page 내용을 담습니다.

mapped user virtual page page table entry -> user-pool frame frame bytes -> code / data / BSS / stack contents free user-pool frame no current user page contents

kernel code / data / BSS

kernel 함수의 기계어와 kernel 전역 상태는 KERN_BASE 위쪽 kernel virtual range에 보입니다.

kernel text scheduler / interrupt / MMU helper instructions kernel data and BSS ready list, locks, allocator state, page tables

thread page / page table metadata

왼쪽 thread page는 user stack이 아니라 kernel이 thread를 관리하기 위한 page입니다.

thread page lower area thread metadata: tid, status, pml4 pointer, magic thread page upper area kernel stack and saved interrupt/syscall registers

bitmap과 page table은 page 내용을 저장하지 않습니다. bitmap은 frame의 free/used 상태를, page table은 VA -> frame 연결을 기록합니다.

실제 user code/data/stack byte는 오른쪽 user pool frame 안에 있고, gap이나 lazy page처럼 frame이 없는 상태는 metadata로만 남습니다.

1 0 1 1 0 0 1 0

metadata와 data의 크기 감각

bitmap은 page당 1 bit라서 작습니다. 큰 메모리는 bitmap이 아니라 실제 frame byte에서 발생합니다.

10000 frames * 1 bit = 10000 bits 10000 bits / 8 = 1250 bytes ~= 1.22 KiB 10000 frames * 4 KiB ~= 39.1 MiB data

VM에 주소 범위가 보인다는 말과 실제 RAM byte가 있다는 말은 다릅니다. 실제 접근 가능한 page는 page table mapping을 통해 physical frame에 닿습니다.

초록은 실제 byte-backed mapping, 주황은 metadata 또는 예약 상태입니다. metadata 자체도 kernel memory에 저장되는 byte지만 user page 내용은 아닙니다.

1

kernel direct map과 KERN_BASE

Pintos는 physical memory를 `KERN_BASE` 위 kernel virtual address에 일대일로 매핑합니다. 그래서 kernel은 physical address를 포인터처럼 직접 쓰지 않고, `KERN_BASE + physical address` 형태의 kernel virtual address로 frame을 접근합니다.

kernel virtual address = KERN_BASE + physical address physical address = kernel virtual address - KERN_BASE

`KERN_BASE`는 동시에 user/kernel 경계입니다. physical frame이 user page에 붙어 있으면, 같은 frame을 user VA와 kernel direct-map VA 양쪽에서 볼 수 있습니다.

이 direct-map 범위는 CPU가 표현할 수 있는 전체 virtual address 공간과 같은 말이 아닙니다. Pintos가 실제로 매핑하는 것은 machine의 physical memory 크기까지이고, OS 정책과 allocator 상태가 실제 사용 가능한 frame 수를 제한합니다.

2

user virtual page와 user pool frame

가운데 아래의 user BSS, data, code, stack은 process별 user virtual memory입니다. 같은 virtual address라도 process마다 다른 frame에 매핑될 수 있습니다.

Process A: user virtual 0x500000 -> physical frame A Process B: user virtual 0x500000 -> physical frame B

`user pool`은 user가 직접 보는 주소 공간이 아니라 user page에 붙일 전역 physical frame의 출처입니다. kernel은 direct map으로 이 frame들을 볼 수 있고, process context가 바뀌어도 user pool 자체가 바뀌지는 않습니다.

1. user page가 실제 frame을 필요로 함 2. allocator가 user pool에서 빈 frame을 선택함 3. page table이 virtual page -> frame 관계를 기록함 4. CPU/MMU가 접근 시 그 관계로 frame을 찾음
3

alias, bitmap, metadata

physical frame이 붙은 user page는 user virtual address와 kernel direct-map address 양쪽에서 보일 수 있습니다. 이것은 복사가 아니라 같은 frame을 가리키는 virtual mapping이 둘이라는 뜻입니다.

user virtual upage ───────┐ ├─ same physical frame kernel virtual KERN_BASE+p ┘

bitmap과 page table은 이 frame의 사용 상태와 연결 관계를 기록하는 metadata입니다. 실제 code/data/stack byte는 metadata가 아니라 frame 안에 있습니다.

4

여러 user 공간과 mapping context

각 process는 자기 user virtual address space와 page table을 하나의 묶음으로 갖습니다. 그래서 같은 `0x500000` 같은 upage라도 Process A에서는 frame A, Process B에서는 frame B, Process C에서는 아직 frame 없음으로 해석될 수 있습니다.

이때 frame A와 frame B는 모두 같은 전역 user pool 안에 남아 있습니다. 멀티스레드 관점에서는 같은 process 안의 thread들이 같은 user mapping 묶음을 공유하고, 다른 process의 thread는 다른 묶음을 봅니다. 왼쪽의 thread page/kernel stack은 thread별 kernel memory이고, 가운데 user 공간은 process별 mapping context입니다.

이 map의 핵심 결론

  • 가운데는 virtual address space, 오른쪽은 실제 physical memory입니다.
  • kernel은 `KERN_BASE + physical address`로 RAM frame을 봅니다.
  • user page는 process별 page table을 거쳐 user-pool frame에 연결됩니다.
  • user pool physical frame은 전역 RAM 안에 남아 있고, switch되는 것은 active page table입니다.
  • CPU의 address 표현 범위보다 Pintos의 실제 resident memory 한계가 더 작습니다.
  • process마다 user address space/page table 묶음이 다르며, 같은 user VA도 다른 frame을 가리킬 수 있습니다.
  • thread page/kernel stack은 thread별이고, 같은 process의 thread들은 같은 user mapping 묶음을 공유합니다.
  • bitmap/page table은 metadata이고, 실제 byte는 physical frame 안에 있습니다.
